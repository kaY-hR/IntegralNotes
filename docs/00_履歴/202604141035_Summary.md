# 2026-04-14 10:35 Summary

## 会話要約

- `data-catalog` / `original-data` / `data-set` / `data-note` などの概念が増え、ユーザー視点ではやや複雑ではないか、という相談があった。
- 極論として「全部フラットに置く」「プログラムの実装や実行は VS Code に寄せる」案もありうる、という問題提起があった。
- 目指したいのは、技術的に VS Code と競うことではなく、「実験者向け・研究向けの Obsidian のような体験」ではないか、という方向性を確認した。

## 整理した論点

- 既存要求では `note-first` / `results-focused` が中核であり、主役は対話実行そのものではなく「解析結果がノートに残り、後から再確認できること」。
- その観点では、すべての概念をユーザー前面に出す必要はない。
- 特に `data-catalog` は保管場所や内部整理の概念としては有用でも、ユーザーの主語にはしなくてよい可能性が高い。
- 一方で `original data` と `dataset` の分離は、ユーザー入力単位と解析単位を分けるための内部契約としては意味がある。

## 提案の骨子

- ストレージをフラットにすること自体は可能だが、体験までフラットにすると IntegralNotes 固有の価値が弱くなる。
- VS Code に任せるべきなのは主に Python 実装や環境調整であり、IntegralNotes が持つべき中心価値は以下。
  - ノートを起点に元データを登録できる
  - 解析 block を置いて結果を残せる
  - 結果と考察を同じ場所で見返せる
  - 後から provenance をたどれる
- したがって、UI で前面に出す語彙は最小化し、まずは
  - note
  - original data
  - analysis block
  - result
  程度に絞る方向が妥当。
- `data-catalog` や `data-note` は、必要なら内部用語または補助 UI に後退させる。
