## 背景

- 解析基盤の具体設計として、`chunk` を中心に据える案がユーザーから提示された。
- 目的は、解析の入出力、sandbox 実行、結果の公開単位、トレーサビリティを一貫して整理すること。

## 議論したポイント

- 解析の input / output 単位は file ではなく folder-based `chunk` にする
- `chunk` は workspace 内の `.chunk/<chunkId>/` に置く
- 解析 plugin は複数 input chunk / 複数 output chunk を扱える
- 可視化も chunk を入力に取るが、役割としては runner と分けた方がよい
- `artifact.md` は全 chunk 必須ではなく、公開したい結果にだけ付けるのが自然
- 実行は `%LocalAppData%/IntegralNotes/.runs/<runId>/` で行い、program master は `.program-masters/<programId>/` に置く
- 実行時の引数受け渡しは `run-args.json` が適切
- `blockId`, `runId`, `chunkId` は分離し、chunk 側に provenance を持たせる
- note path 自体は chunk に保存せず、scan により data flow を復元する方針がよい

## 判断

- `chunk` モデルはかなり筋が良い
- 特に
  - folder 単位で解析を扱うこと
  - 全中間物をユーザー管理対象にしないこと
  - provenance のために ID を分離すること
  が重要
- `runId = blockId` は避け、`1 block : N runs` として扱うべき
