# 2026-04-18 20:33 UI/UX 相談サマリ

## 相談テーマ

- `Notes/20260417.md` に書かれていた違和感をもとに、特に「中間 `json` がユーザーに見えすぎる問題」と、その周辺の note / result / provenance UX を整理した。

## 中間 `json` の位置づけ

- `json` はユーザーが編集・閲覧する成果物ではなく、複数処理の起点になる `analysis-ready` な内部派生物として扱うのが自然、という認識で一致した。
- 理想としては `元データ` や `dataset` をユーザーが選び、可視化や PCA 実行時に内部で必要な `json` を解決する UX が望ましい。
- ただし現時点では、自動 materialization は見送る判断になった。
  - 将来 parameter を持つ converter の同一性をどう扱うか。
  - 同一 format を出力する複数 converter がある場合にどれを選ぶか。
  - 単純な `format` 探索だけでは曖昧さが大きすぎる。
- そのため今は、`元データ -> json` 変換 block は explicit に残しつつ、UI 上で `json` を主役にしない方向が妥当という整理になった。
- 補助案として、`load` / `show` のような user-facing block が内部で `convert2json` まで面倒を見る構成は有力だが、app 全体の暗黙自動解決とは切り分けて考える。

## block output と note 反映

- script が note Markdown を自由に追記するのではなく、output slot 側で「作業 md に自動挿入するか」「どの input 側 note に投影するか」を宣言する方が筋がよい、という整理になった。
- 最小形として、output slot に次の 2 つを持てれば十分という認識で一致した。
  - `auto_insert_to_work_note: bool`
  - `project_to_inputs: string[]`
- `embed / link` の選択は不要で、反映するときは常に `![]()` を挿入すればよい。
- app は output が表示可能かどうかで事前判定せず、既存の viewer / embed fallback 側に委ねる。
- input slot 名は script ごとに自由なので、app 側が slot 名から主対象を推測する設計は避けるべき。
- 複数 input の block では、結果投影先を `1つだけ` に固定する案は採らず、output ごとに `project_to_inputs` で明示する方式が本命になった。
- `.idts` dataset を input にした block の結果投影先は dataset の note のみで、member file 側へ自動 fan-out しない。

## 作業 md と data-note の扱い

- 作業 md への自動結果反映は、block 直下に標準 Markdown の embed / link を出すだけで十分という整理になった。
- provenance や timestamp などの追加メタ情報は、作業 md 側には挿入しない。
- 複数回実行した場合も block 直下に新しい `![]()` を単純追記し、見た目として `一番上が最新` になる運用を許容する。
- 当初は `1 block = latest result only` を metadata / data-note 側で持つ案もあったが、その後の再整理で note への投影ルールとしては採らない方針になった。
- `data-note` も app-managed section を持たず、普通の Markdown として provenance link と `![]()` を末尾へ追記していく方針でよい、という認識で一致した。
- 不要になった古い結果は app が自動整理せず、ユーザーが必要に応じて手で削除する運用を許容する。

## provenance link と block deep link

- `data-note` 側には provenance が必要なので、少なくとも `元の作業 md` と `source block` を指す 1 行リンクが必要という認識で一致した。
- block 個別への deep link は、可能なら user-facing にも `note-path#BLK-xxxx` のような `path + block id` 形式で表現するのが最も自然という判断になった。
- 実装上は、editor 側で hash 部分の `BLK-...` を解決して該当 block 位置へスクロール・ハイライトする導線が必要になる。
- `#BLK-...` の block が見つからない場合は、その md を普通に開くだけでよい。

## provenance pane 構想

- `convert2json` を explicit block として残す以上、処理の見通しを補う補助 UI として block と file の chain を可視化する provenance pane 案が有力になった。
- 方向性は `自動 materialization` より `explicit pipeline + better navigation/visibility` を優先する。
- provenance pane はまず閲覧専用を基本とし、既に実行済みの block の内容や接続を GUI 上で編集するのは不可とする。
- 強いて許すとしても、新規 block 追加程度までに留める。
- ペインの開閉は activity bar のボタンから行い、表示先は note と同じ `flex-layout` の tab を想定する。
- scope は workspace 全体ではなく、`今開いているノート` を基本単位にする。
  - 基本起点は、そのノート内の全 block。
  - file viewer を開いている場合は、その file を起点に関連 node を表示できてもよい。
  - 現在ノート内 block から辿れる別ノートの block / file は芋づる式に表示してよい。
- node は block / file の両方を同程度の重みで扱い、各 node から元の file / block へ飛べれば十分。
- provenance pane では、中間 `json` のような internal file も特別扱いせず、通常の node と同じ見え方で表示する方針になった。
- hover preview は file node hover 時だけ出す前提で、embed と同様にスクロール可能な枠で表示できるとよいが、MVP では未実装でもよいという整理になった。
- 少なくとも block node には hover preview は不要。

## 解析 block input picker

- 解析 block 全体の GUI 化よりも、「input slot 割り当てを popup + filter に寄せること」が UX 改善の主眼という整理になった。
- `>` による block 挿入体験と揃えて、input slot への割り当ても検索 popup ベースで行う方向が自然。
- native file dialog は補助導線に留め、workspace / managed file / dataset を横断して slot 制約付きで絞り込める keyboard-first picker が本命、という結論になった。
- この論点は `docs/Issues.md` の Issue 43 として切り出した。

## 現時点の結論

- 自動 `json` 生成・自動前処理解決は、設計論点が多いため今は保留。
- 当面は `explicit transformation, implicit presentation` を軸にする。
  - 変換 block は明示的に残す。
  - 内部 `json` は user-facing の主役にしない。
  - user-facing output は output slot 宣言に従って note へ自動反映する。
  - provenance pane で file / block chain を見やすくする。
  - input 割り当ては keyboard-first picker に寄せる。
