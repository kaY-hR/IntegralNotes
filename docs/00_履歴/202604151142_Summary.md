## 会話要約

- metadata は内部的に `id`, `path`, `hash` を持ち、`path` または `hash` が変化したときに追跡できる形にしたい方針で合意した。
- そのうえで、derived dataset は immutable なので `.store` 配下に隠して保持する方がよさそう、という整理になった。
- 一方で source dataset は、ユーザーから見やすく触りやすい visible folder として保持したい意見が出た。
- source dataset の materialize には、file は hard link、directory は junction を使う案が有力と整理した。

## 整理したこと

- `derived dataset`
  - immutable を前提にし、canonical 実体は `.store` 配下に置くのが自然
  - 外部変更を積極的に取り込む対象ではなく、想定外変更は warning / dirty 扱いに寄せた方がよい
- `source dataset`
  - ユーザーが直接見て使う bundle なので、visible folder にした方が分かりやすい
  - ただし hard link / junction を使う場合、その中での編集は original data 実体の編集でもあるため、その意味づけは明示が必要
  - source dataset は「copy」ではなく「projection / working view」として定義するのが自然

## 提案した方向

- original data は `id/path/hash` で追跡する
- source dataset は visible path を canonical user-facing path とし、metadata には `members` と `tree hash` を持つ
- source dataset の各 member は original data 実体への hard link / junction で materialize する
- derived dataset は `.store` 配下に置き、必要なら viewer や note link から辿れるようにする

## 注意点

- hard link は file にしか使えず、directory は junction など別方式が必要
- source dataset 内で file 内容を編集すると original data 側も同じ内容になる
- source dataset folder への file 追加・削除・rename を dataset membership の変更として扱うかどうかは別途仕様化が必要
