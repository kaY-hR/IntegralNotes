# Summary

- 解析の input / output 単位を現状の `dataset` 固定から外し、slot ごとに拡張子を宣言して file ベースで扱いたい、という相談があった。
- 想定案として、input slot は `.idts` / `.csv` / `.json` などを受けられるようにし、output slot はユーザー指定名 + decorator で定義した拡張子で個別 file を保存したい、という方向が示された。
- 現状仕様と実装を確認したところ、要求文書・データモデル・Python runner はいずれも `1 input slot = 1 dataset`, `1 output slot = 1 dataset` を前提にしており、SDK も `inputs=["..."]`, `outputs=["..."]` の単純配列しか表現できない状態だった。
- そのため、単に decorator を少し変えるだけではなく、block schema、discovery parser、runner payload、GUI の picker、provenance / GC / note 紐付けまで含めた再設計になると整理した。
- 判断としては、UX の狙い自体には賛成だが、`dataset` を内部契約ごと廃止するより、`dataset` を「唯一の入出力単位」から「slot が選べる 1 つの表現」へ下げる方が安全、という見立てを返す方針にした。
- 特に次を先に決める必要があると整理した。
  - slot は `file / directory / dataset` のどれを受けるか
  - 複数選択を許すか、そのとき Python へは `list[path]` で渡すか一時 directory に束ねるか
  - `.idts` を選んだとき Python へ渡すのは manifest file path か、dataset resolve 後の directory path か
  - derived な単一 file を `ORD-*` に載せるなら、`ORD = original data` という現行用語をどう整理するか
- 追加で、拡張子だけを slot 契約の唯一の意味にすると `.json` や `.csv` の意味が広すぎるため、拡張子 filter とは別に semantic kind を残した方がよい、という論点も挙げた。
