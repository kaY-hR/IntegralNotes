# Summary

- ユーザーから、generated file が持つべき情報は最低限
  - `createdByBlockId`
  - `format`
  で十分ではないか、という整理が示された。
- これを受け、`provenance: source | derived` は独立項目として持たず、`createdByBlockId: null | BLK-*` で代替する方針が妥当だと判断した。
- したがって managed file の最小メタは、概ね
  - `id`
  - `path`
  - `hash`
  - `representation`
  - `formatId`
  - `createdByBlockId`
  で足りる、という見立てになった。
- Markdown の link / embed 記法については、source 記法は今まで通り path ベースを維持する方針で一致した。
- ただし relation 管理の考え方としては、
  - path は user-facing な現在位置
  - ID は内部 identity
  と役割分担するのが良く、外部 rename / move が起きた場合は path 変更を検知して参照元テキストを更新するのが最も親切、という方向で整理した。
- 具体的には、managed file の追跡で `oldPath -> newPath` を確定できたら、Markdown や block source など path 参照を持つ text file を scan して置換する方式が PoC では最も単純で実用的、という回答方針にした。
- あわせて、`path -> ID` の map は identity 決定の source of truth ではなく、現在の file open / note open / viewer 解決を速くするための current index としてだけ持てばよい、という整理にした。
