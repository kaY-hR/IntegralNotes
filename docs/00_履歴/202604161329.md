## 会話概要

- ユーザー要望:
  - Python 実行まわりをもっと直感的にしたい
  - `.py-scripts/` へフォルダコピーする現行方式はいったん忘れたい
  - `cwd` 上の `.py` を直接読み、decorator を付けた関数を app が scan して使える形に寄せたい
  - slash menu では `PCA` のような表示名と `scripts/pca.py:main` のような実体を見せたい
  - integral block は JSON がごちゃついて見えるので、更地にしてシンプルに作り直したい

## 整理した方針

- plugin システム自体は unified のまま維持する
  - Python だけ別物にせず、内部では `general-analysis` のような plugin/provider として扱う
- ただし、Markdown に保存する block 記法は human-readable に寄せる
  - `PYS-...` や bundle copy 前提の情報は見せない
  - Python block は `scripts/pca.py:main` のような `path:function` を canonical にする方向がよい
- dataset 参照は `.idts` ベースに統一する方向が自然
  - app 内部では必要なら managed data ID に解決する
  - md 上は人が見て理解しやすい path を優先する
- block には「実行仕様」を残し、「実行状態」は隠し metadata に逃がす案が有力
  - md に残すのは、どの script/function をどう呼ぶか、どの input/output slot を持つか、どの param を渡すか
  - 最新 output dataset ID や run 状態は `.store/.integral/blocks/...` のような hidden metadata に分離できる

## 設計メモ

- `(py/file/path.py).main(...)` のような独自 DSL は短いが、将来の拡張や複数行編集、escaping を考えると負債になりやすい
- そのため、Markdown block は custom DSL ではなく、YAML 風の薄い構文に寄せる方がよい
- 例:

```integral
id: BLK-...
run: scripts/pca.py:main
in:
  samples: /Data/samples.idts
  labels: /Data/labels.idts
params:
  n_components: 2
out:
  score: auto
```

- slash menu の表示名は decorator から取ればよく、md には canonical な `run:` だけ保存すればよい
