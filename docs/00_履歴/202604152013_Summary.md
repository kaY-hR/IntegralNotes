# 2026-04-15 20:13 Summary

## 概要

Issue 24 の継続として、managed data の `id / path / hash` 追跡を強化した。  
ユーザーから「ソフト上で rename しても json が追随しない」「個別操作パスごとの対症療法ではなく、根本で event 的に hook したい」という指摘があり、`WorkspaceService` の変異イベントを根元に置く構成へ修正した。

## 会話で確認したこと

- managed data の canonical storage は廃止し、original data は基本的に visible path を正とする
- data-note は ID に 1:1 で紐づく system-managed note とし、`.store/.integral/data-notes/{id}.md` に置く
- source / derived の差は、主として visible / hidden の配置差であり、`id / path / hash` の管理契約自体は共通に寄せる
- ソフト内 rename / move / delete / save に対しては、個別 UI ではなく root mutation hook で metadata 追随させる方針で進める

## 実装

- `src/main/workspaceService.ts`
  - `WorkspaceMutation` / `WorkspaceMutationListener` を追加
  - `create / delete / modify / move` を `emitWorkspaceMutations()` で通知するようにした
  - 壊れていた `deleteEntries()` の mutation 収集を修正した
- `src/main/integralWorkspaceService.ts`
  - `handleWorkspaceMutations()` を追加
  - mutation が tracked managed data に関係する場合のみ `reconcileManagedDataMetadata()` を走らせるようにした
- `src/main/main.ts`
  - `WorkspaceService` と `IntegralWorkspaceService` を mutation listener で接続した
- `docs/Issues.md`
  - Issue 24 の実装第6段として上記対応を追記した

## 期待される効果

- ソフト上で file / folder を rename・move したとき、managed data metadata (`.store/.integral/*.json`) が操作直後に追随しやすくなった
- 実装の責務が `WorkspaceService` の変異イベントに寄り、今後の操作経路追加でも追従漏れを起こしにくい構成になった
