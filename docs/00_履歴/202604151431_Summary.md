## 会話要約

- Issue 24 の実装を継続し、今回は `id / path / hash` ベースの metadata schema と、source dataset の `dataset-json` + resolver 層を主に進めた。
- あわせて、system-managed data-note が新 schema と旧 schema のどちらからでも再生成できるようにして、途中状態でも note 同期が壊れにくいようにした。

## 実装内容

- `src/shared/integral.ts`
  - original data / dataset summary に `path`, `hash`, `representation`, `visibility`, `provenance` を追加した。
- `src/main/integralWorkspaceService.ts`
  - original data / dataset metadata を新 schema へ拡張した。
  - original data は visible `path` を主 summary としつつ、内部では hidden `objectPath` を持てる形に整理した。
  - source dataset の既定表現を visible manifest `datasets/*.idts` に変更した。
  - manifest には `datasetId`, `name`, `kind`, `memberIds` を保存するようにした。
  - `datasetId -> readable directory path` の resolve 層を追加し、`dataset-json` は `.store/.integral/materialized-datasets/{datasetId}` へ materialize するようにした。
  - inspect / display / Python 実行の input path 解決が、この resolver を通るように変更した。
  - 起動時・利用時に metadata を再読込し、主に `path changed + hash same` と `path same + hash changed` を吸収する reconciliation を入れた。
- `src/main/dataNote.ts`
  - data-note frontmatter を新 metadata schema に合わせて組み直した。
  - 旧 metadata (`aliasRelativePath`, `storeRelativePath`, `sourceMembers` など) からも正規化して hidden note を再生成できるようにした。
- `src/main/workspaceService.ts`
  - data-note sync 時に新旧 metadata を正規化して扱うようにした。
- `src/renderer/IntegralAssetDialogs.tsx`, `src/renderer/App.tsx`
  - picker や status message の表示を新 schema / user-facing wording に寄せた。

- まだ legacy key を読む正規化コードは main 側に残している。これは過渡期の metadata 再解釈用で、renderer / shared の user-facing 型は新 schema に寄せた。
- reconciliation は confirm UI までは未実装で、今回は「一意に追跡できる rename / move」と「同 path 上の hash 更新」を中心に扱っている。
