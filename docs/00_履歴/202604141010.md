## 会話要約

- `data-catalog` / `data-note` を採る方針になった。
- 続いて `blob` / `chunk` の rename 候補として、
  - `blob -> dataset`
  - `blob -> raw-data`, `chunk -> dataset`
  の比較相談があった。

## 判断

- `blob -> dataset` は不採用寄りと判断した。
  - 現行 blob は「単一 file / directory の import 単位」であり、
  - ユーザー意図としての「ある set を作って import する」意味は薄いため。
- `chunk -> dataset` は比較的自然。
  - block の input / output 単位として、
  - source bundle でも derived result でも、
  - 「処理対象 / 処理結果としてのひとまとまりの data」という説明が通るため。
- `blob -> raw-data` は完全一致ではないが、`blob -> dataset` よりは意図に合うと評価した。
  - 外部世界で既に加工済みの CSV を持ち込む場合でも、
  - workspace 内では「解析前に取り込んだ元データ」と再定義すれば運用可能。

## 補足

- `raw-data` は mass noun なので、code 上の singular entity 名には少し工夫が要る。
  - 例: `RawDataSummary`, `rawDataId`, `raw-data/`
- `chunk -> dataset` を採るなら、
  - `source chunk`
  - `derived chunk`
  - `source-bundle`
  など周辺用語もあわせて再設計した方が整合しやすい。
