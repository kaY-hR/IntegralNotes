# 2026-05-08 15:54 .idts input の Python SDK 化検討

## 相談

- `.idts` input が Python へ readable folder path として渡されていることが、script author / LLM 目線では分かりにくい。
- `.idts` file path を input として渡し、解決は SDK の関数で明示的に行う方が自然ではないか、という方針を相談した。

## 現状の整理

- source dataset の `.idts` に `dataPath` が無い場合、app は実行時に `.store/.integral/materialized-datasets/{datasetId}/` を作る。
- その folder に member file の hardlink / alias を並べ、Python にはその folder path を渡している。
- これは実行時 I/O としては便利だが、`inputs["images"]` に `.idts` file path ではなく folder path が来るため、仕様を知らない script author には意外に見える。
- hardlink / alias の場合、input folder を書き換えると元 data 側へ影響する可能性があるため、read-only 前提を明示したい。

## 改善案

- 実行 payload では `.idts` input に `.idts` file path を渡す。
- Python SDK に dataset 解決 helper を用意する。
  - 例: `resolve_dataset_input(inputs["images"])`
  - 例: `resolve_dataset_files(...)`
  - 例: `prepare_dataset_output(outputs["annotated_images"])`
- SDK helper は workspace 内の見える Python 実装として提供し、LLM / 人間が中身を読めるようにする。
- app 内部の metadata / materialize 仕様は SDK が吸収し、Python script 側には「dataset を解決して files / directory を得る」意図が明示されるようにする。
- output dataset についても、Python は SDK helper で member file を置く directory を得て、manifest / metadata 登録は app 側に残す。

## 判断

- 現行方式は app 内部としては合理的だが、Python callable の第一印象としてはブラックボックス感が強い。
- 仕様変更するなら、`.idts` input / output の解決を SDK の明示 API に寄せる方が、script author と LLM の両方に分かりやすい。
- 少なくとも短期的には、skill / prompt に「`.idts` input は現在 readable folder path として渡る」「script 側で `.idts` manifest や `.store` を直接解決しない」を追記する必要がある。
