## 背景

- Python 解析実行基盤について、従来の file-in / file-out 型とは別に `debugpy` や `jupyter kernel` を使う方向性を検討した。
- 論点は
  - 商用利用してよいか
  - IntegralNotes の設計として何が増えるか
  の 2 点だった。

## 調査したもの

- `debugpy` の PyPI / GitHub
- `jupyter-client` の messaging / kernels / API docs
- `ipykernel` の PyPI / API docs
- Project Jupyter の license / trademark policy

## 結論

- `debugpy` は主実行基盤には向かない。
  - 位置付けはあくまで Python プロセス向けデバッガ
  - 解析 step の入出力、永続化、session 管理は別途必要
- `jupyter kernel` は主実行基盤になり得る。
  - ただしそれは「状態保持付き実行基盤」を採ることを意味し、ELN 本体の責務がかなり重くなる
- 現時点の本命は引き続き
  - file-in / file-out
  - run ごとの成果物記録
  - artifact 参照
  でよい
- その上で
  - `debugpy` は developer 向けデバッグ補助
  - `jupyter kernel` は将来の高度解析モード
  として扱うのが妥当

## 商用利用メモ

- `debugpy` は MIT
- `jupyter-client` は BSD 3-Clause
- `ipykernel` は BSD-3-Clause
- Project Jupyter 全体も modified BSD 系
- したがってコアライブラリ自体は商用利用しやすい
- ただし、Jupyter の商標は別であり、製品名や会社名に `Jupyter` を入れる場合は別途注意が必要

## 設計上の示唆

- `debugpy` を採っても、blob / artifact / run / workspace の設計課題は解けない
- `jupyter kernel` を採ると、
  - session lifecycle
  - interrupt / restart
  - output stream
  - Python 環境管理
  を持つ必要がある
- そのため、Jupyter 路線を採る場合は ELN 本体よりも解析モジュール側に寄せる方が綺麗
