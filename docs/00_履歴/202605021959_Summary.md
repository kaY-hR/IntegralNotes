## 現状実装メモ

- `cwd` 配下の visible な通常 file は、拡張子に関係なく managed file として自動登録される。
- `.md` は managed file だが data-note は作らない。`.py` と `.log` は現状 data-note 対象になり得る。
- `.py` は `general-analysis` plugin の Python callable discovery 対象でもあり、データ除外と混同すると解析 block 候補生成に影響しやすい。
- `MilkdownEditor.tsx` の Markdown link picker は workspace snapshot 由来の全 file を候補にし、`name` / `relativePath` の文字列一致だけでスコアリングしている。
- 空クエリでは全候補が同点になり、最後は path 順になるため、`*.log` や生成ログが大量にあると上位を占有しやすい。

## 今回の合意に近い方向性

- `*.log` や `*.py` を link 候補から消すべきではない。必要な場面ではリンクできる必要がある。
- ただし、普段の note 編集では低優先にしたいファイル種別や path がある。
- そのため「除外」ではなく「ランキング調整」として扱う。
- 設定 Dialog に置くなら「データ認識対象」ではなく「Link 候補の優先ルール」が実態に合う。

## 静的優先ルール案

ユーザー設定例:

```text
[+ 優先ルールの追加]
1: [Data/**] v ^
2: [*.png, *.html, *.jpg] v ^
3: [*.md] v ^
```

挙動案:

- ルールは上から順に評価する。
- 複数ルールに一致する場合は一番上のルールを採用する。
- 一致した候補はその priority group に入れる。
- 同じ priority group 内では既存の文字列一致スコアを使う。
- どのルールにも入らない候補は、最後に今まで通りのアルゴリズムで並べる。
- 候補からは消さない。
- `Data/*` では深い階層に合わないため、ユーザー向けには `Data/**` を推奨する。
- workspace relative path は `/` 区切りで正規化し、基本は case-insensitive match がよい。

## 文脈優先ルール案

静的 glob だけでは利用ケース依存を吸収しきれない。

候補:

- 現在開いている note と同じ folder を優先する。
- 現在 note の配下 / 近い階層を優先する。
- 最近開いた / 編集した file を優先する。
- 現在選択中の sidebar file に近い file を優先する。

MVP では、まず次の 2 つで十分そう。

```text
sameFolderBoost: true
priorityRules: string[]
```

## Relation Graph 方向性

より本質的には、workspace 内の関係 graph から「今の note に近い file」を優先するのが筋がよい。

想定 node:

- workspace file
- managed file / dataset ID
- note block
- Python callable
- data-note target

想定 edge:

- Markdown link / image: `note.md -> file`
- `itg-notes` input / output: `note.md/block -> managed data`
- block-type: `note.md/block -> py:function`
- dataset member: `dataset.idts -> member file`
- data-note: `data target -> data-note`
- frontmatter / YAML 内 ID 参照: `note.md -> managed data`
- 同じ folder: 弱い edge

link picker では、現在 note を起点に 1-3 hop 程度を探索し、候補に `relationScore` を付ける。

例:

- 1-hop: 現在 note が直接参照している file / data / block
- 2-hop: 同じ data を読んでいる別 note や dataset
- 3-hop: 関連 note から参照される file 群

この方向なら「同じファイルを読んでいる md」や「YAML ID で参照している managed data」も link 候補 ranking に反映できる。

## Storage / Index について

- Obsidian graph view は、おそらく graph DB ではなく metadata cache / resolved links / in-memory adjacency map / Pixi.js or WebGL drawing で高速化している。
- IntegralNotes の link picker 用途なら、まずは graph DB よりも precomputed relation index + BFS で十分そう。
- 永続化するなら SQLite の edge table が堅い。

SQLite index 案:

```sql
nodes(id, kind, path, label, hash, mtime)
edges(src, dst, type, weight, source_path, source_hash)
```

- recursive CTE で 1-3 hop traversal できる。
- `.store/.integral/relation-index.sqlite` のような派生 index として扱う。
- Graph view / ProcessChain / AI context / 検索にも共用できる可能性がある。

その他候補:

- NetworkX + pickle: 実験には便利だが、Electron/TypeScript 側と相性が悪く、pickle は不透明で安全性・保守性に難があるため本体 index には不向き。
- Kuzu: SQLite の graph DB 版に近い embedded property graph DB。Cypher が使えるが、native dependency / Electron packaging / migration のコストがある。graph index をプロダクトの中核にすると決めた段階で検討する。
