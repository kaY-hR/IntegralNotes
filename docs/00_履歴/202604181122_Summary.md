# Summary

- ユーザーから、`original data` という概念自体よりも、`ID + path/hash tracking` の基盤さえあれば十分ではないか、という再整理が示された。
- `dataset` についても、「複数 file を束ねられる」「1 file が複数集合に所属できる」利点はあるが、すべての解析入出力を強制的に `dataset` 化する理由は薄い、という認識で話が進んだ。
- 現行実装の確認結果として、Python には `.idts` manifest path ではなく、dataset を resolve した directory path を渡していることを再確認した。
  - derived dataset は manifest の `dataPath` を直接 resolve して `.store/objects/{datasetId}` 相当の directory を渡す
  - source dataset は `memberIds` から staging directory を materialize し、その directory path を渡す
- また、現行の `many input` は `list[path]` ではなく、複数 original data をいったん 1 つの source dataset (`.idts`) に束ね、その dataset を 1 directory path に resolve して Python に渡す方式だと整理した。
- `provenance: source | derived` は、「original data / dataset」の区別ではなく、「その managed entity が block 生成物かどうか」の軸として説明するのが適切、という回答方針にした。
- ただし、概念をさらに削るなら `provenance` も必須ではなく、`createdByBlockId` の有無で代替できる、という見立ても併記する方針にした。
- 今後の推奨方針としては、次の整理が最も筋が良いと判断した。
  - `cwd` 内の user file は広く managed data として追跡する
  - Markdown link 記法そのものは path ベースの標準記法を維持する
  - 内部の relation / graph / provenance は ID ベースで保持する
  - metadata / relation / index は SQLite へ寄せてよい
  - ただし canonical content まで `.store` 平置きへ全面移行するのは、workspace / Git / 外部 editor / Python 実行との相性が悪く、やりすぎの可能性が高い
