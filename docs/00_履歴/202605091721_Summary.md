# 2026-05-09 17:21 package tree / exports / global extension root 方針

## 背景

plugin system の整理の続きとして、Python block、AI skill、runtime plugin、viewer plugin を package としてどうまとめるかを議論した。

`.icv` image compare のように、Python helper / Python block、AI skill、viewer plugin が一式で揃って初めて便利になる機能がある。一方で、Python や skill そのものに個別 manifest を持ち込むと、一般ユーザーが作って共有する軽さが失われる。

## package の基本方針

package は導入する。

ただし package は dependency resolver ではなく、次のための管理単位とする。

- 一式配布
- vendor 配布物の version 管理
- package 内 component の公開範囲管理
- 欠落検知
- 由来管理

Python / skill 側には個別 manifest を持たせない。package だけが `integral-package.json` を持つ。

## global extension root

拡張資産は dev / production で分けず、当面は `%LocalAppData%\IntegralNotes` に統一する。

Electron の `userData` は今後も app settings / logs / chat history などの app 状態に使うが、runtime plugin / package / global skill / global script の root には使わない方針。

```text
%LocalAppData%\IntegralNotes\
  global\
    skills\
    scripts\
  runtime-plugins\
  packages\
```

役割:

- `global/skills`
  - 単体で使える汎用 skill
- `global/scripts`
  - workspace へコピーして使う汎用 Python script stock
- `runtime-plugins`
  - package に属さない単体 runtime plugin の install 先
- `packages`
  - vendor package の install 先

## package tree 型

package は種類別フォルダへ materialize するのではなく、package folder を root とする tree 型を採用する方針。

理由:

- vendor が配布した package folder を丸ごと差し替えれば version up できる
- package 単位の uninstall / rollback が単純になる
- package 間で同名 skill / script / helper があっても衝突しにくい
- `packA -> packB` のような dependency を増やさず、必要なら各 package が同じ helper を内包できる

想定構成:

```text
%LocalAppData%\IntegralNotes\packages\
  cell-image-pack\
    integral-package.json
    runtime-plugins\
      image-compare-viewer\
        integral-plugin.json
    skills\
      cell-image-analysis\
        SKILL.md
    scripts\
      segment_cells.py
      utils\
        image_io.py
```

## exports による public / private 制御

package 内の skill / Python script / runtime plugin は、原則 private とする。

通常 UI や scanner が見るのは `integral-package.json` の `exports` に明示されたものだけにする。

例:

```json
{
  "apiVersion": "1",
  "id": "cell-image-pack",
  "version": "0.1.0",
  "displayName": "Cell Image Pack",
  "exports": {
    "skills": [
      "skills/cell-image-analysis"
    ],
    "pythonBlocks": [
      "scripts/segment_cells.py:main"
    ],
    "runtimePlugins": [
      "runtime-plugins/image-compare-viewer"
    ]
  }
}
```

挙動:

- `global/skills` は従来どおり skill 候補に出る
- `global/scripts` は従来どおり Python stock 候補に出る
- `packages/*/integral-package.json` をまず読む
- package 内では `exports` に載った component だけを通常 registry に登録する
- exported Python block については、指定 file / callable だけを読み、`@integral_block` があるか確認する
- exported runtime plugin については、その plugin の `integral-plugin.json` を読む
- package 内の non-exported skill / script / helper は通常 UI に出さない
- exported skill / script から package 内 helper を相対参照することは許可する

## exports 必須

package では `exports` を必須とする。

`exports` が無い package は invalid として扱う。未指定時に package 内を全走査して全部公開する挙動は採らない。

理由:

- 細胞画像用 skill / 半導体画像用 skill などが全部候補に出ると UI が混乱する
- vendor が public surface を明示できる
- package 内部 helper を安全に隠せる
- scan 対象を `exports` に限定できる

## scan 対象

当面の scan root は次の形にする。

```text
global/skills
global/scripts
packages/*/integral-package.json
packages/*/runtime-plugins|skills|scripts
```

ただし package 内の `runtime-plugins|skills|scripts` は全件を通常公開するのではなく、`integral-package.json` の `exports` に載った path だけを有効化する。

## Python の扱い

Python は解析再現性に直結するため、実行時の source of truth は workspace (`pwd`) の `.py` とする。

global / package 内 script は stock / template として扱い、使うときに workspace へコピーする。package 由来の Python block をコピーする場合は、衝突回避のため package ID を含む path へ配置する案が自然。

例:

```text
scripts/packages/cell-image-pack/segment_cells.py
```
