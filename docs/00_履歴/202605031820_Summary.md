## 基本方針

- `Relation Graph DB` を主機能とする。
- `Link Picker Ranking` は Relation Graph DB の利用者の 1 つにすぎない。
- DB schema や scan 方針を link picker 最適化に寄せすぎず、Graph View / AI context / ProcessChain / 将来のサジェスト全般が読む共通基盤として設計する。
- Relation DB は source of truth ではなく、Markdown / frontmatter / managed metadata / manifest / Python source などから作る派生 index とする。
- 将来 GUI で関係を追加する場合も、DB を直接編集せず、`.md` 本文 link や frontmatter など source 側に書き込み、それを再抽出する。

## Relation DB

- 保存先は `.store/.integral/relation-index.sqlite` のような SQLite 派生 DB とする。
- DB は基本的に `nodes` / `edges` の graph 構造保存に徹する。
- weight / ranking score は DB に固定保存せず、用途ごとの query 時に top N 候補へ計算する。
- 必要なら将来、用途別 cache table は検討するが、MVP では graph 構造を優先する。
- 全体再構築は初回作成、index 破損、schema migration などの保険に留める。
- 通常は managed file の hash / path 変更検知を起点に、その origin 由来の node / edge だけ差し替える。

## Node 方針

- node ID:
  - data-note 共有を含む data 系統合ノード: `noteTargetId`
  - 通常 file / dataset: managed data ID
  - block: `noteId + blockId`
  - Python callable: `relative/path.py:function`
  - folder: workspace relative path
- folder も node として扱う。
- data file と data-note は Graph View 上では一心同体として扱う。
- 同じ data-note を共有している複数 file / dataset も、Graph View 上では `noteTargetId` 単位の 1 ノードに畳む。
- MVP では data 系統合ノードをクリックすると data-note を開く。
- 共有 file / dataset の内訳表示や展開 UI は後回し。
- `nodes` には Graph View が snapshot だけで描けるよう、`kind / label / path / noteTargetId / updatedAt` 程度の denormalized 情報を持たせてよい。
- 正はあくまで managed metadata / Markdown / manifest 側。

## Edge 方針

- edge は有向 edge とする。
- 向きは基本的に「参照を書いている側 -> 参照される側」で統一する。
- 例:
  - Markdown note -> linked file
  - data-note 統合 node -> linked file
  - block -> input data
  - block -> output data
  - block -> Python callable
  - folder -> child
- edge type を持たせ、Graph View で表示 / 非表示や強調を切り替えられるようにする。
- 想定 type:
  - `markdown-link`
  - `markdown-image`
  - `block-input`
  - `block-output`
  - `block-run`
  - `dataset-member`
  - `data-note`
  - `contains`
- 差分更新のため、`edge.src / edge.dst / edge.type` とは別に `originNodeId / originPath / originHash` を持つ。
- `edge.metadata` は JSON text として持たせる方針。
  - 中核情報は通常カラムに持つ。
  - Markdown link label、block slot name、抽出元位置など type 固有の補助情報は JSON に逃がす。

## Scan / Extractor

- scan / DB 登録は built-in plugin ではなく、main 側 core workspace service の責務とする。
- Graph View plugin は read-only API で `nodes / edges` を取得して描画する。
- Link Picker Ranking は後から relation query API を使う利用者として実装する。
- Relation scan 対象は、初期実装では構造を解釈できるものに限定する。
  - `.md` / data-note の Markdown link / image
  - `itg-notes` block
  - `.idts` manifest
  - managed metadata
  - Python decorator / callable
  - folder contains
- `.txt` や `.log` など任意テキスト内の URL / path 抽出は MVP では行わない。
- 抽出ロジックは個別 module に切る。
- scan 本体は built-in `RelationExtractor[]` を順番に呼ぶだけにする。
- MVP では core 内の built-in extractor registry に留める。
- 将来 plugin が extractor を追加できる余地は残すが、最初から公開 plugin API にはしない。

## Graph View

- Activity Bar から開く built-in plugin / tab として実装する。
- 名前は `Relation Graph`。
- 既存の process chain viewer とは別機能とする。
- 描画は `sigma.js` を使う。
- 初期表示は workspace 全体グラフ。
- 現在アクティブな tab の node と、その 3-hop 近傍を強調する。
- Graph View の更新は、開いた時点で DB snapshot を取得し、workspace sync 完了時に再取得する。
- リアルタイム差分 stream は後回し。
- MVP 操作:
  - node click で対象を開く
  - active tab node と 3-hop 近傍を強調
  - edge type filter
  - search
  - zoom / pan
- 後回し:
  - layout 保存
  - 手動配置
  - cluster 折りたたみ
  - 詳細 panel

## Query API

- MVP の core API はまず次の 2 本。
  - `getRelationGraphSnapshot()`
  - `getRelationNeighborhood(originNodeId, maxHops)`
- Graph View は全体 snapshot を使う。
- Link Picker / AI context などは N-hop 近傍 API を使う。
- suggestion 用途では `originNodeId` から最大 3 hop の bounded traversal で候補集合を作る。
- その候補集合にだけ用途別 score を計算する。
- `contains` edge は有用だが膨らみやすいので、link picker などでは低 weight または hop 展開制限の対象とする。

## Frontmatter Relations

- 将来 GUI で関係を追加する場合に frontmatter の `relations:` を使う案を検討した。
- YAML frontmatter は list を持てるため、当初は workspace relative path list として保持する案が出た。
- ただし object form や type / label など将来 schema の検討が膨らむため、MVP では `relations` frontmatter は一旦なしにする。
- 将来の手動 relation は、実際に必要になった時点で、本文 link として出すのか、専用 frontmatter schema を再検討するのかを決める。

## 削除時の扱い

- file / folder が削除、または managed 対象から外れた場合、Relation DB 上の node / edge は即削除する。
- 履歴 graph や missing node 表示は MVP 対象外。

## 実装順

1. main 側 core workspace service に relation scan / SQLite 登録 / query API を実装する。
2. extractor を built-in registry として分離実装する。
3. sigma.js による `Relation Graph` built-in plugin / tab を実装する。
4. その後、link picker ranking を Relation Graph DB に乗せる。
