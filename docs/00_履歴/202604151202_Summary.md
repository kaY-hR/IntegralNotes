## 会話要約

- `source dataset` を JSON manifest、`derived dataset` を `.store` 内 immutable folder として大きく分ける案に対して、その差が大きすぎるのは避けたいという指摘があった。
- `Data/` は絶対的な保存場所ではなく default に過ぎず、`dataset-json` と `original-data` は workspace 内のどこに置いてもよい、という整理になった。
- `source` / `derived` の違いは、「その data-set または元データが `.store` に隠れているかどうか」という visibility の違いだけに寄せたい方針が出た。
- `hash` や `id` の仕組み、追跡方法については `source` / `derived` で差を付けない方がシンプルという考えで合意が進んだ。

## 整理したこと

- `source/derived` を保存形式の違いとして扱うのではなく、provenance / visibility の違いとして扱う方が単純
- モデル上は次の軸を分けると整理しやすい
  - identity: `id`
  - current location: `path`
  - content identity: `hash`
  - representation: `file` / `directory` / `dataset-json`
  - visibility: `visible` / `hidden-in-store`
  - provenance: `source` / `derived`

## 提案した方向

- `original-data` と `dataset` のどちらも、共通で `id`, `path`, `hash` を持つ
- `path` は current canonical path であり、workspace 内のどこでもよい。`Data/` は default に留める
- `source` / `derived` は metadata 上の意味差としてだけ持つ
- hidden 管理したいものは `.store` 配下 path を current path とし、visible にしたいものは通常 path を current path とする
- 起動時 reconciliation、rename 追跡、hash 追跡の仕組みは `original-data` / `dataset`、`source` / `derived` を問わず共通で動かす

## 補足

- この方針にすると、UI と内部実装の両方で「source だから別形式」「derived だから別 API」といった分岐を減らしやすい
- 一方で、`dataset-json` を採るなら `representation` 軸は必要で、実行時に path 群へ resolve する adapter は別途必要
