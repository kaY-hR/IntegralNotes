## 背景

- plugin 仕様について、サードパーティが独立した環境で開発できるべきではないか、という観点の整理が必要になった。
- あわせて、現状の built-in plugin 本体がどこに実装され、どう参照されているかを可視化する必要があった。

## 実施内容

- `docs/20_アーキテクチャ/pluginシステム.md`
  - third-party plugin は別 repo / 別 build 環境で開発できるべき、という前提を明文化。
  - IntegralNotes 本体が公開すべき境界面として、manifest schema、renderer contract、action protocol、SDK の必要性を整理。
  - 現状の built-in plugin 実装位置を以下の粒度で整理。
    - `src/shared/plugins.ts`: manifest / metadata
    - `src/renderer/integralBlockRegistry.tsx`: built-in block definition と preview UI
    - `src/renderer/integralCodeBlockFeature.tsx`: node view / JSON editor / 実行ボタン / IPC
    - `src/main/main.ts`: action 実行の `switch`
    - `src/main/pluginRegistry.ts`: external manifest 発見
  - mermaid で以下を追加。
    - 現状の参照関係
    - 現状の built-in plugin 実行フロー
    - 目標の参照関係
    - 目標の実行フロー

## 判断

- 現状は manifest だけ shared 化され始めた段階で、plugin 実体はまだ本体 repo に埋まっている。
- third-party 開発を成立させるには、IntegralNotes 本体から plugin source への compile-time 参照を切り、artifact と protocol だけを見る構成へ進める必要がある。
