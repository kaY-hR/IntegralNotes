# 2026-05-10 10:55 Markdown validation と block identity 相談

## 背景

Integral block の `id` や実行済み `out` の managed data ID が、コピー&ペーストで重複する問題について仕様相談した。

重複すると、別 block が同じ実行済み結果を持つように見えたり、`createdByBlockId` による実行済み判定が壊れる可能性がある。

## 決まった方針

- block identity は本文編集で気軽に複製されるべきではない。
- 見たまま編集で block をコピー/ペーストする場合は、アプリ側で「新しい未実行ブロック」として扱うのが望ましい。
  - 新しい block ID を発行する。
  - `inputs` と `params` はコピーしてよい。
  - 実行済み output ID は引き継がず、`null` または新しい出力予定 path に戻す。
- text 直接編集モードは draft 扱いにする。
  - text 直接編集中は壊れた中間状態を許す。
  - text 直接編集から見たまま編集へ戻るタイミングで Markdown 全文を validate する。
  - validate NG の場合は切り替えず、ユーザーへ修正内容を示す。
- text 直接編集できるユーザーには、エラーメッセージで `id:` や `out:` を明示してよい。
  - 例: block をコピーして使う場合は、コピー側の `id:` 行を削除し、`out:` に入っている実行済み出力 ID を `null` または新しい出力先 path に戻す。
- AI commit tool も同じ validator を使う。
  - `insertMarkdownAtCursor` は挿入後の Markdown 全文を仮生成して validate する。
  - NG の場合は note に反映せず、validation error を tool result として返す。
  - AI は error 内容を見て修正し、再度 commit tool を呼べる。
- `itg-notes` を含む Markdown を一律禁止するより、共通 validator の結果に基づいて reject する方が汎用的。

## 設計意図

- commit tool ごとに個別ルールを増やさず、validate 項目を追加するだけで入口全体を守れるようにする。
- 将来の paste / import / bulk rewrite / AI commit なども同じ validator に通せる。
- duplicate block ID や duplicate output ID の自動修復は、コピー元判定が難しいため慎重に扱う。
- ユーザーに `id` を意識させない UI 操作では「新しい未実行ブロックにする」という表現に寄せる。

## 残論点

- text mode を保存不可にする場合の UX と status 表示。
- validate error の表示場所と形式。
- block paste を clone-as-draft にする実装範囲。
- output を `null` に戻すか、slot 定義から新しい出力予定 path を再生成するか。
- AI commit tool の terminal tool 扱いを、validation 成功時だけにできるか。
