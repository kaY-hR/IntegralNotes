# 2026-04-14 10:41 Summary

## 会話要約

- `data-catalog` を導入した元の理由として、データはファイル名だけでは内容が分からず、解析 input 選択時に困る、という背景が共有された。
- また、測定者・目的・補足コメントなどを元データ自体のフォーマットに埋め込めない場合があり、その受け皿も必要という話になった。
- そのため、データファイルに 1:1 対応する Markdown を持たせ、そこに
  - 人が書く説明
  - システムが追記する情報
  - 可視化済みグラフ
  を置けるようにし、input 選択時はその Markdown を preview する体験がよい、という意図を確認した。

## 整理した判断

- この発想自体は妥当で、IntegralNotes の独自価値に近い。
- 重要なのは `data-catalog` という「棚・保管場所」の概念ではなく、
  - 各 original data に人間向けの説明ページがあること
  - そのページを入口に選択・理解・再利用できること
  である。
- したがって、主役にすべき概念は `catalog` より `data page` / `data note` 側。
- `catalog` はそれらを一覧するビュー名・内部構造名に後退させる方が、ユーザーには分かりやすい。

## 提案の方向

- ユーザー体験としては「ファイルを選ぶ」のではなく「説明と preview 付きの data page を選ぶ」に寄せる。
- 内部では
  - original data = 実体ファイル
  - data page = 人間向けの文脈と preview
  - dataset = 解析の入出力単位
  の 3 層に整理すると理解しやすい。
- これなら、VS Code が得意な「コード編集」とは役割が分かれ、IntegralNotes は「実験データの意味づけと再確認」に集中できる。
