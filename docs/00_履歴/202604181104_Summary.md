# Summary

- ユーザーから、`original data` の概念自体には強く拘っておらず、ID 管理の目的は path / hash 変更時の追跡を安定させることだけだ、という整理が示された。
- そのため、PoC 段階で後方互換も不要なら、`original data` と `dataset` の二分法を保つ必要は薄い、という前提で再整理する方針になった。
- 一方で `dataset` には
  - 実ディレクトリを作らずに集合を表せる
  - 1 file が複数の集合に所属できる
  という実利はあるが、すべての解析入出力を `dataset` に統一するほどの必然ではない、という認識で一致した。
- 回答方針としては、次の整理が自然だと判断した。
  - `original data` は基盤概念から外す
  - ID / path / hash tracking はすべての managed entity に共通適用する
  - `dataset` は universal な I/O 単位ではなく、複数 file を束ねたいときの optional な集合表現に下げる
  - 単一 file 入出力は、そのまま file slot として扱う
- あわせて、`ORD-*` / `DTS-*` のような意味付き prefix も見直し対象で、概念整理後はより汎用的な ID 体系へ寄せるのが筋、という考えを返す方針にした。
