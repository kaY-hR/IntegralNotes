## 背景

- ELN 上で分析データの binary ファイルをアップロードし、plugin で描画・解析できるようにしたいという相談があった。
- とくに、装置ベンダの DLL / exe を使う専用 plugin だけでなく、任意の `py` を実行する `Python実行Plugin` を主軸にしたいという意図が共有された。
- 解析 chain の中間情報はメモリ保持ではなく、ファイル永続化を前提にした方がよいという方向で整理した。

## 実施内容

- `docs/20_アーキテクチャ/分析データ添付と解析実行.md`
  - 任意ファイル添付、Attachment / Artifact / Job / Step のドメインモデル、workspace レイアウト案を整理した。
  - 現在の plugin 契約で何ができていて、何がまだ表現できないかを明文化した。
  - plugin 契約を拡張する理由を、「ファイル概念の欠如」「解析 step の返り値不足」「viewer 入力不足」「Python plugin と workspace script の分離」「plugin 間連携のための成果物契約不足」の観点でまとめた。
  - `viewer contribution`、`runner contribution`、step 実行結果拡張という追加方針を第一案として整理した。
