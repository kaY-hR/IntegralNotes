# 2026-04-15 11:00

## 会話要約

- `ORD-...` / `DTS-...` を canonical な見せ方にしている現状は、ユーザー視点では分かりにくいという問題提起があった。
- 代わりに、`cwd` の構造・ファイル名・各ファイル hash を `cwd/.integral` に保持し、起動時に path または hash の片側だけが変わった場合は検知して confirm / update する構成の是非を相談した。

## 整理したこと

- user-facing な canonical を ID ベースから path/hash ベースへ寄せる方向自体には賛成と整理した。
- ただし、内部参照まで完全に path/hash のみへ寄せると、以下が不安定になりやすいと指摘した。
  - block の `inputs` / `outputs` が参照する dataset identity
  - `createdByBlockId` や `sourceMembers` を使う provenance
  - rename と content update が同時に起きたときの同一性判定
  - derived dataset の immutable 履歴管理
- 現行実装・要求では、内部参照は ID を正とする前提が強いことを確認した。
  - `docs/10_要求/00_プロダクト概要.md`
  - `docs/10_要求/10_MVP要件.md`
  - `docs/10_要求/20_ユーザー体験.md`
  - `src/shared/integral.ts`
  - `src/main/integralWorkspaceService.ts`

## 提案した方向

- 「`ORD-` / `DTS-` を廃止する」より、「`ORD-` / `DTS-` を user-facing canonical から外す」が筋が良い。
- 具体的には、user-facing には workspace relative path と label を見せ、`.integral` には path/hash snapshot を持たせる。
- 一方で、内部では hidden な stable ID を残し、block schema・provenance・metadata lookup のキーとしてだけ使う hybrid を推奨した。
- 起動時の reconcile は次の分類にすると分かりやすいと整理した。
  - hash same / path changed: rename or move
  - path same / hash changed: content changed
  - path missing / hash duplicate or both changed: ambiguous, manual confirm

## 補足

- もし pure path/hash モデルへ振り切るなら、derived dataset の保存先命名規則と immutability の扱いを先に決めないと、再実行や履歴追跡で破綻しやすい。
