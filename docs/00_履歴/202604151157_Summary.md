## 会話要約

- source dataset を visible folder として materialize する案に対して、別案として「dataset 自体は `Data/` 配下の JSON で管理し、独自拡張子 + 専用 editor で編集・表示する」案が出た。
- その JSON の中身は `original-data-id` のリストを持つイメージであることを確認した。

## 整理したこと

- この案は source dataset に限ればかなり現実的と整理した。
- ただし canonical 実体を JSON だけにすると、普通の filesystem として見える dataset という利点は失われる。
- そのため、意味づけとしては
  - dataset の membership / metadata の canonical は JSON
  - 実データの canonical は original data 側
  と分けるのがよいと整理した。

## 提案した方向

- `source dataset`
  - `Data/xxx.<custom-ext>` の JSON で管理
  - JSON には `datasetId`, `name`, `memberOriginalDataIds[]` などを保持
  - app は専用 editor で編集・表示する
  - 実行時だけ staging folder を組み立てるか、必要に応じて projection folder を materialize する
- `derived dataset`
  - 引き続き `.store` 配下の immutable folder を canonical にする

## 注意点

- JSON manifest 方式は UI 実装と管理は楽になる一方、外部ツールから普通の folder として扱いやすい体験は弱くなる
- Python や plugin に渡すには、実行時に path へ resolve する層が必要
- source dataset の中で file 単位 rename / add / delete を直接触る UX は、manifest 方式では扱わず、member の追加削除 UI に寄せる方が自然
