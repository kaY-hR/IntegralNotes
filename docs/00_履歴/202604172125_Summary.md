- 解析の input 設定や解析結果閲覧で `DTS-...` ID をユーザーが選んだり打ったりする必要があり、UX が悪いという相談があった。
- 望む方向として、基本は dataset 名を表示し、`.idts` file も可能な限り dataset 名ベースにしたい、また hidden 保存のせいで結果閲覧しにくい derived dataset の `.idts` は `Data/` 配下へ出したい、という要望が示された。

## 調査結果

- 実現性は高い。
- 現状の source dataset はすでに `name` を持ち、`src/main/integralWorkspaceService.ts` の `createSourceDatasetManifestRelativePath()` で dataset 名を sanitize した `.idts` file 名を使っている。
- 一方、derived dataset は `executePythonBlock()` 内で
  - dataset 名を `DTS-...` にしている
  - manifest path を hidden `.store/.integral/datasets/{datasetId}.idts` に固定している
  - block source の `out:` へもその hidden `.idts` path を書き戻している
  ため、結果閲覧時に hidden path 前提の UX になっている。
- block 実行や viewer 側の内部参照は `datasetId` を正としており、renderer 側でも `.idts` path から `datasetId` へ解決する層がすでに入っているため、manifest の保存先を `Data/` に変えても内部契約は崩れない。
- UI 側は既に dataset 名優先の箇所が多いが、少なくとも次では ID 露出が残っている。
  - source dataset 作成完了メッセージ
  - データ登録 dialog から作成した直後のメッセージ
  - `.idts` manifest viewer の summary chip / note section
  - dataset data-note の legacy title fallback

## 実装方針

- internal canonical ID は今までどおり `DTS-...` を維持する。
- user-facing な表示は dataset 名を優先し、ID は通常 UI では出さない。
- derived dataset 作成時は dataset 名を `{解析名}_{slot名}_yyyyMMddHHmm` で自動生成する。
- derived dataset の `.idts` manifest は hidden ではなく `Data/{dataset名}.idts` を既定保存先にする。
  - file 名は sanitize し、衝突時は `_1`, `_2` のように suffix を付ければよい。
- 実データ directory は引き続き `.store/objects/{datasetId}/` のままでよい。
- block source の `out:` には visible `Data/...idts` path を書き戻す。
- metadata JSON / data-note / provenance / datasetId 解決は現行基盤をそのまま再利用できる。

## 注意点

- `Data/` 配下の `.idts` は visible file になるため、Explorer や link/embed 補完で自然に見えるようになる。
- 一方で managed data の自動登録は既存 metadata path を `claimedPaths` として先に確保しているため、app 自身が生成した `.idts` が original data として誤登録されるリスクは低い。
- 仕様文書側には、従来の「derived dataset は hidden `.store/.integral/datasets/{datasetId}.idts` を default にしてよい」という記述が複数残っているため、実装前に要求・アーキテクチャ・設計を先に揃える必要がある。

## 結論

- 既存の `datasetId -> metadata -> manifest -> readable path` 基盤を維持したまま実現できる。
- 主変更点は `derived dataset の naming/path 生成`, `UI の表示文言`, `関連仕様文書` の 3 点であり、大きな再設計や migration は不要。
