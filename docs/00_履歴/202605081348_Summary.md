# 2026-05-08 13:48 managed file / dataset 作成時の重さ調査

## 相談

- ファイル管理用の hash 計算の影響で、ファイル作成後に名前を確定すると 1-2 秒固まるのではないか。
- 既に追加済みの data file 群から dataset (`.idts`) を作るだけなのに、非常に時間が掛かった理由を知りたい。

## 調査

- 直近履歴 3 件と `docs/10_要求/00_プロダクト概要.md` を確認した。
- `src/main/integralWorkspaceService.ts` と `src/main/workspaceService.ts` の managed data / dataset 作成経路を確認した。
- `integral-analysis` MCP は無関係という指示に従い、使用していない。

## 分かったこと

- `.idts` manifest の JSON 書き込み自体は軽い可能性が高い。
- `IntegralWorkspaceService.ensureIntegralWorkspaceReady()` が多くの public API 入口で `reconcileManagedDataMetadata()` を呼び、そこで workspace 全体の trackable entry を hash している。
- `collectTrackableWorkspaceEntries()` は directory hash を再帰的に計算した後、さらに子要素へ再帰して file / directory hash を再計算するため、深い tree では同じ file を複数回読む可能性がある。
- `createDatasetFromWorkspaceEntries()` は選択 path ごとに `findManagedFileMetadataByPath()` を呼び、そのたび metadata JSON 群を走査するため、既存 managed file だけでも O(選択数 * metadata数) になり得る。
- dataset 作成後の `syncManagedDataNotes()` は全 metadata を走査し、dataset note 用に member metadata を読み直して Markdown link 群を生成する。
- 通常の file 作成時の固まりは hash だけでなく、`WorkspaceService.ensureWorkspaceReady()` による data-note 同期、再帰的 snapshot 作成、managed directory / pending tracking issue がある場合の full reconcile が候補。

## 改善候補

- workspace ready のたびに data-note sync / full reconcile しない。
- dataset 作成は UI から managed file ID を渡すか、path -> metadata map を 1 回だけ作る。
- full hash は明示更新・起動時・関連 mutation 時に限定し、mtime/size cache や bottom-up hash に寄せる。
- `syncManagedDataNotes()` を全件同期ではなく、変更された managed data / dataset だけの差分更新にする。
