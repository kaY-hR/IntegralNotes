## 議論の流れ

- 最初の問題意識は「PCA の結果が json の note に書かれてしまう」「本来は idts 側の note にぶら下がるべきではないか」
- 途中で「中間ファイルは note を持たず、本体ファイルや dataset の note に集約したい」という話に進んだ
- さらに整理すると、`json` / `csv` / `txt` などは本体そのものではなく、元データや dataset から得られた抽出物・添付物として見たい、という意図が明確になった
- ただし、単純な「本体から添付へスター構造」では `結果2は結果1に依存する` ような chain を表現しづらく、`ProcessChainViewer` 的な依存グラフの方が本質に近い、という認識になった
- そのうえで、UX としては「中間ファイルからノートを開くと親の note が開く」体験が直感的ではないか、という具体案が出た

## 現時点で有力だった設計案

- `data-note を共有する` ではなく、より正確には `artifact は自分専用 note を持つ代わりに、別 artifact を note owner として参照できる` モデルにする
- output slot に対して「この output slot の data-note は、この input slot の data-note を共有する」という指定を持てるようにする案が有力
- 実装概念としては `share_note_with_input` より、`noteOwnerId` または `noteTargetId` のような内部表現の方が筋が良い、という整理をした
- このモデルなら:
  - 中間ファイルの `ノート` ボタンは残せる
  - 押すと親 input 側の note が開く
  - 後から note への追記が起きても親側に集約できる
  - 中間ファイルごとに専用 note を増やさずに済む

## 設計上の暫定整理

- `managed file / dataset` は保存・追跡の技術単位であり、意味の単位とは限らない
- 一方で `dataset` 自体は「試料の集合」「LotX のような集合単位」として、かなり本尊に近い概念とも言える
- ただし、より厳密な `sample / lot / 試料セット` まで UI に導入すると、OS 的なファイル理解から離れすぎて学習コストが上がる懸念が強い
- そのため、議論の途中では:
  - 「v1 は dataset を主語に残す」
  - 「ただし note の所有は artifact ごとではなく共有・継承できるようにする」
  - 「依存関係の正は process chain 側に寄せる」
  という折衷案がかなり有力だった
