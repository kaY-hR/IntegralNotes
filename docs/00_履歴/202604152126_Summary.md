# 2026-04-15 21:26 Summary

## 会話要約

- `data-note` が `.store/.integral/data-notes/` に hidden / system-managed 化された一方、現状は「ある data を見たときに、その data の data-note も自然に見える」導線がまだ弱い、という相談があった。
- 特に original data を explorer 選択や link open で開いたとき、renderable file は preview できるが data-note へ自然につながらず、non-renderable file は app 内で見る術が薄い、という課題感が共有された。
- `data-note` 本文は user-editable な説明 note であり、`.store/.integral/{id}.json` の system metadata とは役割が違うため、UI 上で「メタ情報編集」と呼ぶと概念が混ざる、という整理を行った。

## 提案の主軸

- UX は `file path を開く` ではなく `managed data entity を開く` に寄せるのが自然、という方向を推奨した。
- 具体的には、managed data に対しては `managed-data view` のような合成ビューを用意し、その中で
  - main pane: data preview / file info / external open
  - secondary pane: data-note
  を扱う構成が、storage 設計とも整合しやすいと整理した。
- renderable / non-renderable で UI を完全に分けるより、同じ view を使い、default の pane 配分だけ変える方がシンプル、という話になった。
  - renderable: preview 主、note 副
  - non-renderable: note 主、preview / open action 副

## 設計メモ

- renderer 側の現状は、`data-note` open がまだ `targetId -> hidden note path -> workspace file open` という file-path 寄りの流れになっているため、Issue 24 の entity-first 方針とは少しズレが残っている。
- 今後実装するなら、
  - `relativePath -> managed data` 解決
  - `openManagedData(targetId)` の新導線
  - `managed-data::{id}` の tab identity
  あたりを first-class にすると、rename / move / hidden path 差し替えにも強い構成にしやすい。
