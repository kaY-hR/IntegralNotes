# 2026-04-30 datatype 導入と kind / format 整理

## 背景

- 解析を組み合わせて使うには、解析 output と input の意味的な互換性を共有できるラベルが必要。
- 既存の `kind` / `format` / `acceptedKinds` / `producedKind` は責務が重なり、`.idts` の物理表現と解析上の意味分類が混ざっていた。
- 後方互換性は考慮せず、あるべき設計に寄せる方針になった。

## 決定

- data contract としての `kind` は削除する。
- `format` ではなく `datatype` を導入する。
- `datatype` は任意の string とし、MVP では registry や別 ID を持たない。
- Python decorator の input slot は要求 `datatype`、output slot は生成 `datatype` を宣言する。
- generated managed file / dataset は output slot の `datatype` を metadata に保存する。
- `.idts` は拡張子 / representation で識別し、datatype そのものとはしない。
- input picker では datatype 完全一致を強い候補とし、extension / representation が合うものは fallback 候補にできる。
- skill や user 定義の datatype は `{userId}/xxxx` のような namespace を推奨するが、enforcement はしない。
- app settings に任意の `userId` を追加し、LLM が datatype 名を提案するときの prefix として使えるようにする。

## 実装

- shared 型、workspace 型、metadata、data-note frontmatter、`.idts` manifest view を `datatype` に更新した。
- Python SDK、sample block、Shimadzu LC sample、block 実装 skill、inline AI prompt を `datatype` 契約へ更新した。
- input picker / dataset picker / process chain / manifest viewer / AI context の表示と matching を `datatype` に変更した。
- 設定 dialog に `userId` を追加し、空白と file / folder 名に使えない文字を拒否する validation を入れた。
- 仕様文書と `docs/Issues.md` を `datatype` 方針に更新し、Issue 45 を完了扱いにした。

## 確認

- `npm run build` が成功した。
- 旧 field 名の検索で、data contract として残る `formatId` / `acceptedKinds` / `producedKind` / `slot.format` / `bundle/idts` などが無いことを確認した。
- renderer の `alert` / `confirm` / `prompt` 使用が無いことを確認した。
