## テーマ

- ELN 上で binary 分析データをアップロードし、plugin で描画・解析・chain 実行したい。
- とくに `Python実行Plugin` を主軸にしたい。

## 話したこと

### 1. 先に整理した設計案

- 任意ファイル添付、成果物、解析実行を区別して扱う案を整理した。
- そのための共通語彙として、次を置く案を出した。
  - `Attachment`
  - `Artifact`
  - `Job / Step`
- ノート上では raw path ではなく参照 block を埋め込む案を出した。
  - `Core.AttachmentRef`
  - `Core.ArtifactViewer`
  - `Core.AnalysisStep` / `Core.AnalysisChain`
- plugin 契約は、今の `block描画 + 単発action` だけでは足りないため、
  - viewer contribution
  - runner contribution
  - step 実行結果としての output artifact 情報
  を足す必要がある、という説明をした。

### 2. なぜ plugin 契約拡張が必要かの説明

- 現在の契約には「ファイル」や「成果物」の概念がない。
- 現在の host action result は `summary` / `logLines` 中心で、chain 実行の入力として弱い。
- renderer に渡せる情報も block JSON 中心で、viewer が参照対象を理解するには足りない。
- `Python実行Plugin` を主軸にする場合、install 済み plugin と workspace 上の `.py` を分けて扱う必要がある。
- plugin 間連携には、少なくとも output の意味を共有するための成果物契約が必要、という整理をした。

### 3. ユーザーから出たシンプル案

ユーザーから、次のようにもっと単純化できるのでは、という提案があった。

- upload しても workspace 内に置くだけ
- 中間処理しても処理後ファイルを workspace 内に置くだけ
- 解析しても input path を受けて output file を workspace 内に置くだけ

これを受けて、MVP は path ベースでかなり簡素化できるという話をした。

- upload = workspace 配下へコピー
- viewer = path を受けて読む
- analysis = input path 群を受けて output path 群を書く
- chain = 前段 output path を次段 input path に渡す

## 現時点の整理

- Attachment / Artifact / Job を明示する
- ノートは ID 参照でつなぐ
- 将来の拡張や監査、plugin 間連携には強い

- plugin 契約の拡張も最小限にする
- MVP は作りやすい
- ただし rename / move / 由来追跡 / 掃除には弱い
