## 会話要約

- `Artifact -> DataCatalog`、`Blob -> RawData` へ、ドキュメント・ソースコード・App 表示・フォルダ名まで含めて全体 rename したい、という相談があった。
- まずはアイデア自体の妥当性評価を求められた。

## 確認したこと

- 現行定義では、
  - `blob` は「ユーザーが持ち込む元データの永続単位」
  - `artifact note` は「ノート上でデータを参照するための Markdown facade」
  となっている。
- `blob` には raw measurement file だけでなく、手で追加した CSV / text や directory も含む。
- `Artifacts/`, `.blob/`, `artifactRelativePath`, `BlobPickerDialog` など、命名は docs / code / workspace layout に広く浸透している。
- `artifact` は plugin 配布物の意味でも使われており、用語衝突は実際にある。

## 判断

- 方向性として「技術用語よりユーザー語彙へ寄せる」は妥当。
- ただし、
  - `Artifact -> DataCatalog`
    - folder / 機能名としては分かりやすい
    - ただし 1 ファイル単位の `artifact note` をそのまま `DataCatalog` と呼ぶと、collection 名っぽさが強く少しズレる
  - `Blob -> RawData`
    - UI 上では直感的になりうる
    - ただし現行 blob の定義より意味が狭く、加工済み CSV や任意投入テキストまで含む現在の責務とは少し合わない
  と評価した。

## 推奨

- `Artifact` は rename した方がよい可能性が高い。
- `Blob` は `RawData` より `SourceData` または `ImportedData` の方が定義と齟齬が少ない。
- `DataCatalog` を採るなら、
  - collection/folder 名として `DataCatalog`
  - 個別 Markdown は `data entry` / `data note`
  のように分ける案が自然。
- 未リリースなので全体 rename はまだ可能だが、影響範囲が広いため
  1. 用語決定
  2. UI 表示 rename
  3. docs rename
  4. path / code / migration rename
  の段階実施が安全、と提案した。
