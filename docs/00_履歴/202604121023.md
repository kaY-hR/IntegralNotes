## 背景

- ELN 側で `blob / artifact.md / workspace(or run)` の責務分離を議論していたが、本体責務が膨らみ始めた。
- 既存の別ソフト `Integral` が、分析実行専用の責務をどのように切り出しているかを確認した。

## 調査内容

- `C:\Users\shimadzu\Desktop\_Root\10_Integral\Integral` を確認した。
- 要件・設計文書、Current Spec、`workspace.ts` / `execution.ts`、TemplateWorkflow / TemplateProcessBox の実装を重点的に読んだ。

## 確認できたこと

- `Integral` は `Workspace` と `Storage` を明確に分けている。
  - `Workspace` は実行単位の mutable な作業場
  - `Storage` は長期保存用
- `DataBox` はフォルダ単位のデータ容器で、実行前にクリアされうる中間生成物として扱われる。
  - `.integral-keep` がある DataBox は保持される
  - `DB` は `.integral-redirect` により Storage 配下の実体を参照する
- `ProcessBox` は
  - `.integral-process-master`
  - `.integral-process-workspace`
  に分離されており、定義と実行時状態が分かれている。
- 単体 Process 実行では `TempRuns` の一時 Workspace を切っており、Template 実行では `TemplateExecuteSpace` に Run を複製して read-only 化している。
- Process 実行は folder path 契約が中心で、Template の `utils.py` も「接続先 Box のフォルダパス解決」を前提にしている。

## ELN 側への示唆

- `Integral` の設計は、今回 ELN で議論している
  - mutable な作業場
  - 永続データ
  - 実行ごとの隔離
  - 実行定義と実行時状態の分離
  とかなり近い。
- 一方で、`Integral` は main process / IPC / ファイルコピー / redirect解決 / 実行前クリア / watch まで本体責務として持っており、責務はかなり大きい。
- そのため、ELN 側で同じものを本体へ直接入れるより、
  - ELN: note / artifact.md / frontmatter / 参照UI
  - IntegralAnalysis相当モジュール: run workspace / chain 実行 / path 解決 / 隔離実行
  の分離を考える価値が高い。

## 暫定コメント

- `Integral` で既に成立している「Workspace は mutable、Storage は persistent、Run は複製して隔離」という考え方は、ELN 側でもかなり有効。
- ただし `Integral` の path/folder 契約をそのまま ELN 本体へ持ち込むと、本体が重くなりすぎる懸念が強い。
- したがって、ELN では
  - `blob + artifact.md` を正式データ
  - `run workspace` を解析モジュール側の責務
  として分ける方向が筋が良い。
