## 会話要約

- 後から専用 viewer を追加できる viewer plugin system の設計パターンを検討した。
- 候補として、特殊な Markdown、特殊な JSON、拡張子ベースの認識を比較した。

## 整理した論点

- 現状の app は block plugin と viewer を別系統で持っており、viewer 側は標準では拡張子ベースで判定している。
- 要件上も `html / image / text` の標準表示は拡張子ベースで成立している。
- そのため、viewer plugin は「既存の標準 viewer を置き換える」より「標準 viewer の上に特化 viewer を追加する」方が自然。

## 推奨方針

- viewer の基本ルーティングは file-oriented にする。
- 第一候補は拡張子ベース。
- ただし `.json` や `.md` のような汎用拡張子は、拡張子だけで plugin に振らず、frontmatter や schema marker で明示 opt-in させる。
- Markdown 本文の特殊記法は viewer 選択の主経路にはせず、必要なら embed 記法や note widget として別機能に分ける。

## 補足

- 既存の external plugin renderer は `iframe` ベースなので、viewer plugin でも流用しやすい。
- 実装するなら plugin manifest に viewer contribution を追加する方向がよさそう、という前提で整理した。
