# ノート画像添付

## 目的

Milkdown 上で画像を挿入・貼り付けしたとき、Markdown source に `blob:` URL を残さず、workspace 内 file として永続化する。  
これにより、app 再起動後も同じ画像を再表示できるようにする。

この文書は次を定義する。

- note 本文に保存する画像記法
- 画像 file の保存先と命名規則
- Milkdown の image insert / paste / drop を束ねる upload hook 方針
- editor 再表示時の DOM 用 URL 解決

## 前提

Markdown の

```md
![alt](target)
```

は標準 Markdown の image 記法であり、IntegralNotes 独自 flavor ではない。  
通常 link の `[label](target)` と別物であり、画像表示にはこちらを使う。

## スコープ

対象:

- Milkdown の image insert UI
- note editor 上での clipboard paste
- note editor 上での file drop
- note 本文に保存する image source
- editor 再表示時の image 描画
- IntegralNotes 内 rename / move 時の image target 追従

対象外:

- explorer 上での file / directory / image 貼り付けの具体仕様
- 保存先 folder を settings で変更する機能
- image 以外の任意 binary attachment
- 外部 URL `https://...` の画像取得や mirror
- 既に保存済みで壊れている `blob:` URL の自動復元

最後の点は重要で、`blob:` URL はその時点の renderer process にだけ生きる一時 URL である。  
app 再起動後は元の Blob 自体が消えるため、source に `blob:` 文字列だけが残っていても、後から完全復元することはできない。

## 基本方針

- source 上の記法は標準 Markdown image 記法 `![alt](target)` を使う
- app が自動挿入する canonical form は `![](/Data/yyyyMMdd-HHmm-RRR.ext)` とする
- app の resolver は `/path/from/cwd` と `path/from/cwd` の両方を許容する
- 自動保存先は workspace root 配下の `Data/` とする
- `blob:` URL や `URL.createObjectURL()` の戻り値は source に残さない
- 画像の永続化は note save 時ではなく、upload 発生時に完了させる

最後の方針により、画像 bytes を確実に取得できるタイミングで file 保存を済ませる。

## 保存記法

### canonical form

app が自動挿入する source は次の形式とする。

```md
![](/Data/20260414-1730-123.png)
```

MVP では alt text は自動生成しない。  
必要ならユーザーが後で `![peak chart](...)` のように編集してよい。

### 読み取り時に許容する form

app は次も同じ target として扱う。

```md
![](Data/20260414-1730-123.png)
![](/images/reference.png)
```

つまり:

- 自動出力は先頭 `/` 付きへ統一する
- 読み取りは `/` あり / なし両方を受ける
- 自動保存先は `Data/` だが、手書き image link は workspace 内の他 path も許容する

## 保存先

### 既定 folder

- note editor から挿入した画像は `Data/` 配下に保存する
- `Data/` が無ければ app が自動作成する

### `Data/` 内での位置づけ

- ここに置く画像は ordinary workspace file である
- 特別な alias ではない
- image attachment 専用の別 metadata は持たない
- 他の workspace file と同様に managed file tracking 対象になりうる

## file 名規則

保存 file 名は次の形式とする。

```text
yyyyMMdd-HHmm-RRR.ext
```

例:

```text
Data/20260414-1730-123.png
Data/20260414-1731-004.jpg
```

規則:

- `yyyyMMdd-HHmm` はローカル時刻
- `RRR` は 3 桁の random 数
- `ext` は image format に応じて決める

拡張子:

- file picker 由来なら元 file の拡張子を優先する
- clipboard image のように file 名が無い場合は mime type から決める
- 判定不能なら `.png` を fallback とする

同名衝突時:

- 別の `RRR` を振り直して再試行する

## Milkdown 統合方針

### 共通 upload hook を使う

Milkdown / Crepe では image insert UI と paste / drop が別 UI に見えても、upload 処理自体は共通 hook へ寄せられる。  
IntegralNotes では、この共通 upload hook を 1 つ差し替えて永続化を行う。

方針:

- `Crepe` の image feature config で `onUpload` を差し替える
- inline image と image block の両方に同じ永続化 policy を適用する
- paste / drop も同じ upload policy を通す

これにより、entry point ごとに別々の保存ロジックを持たない。

## HTML clipboard 画像

clipboard に `text/html` と image file が同居するケースでも file uploader を通す。  
そのため、Milkdown の upload plugin では HTML clipboard 画像を無視せず吸収できる設定を有効にする。

## upload hook の責務

upload hook は次を行う。

1. 受け取った `File` から保存先 path を決める
2. `Data/` 配下へ実 file を書く
3. Markdown source 用の canonical target `/Data/...` を返す

ここで返す値が、そのまま note 本文に残る `src` になる。  
したがって、戻り値は `blob:` ではなく workspace path でなければならない。

## editor 上の再表示

Markdown source に `/Data/...` を保存しても、そのまま DOM の `img.src` に入れると browser は workspace file として解釈できない。  
そのため editor 描画時には、source 上の workspace path を DOM 用の loadable URL へ変換する必要がある。

方針:

- source 上は `/Data/...` のまま保持する
- editor 描画時だけ、workspace path を absolute file URL または同等の app 内 URL へ変換する
- この変換は Milkdown の image URL proxy 機構で一元的に行う

これにより:

- source は人間に読める workspace path のまま残る
- 再オープン後も画像が表示できる
- 同じ source を rename / move 更新の対象にもできる

## rename / move 追従

- IntegralNotes 内の explorer 操作で image file を rename / move した場合、`cwd` 配下 `.md` 内の image target も更新する
- 対象は `/old/path` と `old/path` の両方
- note file 自体の move では、root-relative path なので image target を変えなくてよい

この更新は `docs/30_設計/60_ノートリンク記法.md` で定義する Markdown link 更新と同じ hook に寄せてよい。  
違うのは source 記法が `[label](...)` ではなく `![alt](...)` である点だけである。

## explorer 貼り付けとの分離

explorer 上の `貼り付け` で clipboard image を file 化する機能は別問題である。  
それは tree 操作としての file 作成であり、note editor への image attachment とは責務が違う。

つまり:

- explorer 貼り付けは `workspace tree 操作`
- note image attach は `Milkdown upload`

として別 Issue / 別仕様で扱う。

## 将来拡張

将来の候補:

- alt text の自動生成
- 保存先 folder の設定 UI
- image 以外の attachment
- drag-drop 以外の external resource 取り込み
- 既存の `blob:` source を、同一 session 中だけ救済する migration

ただし MVP では、`Data/` 固定 + workspace path 保存 + 再オープン後の再表示を優先する。
