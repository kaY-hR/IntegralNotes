# プラグイン定義

## plugin の種別

MVP では plugin を次の 3 種に分ける。

- core plugin
- general-analysis plugin
- device / domain plugin

## core plugin

app に内蔵される plugin。

例:

- `core-display`

役割:

- 標準表示 block の提供

## general-analysis plugin

固定 ID:

- `general-analysis`

役割:

- `.py-scripts/` を走査する
- `PYS-...` を block-type として公開する
- Python 実行を行う

## device / domain plugin

例:

- `shimadzu-lc`

役割:

- 独自 UI の提供
- 装置通信
- 固有処理の実行

## plugin が持つべき情報

### 共通

- `id`
- `displayName`
- `description`
- `version`

### 静的 block-type plugin

- block-type 一覧
- 必要なら slot 定義
- 必要なら settings UI

### 動的 block-type plugin

- workspace を見て block-type 一覧を生成する手段

`general-analysis` はこれに当たる。

## block-type 定義

### Python の場合

block-type は `PYS-...`

### 装置 plugin の場合

plugin 内で stable な ID

例:

- `run-sequence`
- `export-chromatogram`

## settings UI

plugin は必要なら custom settings UI を持ってよい。  
ただし最終的に編集されるのは共通 block schema である。

## result UI

MVP では result UI override を共通契約にしない。  
結果は原則として output dataset に書き、標準表示 block で閲覧する。

## general-analysis の走査対象

```text
Workspace/
  .py-scripts/
    PYS-7K2M9Q4D/
      script.json
      ...
```

走査条件:

- `script.json` が存在する
- `scriptId` が `PYS-...`
- `entry` が存在する

## kind 制約

slot 定義に `acceptedKinds` / `producedKind` を持てるが、MVP では strict には使わない。  
将来の GUI 候補絞り込みや validation に使う前提で保持する。
