# プラグイン定義

## plugin system の基本方針

IntegralNotes の plugin system は 1 つに統合する。  
配布、install/uninstall、manifest、registry は共通化し、plugin package を複数系統に分けない。

ただし plugin が提供する contribution type は分ける。

- `blocks`
- `viewers`

つまり「plugin system は統合、runtime contract は contribution ごとに分離」とする。

## plugin の種別

MVP では plugin を次の 3 種に分ける。

- core plugin
- general-analysis plugin
- device / domain plugin

これらは install 単位の別 system ではなく、同一 plugin system 上の役割分類である。

## core plugin

app に内蔵される plugin。

例:

- `core-display`

役割:

- 標準表示 block の提供
- built-in viewer の既定動作の提供

## general-analysis plugin

固定 ID:

- `general-analysis`

役割:

- workspace の `.py` を走査する
- decorator 付き関数を block-type として公開する
- Python 実行を行う

Python block-type の canonical 形は `relative/path.py:function` とする。

## device / domain plugin

例:

- `shimadzu-lc`

役割:

- 独自 UI の提供
- 装置通信
- 固有処理の実行
- 必要なら固有拡張子向け viewer の提供

## plugin manifest が持つべき情報

### 共通

- `apiVersion`
- `id`
- `namespace`
- `displayName`
- `description`
- `version`

### contribution

- `blocks?`
- `viewers?`

plugin package は少なくとも `blocks` または `viewers` のどちらか 1 つ以上を持つ。  
既存 plugin 互換性のため、`viewers` は optional とする。

### runtime / UI

- `host?`
- `renderer?`

`host` は主に block action 実行に使う。  
`renderer` は block custom UI や viewer UI の asset 参照元として使ってよい。

## block contribution

### block-type plugin

- block-type 一覧
- 必要なら slot 定義
- 必要なら settings UI

### 動的 block-type plugin

- workspace を見て block-type 一覧を生成する手段

`general-analysis` はこれに当たる。

### block-type 定義

#### Python の場合

block-type は `relative/path.py:function`

#### 装置 plugin の場合

plugin 内で stable な ID

例:

- `run-sequence`
- `export-chromatogram`

### settings UI

plugin は必要なら custom settings UI を持ってよい。  
ただし最終的に編集されるのは共通 block schema である。

## viewer contribution

viewer contribution は、workspace file または dataset 内 file を read-only に描画するための拡張点である。

### first version の方針

- matcher は extension-only に限定する
- `extensions: [".foo", ".bar"]` の配列で解決する
- `.md` の note editor / note tab override は first version の対象外

特殊な JSON schema や Markdown frontmatter での viewer 切替は、必要になったら後から matcher を拡張する。

### viewer が持つべき情報

- `id`
- `displayName`
- `extensions`
- `description`
- `renderer`

`renderer` は first version では `iframe` を前提にしてよい。

### viewer 解決順

file を app 内で表示するときは次の順で解決する。

1. 拡張子に一致する plugin viewer
2. built-in viewer
3. unsupported / 外部アプリ導線

この順序にすることで、既存 plugin が `viewers` を持たなくても挙動は変わらない。

### viewer の責務

- file path または file content を受けて read-only preview を描画する
- file を直接編集しない
- file 自体が `.idts` でも独自拡張子でも、表示契約は同じにする
- app から `presentation: "full" | "embed"` の表示文脈が渡されたら、それに応じて UI 密度を変えてよい

viewer plugin は「結果を生成する plugin」ではなく「既存ファイルを表示する plugin」である。  
結果生成は従来どおり処理 plugin / 可視化 plugin が担う。

### embed での扱い

- note 本文の `![]()` 埋め込みでも同じ viewer contribution を使ってよい
- 初版の embed は preview-only とし、plugin iframe の直接操作は前提にしない
- ユーザーが詳細操作したい場合は embed を click して通常の file tab を開く

## result UI

block の result UI override は共通契約にしない。  
結果は原則として output dataset に書き、標準表示 block または viewer contribution で閲覧する。

## general-analysis の走査対象

```text
Workspace/
  scripts/
    pca.py
    normalize.py
```

走査条件:

- `cwd` 配下の `.py` である
- decorator 付き関数がある
- function 名と slot 定義が読める

## kind 制約

slot 定義に `acceptedKinds` / `producedKind` を持てるが、MVP では strict には使わない。  
将来の GUI 候補絞り込みや validation に使う前提で保持する。
