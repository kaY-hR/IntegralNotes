# ノートリンク記法

## 目的

通常ノート本文から、workspace 内の file を自然に参照・移動できるようにする。  
MVP では Obsidian 風 wiki link ではなく、標準 Markdown link を使う。

この文書は次を定義する。

- note 本文に保存するリンク記法
- Milkdown 上での補完 UX
- link click 時の open 挙動
- app 内 rename / move 時の link 追従

## スコープ

対象:

- 通常 note 本文から workspace 内 file へ張る link
- `.md`, `html`, `image`, `text`, `unsupported` を含む workspace file 全般
- Milkdown 上での link 挿入補完
- IntegralNotes 内での rename / move に対する link 更新

対象外:

- system-managed data-note の open 導線
- Obsidian 風 wiki link `[[...]]`
- 標準 Markdown image 記法 `![alt](target)` の upload / persist / rendering 詳細
- heading link の補完
- 外部ツールや OS 上での rename / move 追従
- `LC バイナリを link したら display block を出す` ような高度な link action

後者は将来拡張として別途扱う。

## 基本方針

- source 上の記法は標準 Markdown link `[label](target)` を使う
- app が自動挿入する canonical form は `[label](/path/from/cwd)` とする
- app の link resolver は `/path/from/cwd` と `path/from/cwd` の両方を許容する
- target は document 相対ではなく `cwd` ルート相対で解釈する

この path 規則は一般 Markdown viewer 互換よりも、workspace 内 rename / move 追従のしやすさを優先したものである。

## 対象ファイル

- link target は `cwd` 配下の file とする
- directory は MVP の link target に含めない
- 補完候補は `cwd` 配下の全 file を対象とする

## 保存記法

### canonical form

app が補完で挿入する source は次の形式とする。

```md
[Experiment A](/notes/Experiment A.md)
[sample.csv](/data/sample.csv)
```

### 読み取り時に許容する form

app は次も同じ target として扱う。

```md
[Experiment A](notes/Experiment A.md)
[sample.csv](data/sample.csv)
```

つまり:

- 出力は先頭 `/` 付きへ統一する
- 読み取りは `/` あり / なし両方を受ける

## label 生成規則

補完候補を選んだときの label は次の規則で自動生成する。

- `.md` は拡張子を除いた basename
- それ以外は拡張子を含む basename

例:

- `notes/Experiment A.md` -> `[Experiment A](/notes/Experiment A.md)`
- `data/sample.csv` -> `[sample.csv](/data/sample.csv)`
- `images/peak.png` -> `[peak.png](/images/peak.png)`

ユーザーは挿入後に label を自由に書き換えてよい。

## 補完 UX

### trigger

- Milkdown 上で `[` を打った時点で候補一覧を開く

### 候補一覧

- 主表示は file 名
- 補足表示は `cwd` ルート相対 path
- 同名 file が複数ある場合も、主表示 + 補足 path で区別する

例:

- 主表示: `Experiment A.md`
- 補足: `notes/Experiment A.md`

### 選択時の挙動

- 候補を選ぶと `[label](/path)` をその場に挿入する
- `.md` では label は拡張子なし
- 非 `.md` では label は拡張子あり

### plain Markdown としての継続

- 補完を使わず手で `[label](target)` を書いてもよい
- 補完を閉じた場合も、通常の Markdown 編集は継続できる

## link 解決規則

### path 解決

- `target` が `/foo/bar.md` なら、workspace root から `foo/bar.md` を引く
- `target` が `foo/bar.md` でも、workspace root から `foo/bar.md` を引く
- `http://`, `https://`, `mailto:` など明示的な外部 scheme はこの仕様の対象外とする

### 存在しない target

- file が存在しない場合は app 内 open は行わず、error として扱う
- UI 上の具体表示は実装時に決めるが、少なくとも silent ignore はしない

## click 時の open 挙動

link click 時は target file の kind に応じて既存 open 経路を使う。

### Markdown

- `.md` は main app 内 tab として開く

### Renderable / text

- `html / image / text` は既存 viewer 経路で開く

### Unsupported

- main app 内で表示できない file は、既存の `外部アプリで開く` 導線に寄せる

つまり、note link は新しい viewer を持たず、workspace file を開く既存導線の薄い入口として実装する。

## rename / move 追従

### 対象

- IntegralNotes 内の explorer 操作から行われた rename / move のみ対象
- 外部ツールや OS 上の変更は追従対象外

### 更新対象

- `cwd` 配下の全 `.md` を走査して link target を更新する

### 置換対象

- `/old/path`
- `old/path`

の両方を `new/path` へ更新する。

### 前提

- `cwd` ルート相対 path で保存するため、document 自体の移動では link target を変えなくてよい
- target file の rename / move だけを追えば十分

## heading link は MVP 対象外

MVP では file 単位の link のみ扱う。

扱わないもの:

- `[label](/note.md#heading)` の補完
- heading slug 化規則
- heading rename 追従

これは file link より一段重い問題なので、別段階で扱う。

## data-note は別導線

data-note は `.store/.integral/data-notes/{id}.md` に置かれる system-managed note であり、この文書の Markdown link 補完仕様には含めない。

方針:

- 通常 note の link 補完対象は `cwd` 配下の user-managed workspace file に限る
- data-note は data ID から app が専用導線で開く
- data-note を file path としてユーザーに管理させない

## 将来拡張

将来の候補:

- heading link 補完
- link 挿入時の fuzzy search 強化
- link target file の rename / move 以外の整合性チェック
- file kind に応じた高度 action
  - 例: `LC バイナリ file` の link から display block 挿入
- data-note への専用 open action

ただし MVP では、link は既存 open 導線への入口に留める。
