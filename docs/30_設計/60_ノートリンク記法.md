# ノートリンク記法

## 目的

`Issue 21` と `Issue 22` はどちらも「note 本文から何かへ辿れるようにしたい」という話だが、実は同じ問題ではない。  
この文書では、選択肢と判断材料を先に整理し、意思決定をしやすくする。

## まず分けて考える論点

### 1. 通常 note 同士の内部リンク

対象:

- `notes/a.md` から `notes/b.md` を開く
- heading 付きリンクで note 内位置へ飛ぶ

候補記法:

- `[[b]]`
- `[[notes/b.md]]`
- `[b](notes/b.md)`

### 2. data-note から canonical data へのリンク

対象:

- `data-catalog/*.md` から `.store/{datasetId}/` や alias path へ辿る

候補記法:

- `[dataset 実体](../.store/DTS-.../)`
- `[[../.store/DTS-.../|dataset 実体]]`
- app 専用 button / command

### 3. app 内での click 挙動

候補:

- workspace 内 `.md` は app 内 tab で開く
- renderable file は app 内 viewer で開く
- それ以外は OS 既定アプリで開く
- クリック可能にはせず、将来対応に回す

### 4. 保存時の canonical format

候補:

- source に書かれた記法をそのまま保持する
- `[[...]]` を標準 Markdown link へ変換して保存する
- 標準 Markdown link を `[[...]]` へ変換して保存する

raw mode を将来入れる前提なら、source の文字列表現を安定に往復できる方式が望ましい。

## 選択肢

### A. 標準 Markdown link を正とする

例:

- `[Experiment A](notes/Experiment A.md)`
- `[dataset 実体](../.store/DTS-1234/)`

利点:

- GitHub, VS Code, 一般 Markdown viewer で最も自然
- Milkdown 側は既存 link 機能を流用しやすい
- parse / serialize の custom syntax が不要
- raw markdown mode と競合しにくい

弱点:

- Obsidian 風の軽い記法にはならない
- shortest path や alias の UX は app 側 helper が要る
- Obsidian との source 互換は弱い

向いている場合:

- 可搬性を最優先したい
- editor 拡張コストを抑えたい

### B. 通常 note だけ wiki link を使い、data link は分ける

例:

- `[[Experiment A]]`
- `[[Experiment A#考察]]`
- `[dataset 実体](../.store/DTS-1234/)`

利点:

- note-first の内部リンク体験を改善しつつ、filesystem / canonical data link は標準的に保てる
- `wiki note link` と `data 実体 link` の責務を混ぜずに済む
- `Issue 22` を単独で進めやすい

弱点:

- link 記法が 2 種類になる
- app 内 link resolver と data-note template の両方を考える必要がある

向いている場合:

- Obsidian 風 UX は欲しいが、`.store` へのリンクまで wiki 記法に背負わせたくない
- まず通常 note での内部リンクだけ切り出して実装したい

### C. note も data link も wiki link 記法へ寄せる

例:

- `[[Experiment A]]`
- `[[../.store/DTS-1234/|dataset 実体]]`

利点:

- 見た目の記法を統一しやすい
- Obsidian 風の source 体験には最も寄る

弱点:

- `wiki link` と `filesystem path link` の意味が混ざる
- `.store` path, folder, heading, alias を同じ resolver で扱う必要があり複雑
- 一般 Markdown viewer では非標準記法になりやすい
- `Issue 21` と `Issue 22` が強く結合し、先に進めにくい

向いている場合:

- Obsidian 互換を最優先する
- path 解決ルールと custom syntax 保守を app 側で強く持つ前提を許容する

## 判断材料

### 1. 外部ツール互換

確認したいこと:

- GitHub preview で意味が通るか
- VS Code の markdown preview で意味が通るか
- Obsidian へ貼ったとき期待どおりか

### 2. Milkdown 拡張量

現状:

- `src/renderer/MilkdownEditor.tsx` は `Crepe` を素の構成で起動している
- `@milkdown/crepe` の標準は `commonmark` + `gfm`

追加で必要になりうるもの:

- `remark` plugin
- `parseMarkdown` / `toMarkdown` 拡張
- ProseMirror schema
- click handler
- autocomplete / rename 追従をやるなら resolver と index

### 3. link 解決規則

決める必要がある:

- `[[aaa]]` を `basename` で引くか
- `[[folder/aaa.md]]` を `relative path` として扱うか
- duplicate file name があるときどうするか
- rename / move 後に link を自動更新するか

### 4. raw markdown mode との整合

`Issue 3` と関係する。  
wiki link を入れるなら、WYSIWYG と raw を往復しても source が壊れないことを優先したい。

### 5. `Issue 21` との境界

判断ポイント:

- `note to note` link と `note to data` link を同じ syntax に載せる必要が本当にあるか
- data-note には `本文リンク` ではなく `Open dataset` button の方が自然ではないか

## 決める順番

1. `Issue 22` を通常 note 同士の内部リンクに限定するか決める
2. `[[target]]` の解決単位を `basename / relative path / ID` のどれにするか決める
3. MVP で `表示だけ / クリック可能 / autocomplete・rename 追従` のどこまで入れるか決める
4. `Issue 21` の data-note canonical link を wiki 記法へ寄せるか、標準 Markdown link か、button 導線にするか決める

## 現時点の暫定推奨

まずは `選択肢 B` が最も扱いやすい。

理由:

- `wiki note link` と `canonical data path link` は意味が違う
- `Issue 22` を通常 note 向けの editor 拡張として独立に進められる
- `Issue 21` は data-note template / viewer / app 内導線の問題として別に最適化できる

つまり、最初に決めるべきは「Obsidian 風 wiki link を通常 note の内部リンクとして採るか」であり、`.store` へのリンク表現まで一緒に決めない方がよい。
