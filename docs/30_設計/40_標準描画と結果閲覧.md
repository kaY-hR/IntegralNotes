# 標準描画と結果閲覧

## 位置づけ

MVP では、renderable なファイルの閲覧は app の標準機能とする。  
可視化 plugin は「renderable でない dataset を renderable dataset に変換する処理 plugin」として扱う。

加えて、標準 viewer だけでは不足する場合に備え、拡張子ごとの専用 viewer plugin を追加できるようにする。  
viewer plugin は既存の plugin system に統合し、`viewers` contribution として提供する。

### Dataset Manifest Entry

- `.idts`

加えて、拡張子が未登録でも text と読める file は built-in text viewer で扱ってよい。

`.idts` は raw text viewer や専用 manifest viewer ではなく、dataset の data-note へ入る entrypoint として扱う。
workspace file として `.idts` を開いた場合、app は manifest から dataset / note target を解決し、その dataset の system-managed data-note を note tab として開く。

manifest の詳細は内部解決に使ってよいが、ユーザー向けの第一表示は data-note 本文である。

dataset の中身を結果表示として見る場合は、`.idts` file tab ではなく dataset ID から readable directory path を resolve し、表示 block / viewer registry 側で folder 内 file を読む。

## 専用 viewer plugin

### 初版の matcher

- 拡張子ベースのみ

例:

- `.spc`
- `.abf`
- `.shim-result`

### 役割

- built-in viewer では不足する file を app 内で read-only preview する
- dataset 表示と workspace file 表示の両方で同じ viewer registry を使う
- note 本文の `![](/path)` embed preview でも、基本は同じ viewer registry を使えるようにする
- ただし `.idts` だけは例外的に data-note 表示へ流す

### 初版で扱わないもの

- JSON schema 判定
- Markdown frontmatter 判定
- note editor / note tab の override

`.md` は引き続き note tab で扱い、viewer plugin の初版対象には含めない。

## Image Compare Viewer

`.icv` は位置合わせ済み画像を比較するための JSON manifest として扱う。  
`.idts` と同じく複数 file を 1 つの入口から扱うが、`.idts` が dataset / data-note entrypoint であるのに対し、`.icv` は viewer plugin (`plugins/image-compare-viewer`) で file tab / embed 上に画像比較 UI を表示する。

`.icv` は built-in viewer ではなく、通常の `viewers` contribution として提供する。  
そのため runtime で表示するには `image-compare-viewer` plugin が installed plugin root に install されている必要がある。

最小形:

```json
{
  "name": "Cell image comparison",
  "images": [
    { "id": "rgb", "label": "RGB", "path": "rgb.png" },
    { "id": "uv", "label": "UV", "path": "uv.png", "opacity": 0.65 }
  ]
}
```

`images` の代わりに `layers` または `members` を使ってよい。  
各 layer は次を持てる。

- `path` / `relativePath` / `src` / `file`: workspace 内画像 path。manifest からの相対 path も許可する。
- `id`
- `label` / `name` / `displayName`
- `opacity`: `0.0` から `1.0`。`0` から `100` の値も percent として解釈する。
- `scale`
- `offsetX`
- `offsetY`
- `rotation` / `rotate` / `rotationDeg`

manifest root には任意で `width` / `height` または `canvas.width` / `canvas.height` を持てる。  
未指定の場合は、読み込めた画像の natural size を表示 canvas として使う。

### embed preview

workspace 内 file を `![](/path/from/cwd)` で埋め込むときも、viewer 解決順は file tab と揃える。

1. `.idts` なら path から managed data ID / note target を解決し、対応する data-note Markdown を表示する
2. それ以外で拡張子に一致する plugin viewer
3. それ以外で built-in viewer
4. 未対応として扱う

embed は read-only context だが、preview 自体は通常の iframe / preview として描画し、少なくとも scroll と text 選択を阻害しない。  
そのため host 側は surface 全体に click hitbox を被せず、iframe や markdown preview の pointer event を奪わない。

block 実行後や data-note 追記で app が自動挿入した `![]()` も、通常の note embed と同じ viewer 解決規則で扱う。  
事前に「表示可能か」を判定せず、standard embed / managed data note fallback の既存解決に委ねる。

workspace target を持つ embed には、右上に flat な `[別タブで開く]` button を出す。  
通常の workspace file は file tab、managed data note fallback は note tab をこの button から開く。

`.idts` embed の `[別タブで開く]` も、manifest file ではなく対応する data-note tab を開く。

### viewer plugin からの関連 workspace file 解決

viewer plugin は `integral:viewer-resolve-workspace-file-url` を parent window へ `postMessage` し、manifest から参照している workspace file を data URL として取得できる。

request:

```json
{
  "type": "integral:viewer-resolve-workspace-file-url",
  "payload": {
    "requestId": "request-1",
    "path": "overlay.png"
  }
}
```

host は request 元 viewer file の path を基準に相対 path を解決し、workspace 外への escape は `WorkspaceService.resolveWorkspaceFileUrl()` 側で拒否する。

response:

```json
{
  "type": "integral:viewer-resolve-workspace-file-url-result",
  "payload": {
    "requestId": "request-1",
    "path": "overlay.png",
    "relativePath": "Data/overlay.png",
    "url": "data:image/png;base64,..."
  }
}
```

失敗時は `payload.error` を返す。

## 表示 block

標準表示も共通 block schema で表す。

```json
{
  "id": "BLK-7A8B9C0D",
  "plugin": "core-display",
  "block-type": "dataset-view",
  "params": {},
  "inputs": {
    "source": "DTS-9X4Q2M1A"
  },
  "outputs": {}
}
```

## 表示ルール

- input slot `source` に指定された dataset を読む
- app は input slot `source` の managed data ID を readable directory path に resolve する
- resolve 結果の directory 内ファイルを走査する
- file ごとに viewer を解決する
- 表示可能な file を flex-layout で表示する

手動作成 dataset と block 生成 dataset は同じ viewer contract で扱い、dataset の current path や representation は viewer 側に漏らさない。

### viewer 解決順

1. 拡張子に一致する plugin viewer
2. built-in viewer
3. 未対応として扱う

dataset viewer では、どの file も 1 つの route で解決する。  
「built-in renderable だけを先に拾う」のではなく、plugin viewer と built-in viewer の両方を同じレイヤーで判定する。

### 並び順

MVP では固定順でよい。  
特に意味を持たせない。

## 対象ファイルがない場合

- 「表示可能なファイルがありません」を出す
- dataset 内に何があるかの一覧は別途見られるようにしてよい

## ノート内リンクから file を開く場合

通常 note 本文の Markdown link から workspace file を開く場合も、既存の file 表示能力を使う。

- `.md` は note tab で開く
- 非 `.md` file は、まず plugin viewer を探す
- plugin viewer が無ければ built-in viewer を使う
- どちらも無ければ外部アプリ導線へ寄せる

つまり、note link は viewer の新形式ではなく、既存の workspace file open 導線の入口である。  
詳細は `docs/30_設計/60_ノートリンク記法.md` を参照。

## 可視化 plugin の考え方

可視化 plugin は次の責務を持つ。

- input dataset を読む
- output dataset に `html / image / text` を書く

つまり、可視化 plugin も本質的には処理 plugin である。  
viewer plugin と役割を混同しない。

### viewer plugin との違い

- 可視化 plugin: dataset から新しい結果 file を生成する
- viewer plugin: 既存 file をそのまま表示する

## 装置 plugin の結果閲覧

装置 plugin も同じ整理にする。

- 装置 plugin が output dataset を作る
- output dataset に renderable な結果があれば built-in viewer または専用 viewer plugin で表示する
- renderable でなければ別の可視化 plugin を使う
