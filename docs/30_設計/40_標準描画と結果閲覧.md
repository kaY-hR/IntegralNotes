# 標準描画と結果閲覧

## 位置づけ

MVP では、renderable なファイルの閲覧は app の標準機能とする。  
可視化 plugin は「renderable でない dataset を renderable dataset に変換する処理 plugin」として扱う。

加えて、標準 viewer だけでは不足する場合に備え、拡張子ごとの専用 viewer plugin を追加できるようにする。  
viewer plugin は既存の plugin system に統合し、`viewers` contribution として提供する。

### Dataset Manifest

- `.idts`

加えて、拡張子が未登録でも text と読める file は built-in text viewer で扱ってよい。

`.idts` は作成元を問わない dataset manifest の built-in viewer で扱う。  
表示内容は少なくとも次を含む。

- dataset 名 / kind
- `memberIds` がある場合は、解決した managed file 名と workspace path へのリンク
- `dataPath` がある場合は、bundle resolve 情報
- 対応する managed data note の埋め込み表示

これは generic text viewer ではなく、dataset の内容確認を main app 内で完結させるための built-in viewer とする。

## 専用 viewer plugin

### 初版の matcher

- 拡張子ベースのみ

例:

- `.spc`
- `.abf`
- `.shim-result`

### 役割

- built-in viewer では不足する file を app 内で read-only preview する
- dataset 表示と workspace file 表示の両方で同じ viewer registry を使う
- note 本文の `![](/path)` embed preview でも、基本は同じ viewer registry を使えるようにする
- ただし `.idts` だけは例外的に既存 DataSetViewer 系へ流す

### 初版で扱わないもの

- JSON schema 判定
- Markdown frontmatter 判定
- note editor / note tab の override

`.md` は引き続き note tab で扱い、viewer plugin の初版対象には含めない。

### embed preview

workspace 内 file を `![](/path/from/cwd)` で埋め込むときも、viewer 解決順は file tab と揃える。

1. `.idts` なら datasetId を解決し、既存 DataSetViewer を直接使う
2. それ以外で拡張子に一致する plugin viewer
3. それ以外で built-in viewer
4. 未対応として扱う

embed は read-only context だが、preview 自体は通常の iframe / preview として描画し、少なくとも scroll と text 選択を阻害しない。  
そのため host 側は surface 全体に click hitbox を被せず、iframe や markdown preview の pointer event を奪わない。

block 実行後や data-note 追記で app が自動挿入した `![]()` も、通常の note embed と同じ viewer 解決規則で扱う。  
事前に「表示可能か」を判定せず、standard embed / managed data note fallback の既存解決に委ねる。

workspace target を持つ embed には、右上に flat な `[別タブで開く]` button を出す。  
通常の workspace file は file tab、managed data note fallback は note tab をこの button から開く。

`.idts` は file tab で開いた場合には dataset manifest viewer を使うが、embed の場合だけは既存 DataSetViewer を直接表示する。  
これは一般的な viewer plugin 整合性よりも、dataset embed の見え方統一を優先した例外仕様である。
そのため `.idts` embed では IntegralNotes 標準 embed の click-to-open / resize wrapper を使わない。

## 表示 block

標準表示も共通 block schema で表す。

```json
{
  "id": "BLK-7A8B9C0D",
  "plugin": "core-display",
  "block-type": "dataset-view",
  "params": {},
  "inputs": {
    "source": "DTS-9X4Q2M1A"
  },
  "outputs": {}
}
```

## 表示ルール

- input slot `source` に指定された dataset を読む
- app は datasetId を readable directory path に resolve する
- resolve 結果の directory 内ファイルを走査する
- file ごとに viewer を解決する
- 表示可能な file を flex-layout で表示する

手動作成 dataset と block 生成 dataset は同じ viewer contract で扱い、dataset の current path や representation は viewer 側に漏らさない。

### viewer 解決順

1. 拡張子に一致する plugin viewer
2. built-in viewer
3. 未対応として扱う

dataset viewer では、どの file も 1 つの route で解決する。  
「built-in renderable だけを先に拾う」のではなく、plugin viewer と built-in viewer の両方を同じレイヤーで判定する。

### 並び順

MVP では固定順でよい。  
特に意味を持たせない。

## 対象ファイルがない場合

- 「表示可能なファイルがありません」を出す
- dataset 内に何があるかの一覧は別途見られるようにしてよい

## ノート内リンクから file を開く場合

通常 note 本文の Markdown link から workspace file を開く場合も、既存の file 表示能力を使う。

- `.md` は note tab で開く
- 非 `.md` file は、まず plugin viewer を探す
- plugin viewer が無ければ built-in viewer を使う
- どちらも無ければ外部アプリ導線へ寄せる

つまり、note link は viewer の新形式ではなく、既存の workspace file open 導線の入口である。  
詳細は `docs/30_設計/60_ノートリンク記法.md` を参照。

## 可視化 plugin の考え方

可視化 plugin は次の責務を持つ。

- input dataset を読む
- output dataset に `html / image / text` を書く

つまり、可視化 plugin も本質的には処理 plugin である。  
viewer plugin と役割を混同しない。

### viewer plugin との違い

- 可視化 plugin: dataset から新しい結果 file を生成する
- viewer plugin: 既存 file をそのまま表示する

## 装置 plugin の結果閲覧

装置 plugin も同じ整理にする。

- 装置 plugin が output dataset を作る
- output dataset に renderable な結果があれば built-in viewer または専用 viewer plugin で表示する
- renderable でなければ別の可視化 plugin を使う
