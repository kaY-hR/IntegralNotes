# ワークスペースレイアウト

## 全体

```text
Workspace/
  <user-managed notes, scripts, folders>
  Data/
  .store/
  .codex/
  .claude/
```

## ノート配置

- 通常ノートはユーザーが任意のフォルダに配置する
- 初回ワークスペース設定時に `Notes/` は自動作成しない
- `Notes/` という名前のフォルダを手動で使うこと自体は妨げないが、system 管理ディレクトリではない
- Python script も通常の workspace file として任意フォルダへ置ける

## workspace template

app は workspace 初期化用の template folder を持つ。  
template は `.codex/skills`、`.claude/skills`、`.vscode/settings.json`、`scripts/integral/` など、標準で展開したい workspace file 一式をまとめて保持する。

source:

- 開発時: 起動中 source tree の `Notes/`
- packaged app: `process.resourcesPath/workspace-template/`
- packaging では `Notes/` の中身を `workspace-template/` として同梱する

展開:

- `cwd` を開いたとき、空扱いなら template の中身を `cwd` へ自動展開する
- 空扱いでなければ自動展開しない
- menu の `初期化/更新` から明示実行した場合は、空扱いでなくても強制上書き展開する
- 同じ相対 path の file は上書きする
- 同じ相対 path の directory は中へ再帰 copy する
- MVP では削除同期、merge、user edit 検知、conflict file 生成、version 管理は行わない
- template 側から消えた file が `cwd` に残っていても削除しない

空扱い:

- visible な通常 file / directory が無い状態
- `.git`、`.` で始まる entry、Windows hidden 属性の entry だけがある状態

安全性:

- 展開先は現在開いている workspace root 配下に限定する
- template 内 entry の相対 path は展開前に正規化し、workspace root の外へ出る path は許可しない
- symlink など通常 file / directory 以外の entry は MVP では展開対象外にする

## data-note

data-note は managed data に紐づく system-managed Markdown とする。

例:

```text
.store/
  .integral/
    data-notes/
      FL-7K2M9Q4D.md
      DTS-9X4Q2M1A.md
```

方針:

- `.md` を除く managed file の data-note は自動作成
- dataset の data-note も作成元を問わず自動作成
- file 名は target ID 固定とし、ユーザーに管理させない
- rename / move は system 管理とし、ユーザーにはさせない
- app 上では本文だけを編集・表示する
- owner である managed data を管理対象から外したときは、対応する metadata JSON と data-note も同時に削除する

## Data

user-facing path の default と、note 本文へ添付した画像の既定保存先に使ってよい。

例:

```text
Data/
  sample01.lcd
  samples.idts
  20260414-1730-123.png
```

方針:

- `Data/` は special directory ではなく default location である
- managed file も dataset manifest も、workspace 内の任意 path を current path にしてよい
- note image attachment も ordinary workspace file であり、他の workspace file と同様に managed file tracking 対象になりうる
- 起動時または Sync 時には `cwd` を scan し、未管理の workspace file を managed file として自動登録してよい
- ただし `.` または `_` で始まる directory 配下、`.` で始まる file、`.store` などの system-managed directory 配下は自動登録対象から除外する

## .store

```text
.store/
  .integral/
    FL-7K2M9Q4D.json
    DTS-9X4Q2M1A.json
    data-notes/
      FL-7K2M9Q4D.md
      DTS-9X4Q2M1A.md
    runtime/
      BLK-1F8E2D0A/
        analysis-args.json
        stdout.log
        stderr.log
  objects/
    DTS-9X4Q2M1A/
      result.html
      table.csv
```

補足:

- managed file の実体を `.store/objects/` に複製する構成は採らない
- `.store/objects/` は `.idts` dataset が指す hidden bundle directory に使う
- 単体 file output は `.store/objects/` を使わず、指定した workspace path 自体を current path にする

### input bundle dataset

```text
Data/
  samples.idts
```

`samples.idts` 例:

```json
{
  "datasetId": "DTS-1A2B3C4D",
  "name": "samples",
  "memberIds": ["FL-7K2M9Q4D", "FL-9X4Q2M1A"]
}
```

ルール:

- bundle dataset の current path は visible `Data/{dataset-name}.idts` を default にしてよい
- 手動作成でも block 生成でも、dataset の entrypoint は `.idts` で統一する
- dataset が参照する managed file を管理対象から外した場合は、対応する `.idts` manifest の `memberIds` からもその ID を除去し、dataset metadata を再同期する

## runtime metadata

```text
.store/
  .integral/
    runtime/
      BLK-1F8E2D0A/
        analysis-args.json
        stdout.log
        stderr.log
```

### ルール

- 実行補助 file や log は `.store/.integral/runtime/` に置く
- Python source file 自体は workspace 内の user-managed path をそのまま使う
- 実行のたびに runtime 配下を上書きまたは入れ替えてよい
- bundle dataset の既定 `.idts` manifest は `Data/` に置き、内部参照は `datasetId` を正とする

## GC 対象

MVP では、次を GC 対象にできる。

- どこからも参照されない dataset
- 不要になった runtime log

一方、次は GC 対象にしない。

- managed file
- user-managed な `.py` file
- data-note

補足:

- data-note は単独 GC しないが、owner である managed data を明示的に管理対象から外したときは一緒に削除してよい

## 補足

- `.integral` を無視すれば `.store` は普通の filesystem として読める
- app からは `.store` 自体を非表示にしてよい
