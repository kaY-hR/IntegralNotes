# IntegralNotes プロダクト概要

## 位置づけ

IntegralNotes は、研究者が実験ノートを中心にして

- 元データを登録する
- 処理 block を配置する
- 処理を実行する
- 結果を閲覧する
- 考察を記録する

を一気通貫で行うための ELN である。

## コンセプト

- note-first
- block-based
- plugin-extensible
- results-focused

このプロダクトで一番重視するのは「解析結果がノートに残り、後から再確認できること」である。  
Jupyter のような対話実行そのものを主役にするのではなく、ノートと結果の結びつきを主役にする。

## 対象ユーザー

- Python や装置固有 UI を使って解析したい研究者
- Markdown ノートを中心にデータと考察を残したい研究者
- LLM で block を生成しながら解析フローを組みたい利用者

## プロダクト原則

### 1. すべての block は共通 schema を使う

装置制御、Python 解析、標準表示を別々の JSON 形式にしない。  
すべての block は同じ外形を持つ。

### 2. ユーザー入力単位と解析単位を分ける

- 元データ登録単位は `blob`
- 処理入出力単位は `chunk`

ユーザーは blob を扱いやすく、処理系は chunk を扱いやすい。  
この二層を分ける。

### 3. 処理 block と表示 block を分ける

- 処理 block は `chunk -> chunk`
- 表示 block は renderable な chunk を読む

可視化 plugin は、非閲覧データを `html / image / text` に変換する処理 plugin とみなす。

### 4. Python はまず自由記述を優先する

MVP では Python 解析の自由度を優先する。

- `params schema` は持たない
- `params` は常に `{}` とする
- `cwd` 内の任意の `.py` を登録できる
- `analysis-args.json` に `inputs / outputs` の絶対パスだけを渡す

### 5. 未リリースのため互換性は持たない

旧 block schema や旧 plugin schema との互換性は一旦考えない。  
今回決める schema を唯一の正とする。

## MVP で解きたいこと

- 元データを blob として登録できる
- blob から source chunk を作って処理に渡せる
- Python スクリプトを登録し、処理 block として実行できる
- 装置 plugin も同じ block schema 上で動かせる
- `html / image / text` を標準表示できる
- 処理結果をノートから参照できる

## MVP でまだ解かないこと

- Python 環境の自動構築
- pip 依存の自動解決
- Jupyter kernel の統合
- kind/format 制約の厳密 enforcement
- block 間の高度なワークフロー編集 UI
- 自動 import 解析による同梱ファイル推定
