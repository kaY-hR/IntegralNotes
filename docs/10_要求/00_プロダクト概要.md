# IntegralNotes プロダクト概要

## 位置づけ

IntegralNotes は、研究者が実験ノートを中心にして

- 元データを登録する
- 処理 block を配置する
- 処理を実行する
- 結果を閲覧する
- 考察を記録する

を一気通貫で行うための ELN である。

## コンセプト

- note-first
- block-based
- plugin-extensible
- results-focused

このプロダクトで一番重視するのは「解析結果がノートに残り、後から再確認できること」である。  
Jupyter のような対話実行そのものを主役にするのではなく、ノートと結果の結びつきを主役にする。

## 対象ユーザー

- Python や装置固有 UI を使って解析したい研究者
- Markdown ノートを中心にデータと考察を残したい研究者
- LLM で block を生成しながら解析フローを組みたい利用者

## プロダクト原則

### 1. すべての block は共通 schema を使う

装置制御、Python 解析、標準表示を別々の JSON 形式にしない。  
すべての block は同じ外形を持つ。

### 2. ユーザー入力単位と解析単位を分ける

- 元データ登録単位は `original data`
- 処理入出力単位は `dataset`

ユーザーは original data を扱いやすく、処理系は dataset を扱いやすい。  
この二層を分ける。

### 3. データ実体は canonical store に置く

- system が認識した original data / dataset の canonical 実体は `.store/` に置く
- metadata は `.store/.integral/` に分離する
- app 内部の参照は path ではなく ID を正とする
- user-facing な alias が必要な場合は
  - file は hard link
  - directory は junction
  で表現する
 - データ登録画面から `cwd` 外を能動登録する場合は、canonical 実体を `.store` にコピーし、`Data/` に alias を置く
 - すでに `cwd` 内にある file / directory を登録する場合は、canonical 実体だけ `.store` に寄せ、元の path を alias として残す
### 4. 処理 block と表示 block を分ける

- 処理 block は `dataset -> dataset`
- 表示 block は renderable な dataset を読む

可視化 plugin は、非閲覧データを `html / image / text` に変換する処理 plugin とみなす。

### 5. Python はまず自由記述を優先する

MVP では Python 解析の自由度を優先する。

- `params schema` は持たない
- `params` は常に `{}` とする
- `cwd` 内の任意の `.py` を登録できる
- `analysis-args.json` に `inputs / outputs` の絶対パスだけを渡す

### 6. 未リリースのため互換性は持たない

旧 block schema や旧 plugin schema との互換性は一旦考えない。  
今回決める schema を唯一の正とする。

### 7. GUI はフラットなデスクトップ UI を優先する

IntegralNotes の GUI は、VS Code / Obsidian のようなフラットな desktop UI に寄せる。

- 大きい角丸 card
- 強い shadow
- 強い gradient
- pill 形状の button
- 装飾のためだけの外枠や見出し

は避ける。  
1px border と text hierarchy を中心に構成し、本体 UI と plugin / block / dialog の見た目を揃える。  
特に標準 viewer は、余計な飾りを持たず、必要な操作と表示だけを前面に出す。

## MVP で解きたいこと

- `cwd` 内外の file / directory を original data として管理下に置ける
- original data / dataset の canonical 実体を `.store` に保持できる
- original data から source dataset を作り、普通の file / directory 群として処理に渡せる
- Python スクリプトを登録し、処理 block として実行できる
- 装置 plugin も同じ block schema 上で動かせる
- `html / image / text` を標準表示できる
- note 本文から workspace file へ Markdown link で移動できる
- note 本文へ貼り付けた画像を workspace 内 file として永続化できる
- 処理結果をノートから参照できる

## MVP でまだ解かないこと

- Python 環境の自動構築
- pip 依存の自動解決
- Jupyter kernel の統合
- kind/format 制約の厳密 enforcement
- block 間の高度なワークフロー編集 UI
- 自動 import 解析による同梱ファイル推定
