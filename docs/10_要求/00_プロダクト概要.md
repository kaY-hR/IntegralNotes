# IntegralNotes プロダクト概要

## 位置づけ

IntegralNotes は、研究者が実験ノートを中心にして

- file を追跡管理する
- 処理 block を配置する
- 処理を実行する
- 結果を閲覧する
- 考察を記録する

を一気通貫で行うための ELN である。

## コンセプト

- note-first
- block-based
- plugin-extensible
- results-focused

このプロダクトで一番重視するのは「解析結果がノートに残り、後から再確認できること」である。  
Jupyter のような対話実行そのものを主役にするのではなく、ノートと結果の結びつきを主役にする。

## 対象ユーザー

- Python や装置固有 UI を使って解析したい研究者
- Markdown ノートを中心にデータと考察を残したい研究者
- LLM で block を生成しながら解析フローを組みたい利用者

## プロダクト原則

### 1. すべての block は共通 schema を使う

装置制御、Python 解析、標準表示を別々の JSON 形式にしない。  
すべての block は同じ外形を持つ。

### 2. 管理対象 file と bundle 入出力を分ける

- 基本の追跡単位は `managed file`
- 複数 file を 1 つの入出力として扱いたい場合だけ `dataset (.idts)` を使う

通常の slot は file path をそのまま扱い、bundle が必要な場面だけ `.idts` に寄せる。

### 3. 管理対象データは `id / path / hash` で追跡する

- managed file / dataset は stable ID を持つ
- app 内部の参照は ID を正とする
- metadata は `.store/.integral/` に分離する
- current path は workspace 内のどこでもよい
- `visibility` はその path を app が hidden 扱いするかどうかの属性であり、visible / hidden の二重 canonical copy を意味しない
- dataset の current path は `.idts` manifest を指す
- user-facing な dataset の current path は visible `Data/*.idts` を default にしてよい
- `Data/` は visible path の default に過ぎず、必須ではない
- file は content hash、directory は tree hash で追跡する
- 起動時には path / hash の差分から rename / move / content update を検知する

### 4. data-note は system-managed にする

- managed file / dataset のうち、通常の Markdown note (`.md` managed file) を除く data に `1 data = 1 data-note` を持たせる
- data-note は ID に 1:1 で紐づけて管理する
- 保存場所は `.store/.integral/data-notes/` 配下とし、workspace 上の user-managed file としては扱わない
- ユーザーは data-note の本文だけを編集する
- data-note の rename / move は system 管理とし、ユーザーにはさせない
- data-note は「ファイルへのリンク集」ではなく、「この data は何か」を説明するための note として扱う

### 5. 処理 block と表示 block を分ける

- 処理 block は slot 契約に従って file path または bundle path を読む
- 表示 block は renderable な file や `.idts` manifest を読む

可視化 plugin は、非閲覧データを `html / image / text` に変換する処理 plugin とみなす。

### 6. Python は workspace file + function discovery を優先する

MVP では Python 解析の自由度を優先する。

- `params schema` は持たない
- `params` は free-form object とする
- `general-analysis` plugin が `cwd` 配下の `.py` を走査し、decorator 付き関数を block 候補として出す
- Python callable の canonical 参照は `relative/path.py:function` とする
- `analysis-args.json` に `inputs / outputs / params` を渡す

### 7. 未リリースのため互換性は持たない

旧 block schema や旧 plugin schema との互換性は一旦考えない。  
今回決める schema を唯一の正とする。

### 8. GUI はフラットなデスクトップ UI を優先する

IntegralNotes の GUI は、VS Code / Obsidian のようなフラットな desktop UI に寄せる。

- 大きい角丸 card
- 強い shadow
- 強い gradient
- pill 形状の button
- 装飾のためだけの外枠や見出し

は避ける。  
1px border と text hierarchy を中心に構成し、本体 UI と plugin / block / dialog の見た目を揃える。  
特に標準 viewer は、余計な飾りを持たず、必要な操作と表示だけを前面に出す。

## MVP で解きたいこと

- `cwd` 内外の file / directory を managed file として管理下に置ける
- managed file / dataset を `id / path / hash` で追跡できる
- current path と visibility を分離した同じ managed-data model で扱える
- dataset を `.idts` entrypoint で管理できる
- 複数 managed file から `.idts` dataset を作り、実行時には file / directory 群として処理に渡せる
- system-managed な data-note を managed file / dataset にぶら下げられる
- Python スクリプトを登録し、処理 block として実行できる
- 装置 plugin も同じ block schema 上で動かせる
- `html / image / text` を標準表示できる
- note 本文から workspace file へ Markdown link で移動できる
- note 本文へ貼り付けた画像を workspace 内 file として永続化できる
- 処理結果をノートから参照できる

## MVP でまだ解かないこと

- Python 環境の自動構築
- pip 依存の自動解決
- Jupyter kernel の統合
- kind/format 制約の厳密 enforcement
- block 間の高度なワークフロー編集 UI
- 自動 import 解析による同梱ファイル推定
