# ユーザー体験

## 1. Workspace を開く

1. ユーザーは `cwd` を開く
2. app は `.store/` などの system-managed path を除いた workspace file を scan する
3. 未管理 file には ID を割り当てて managed file として取り込む
4. `.md` も画像も CSV も HTML も同じ managed file として追跡する
5. file の中身が変われば hash を更新し、path が変われば tracking によって追跡する

## 2. Python 処理 Block を置く

1. ユーザーは editor 上で `>` を入力する
2. app は `cwd` 配下の `.py` を走査し、decorator 付き関数一覧を出す
3. 候補一覧では `PCA` のような displayName を主表示し、`scripts/pca.py:main` を補足表示する
4. ユーザーが候補を選ぶ
5. app は `run: scripts/pca.py:main` を持つ YAML `itg-notes` block をノートへ挿入する

## 3. Block Source の見え方

Python block の note source は、人が読める最小形にする。

```itg-notes
id: BLK-1F8E2D0A
run: scripts/pca.py:main
in:
  samples: /Data/samples.csv
  labels: /Data/labels.csv
params:
  n_components: 2
out:
  score: /Results/score.csv
  report: /Results/report.html
```

bundle を使う場合:

```itg-notes
id: BLK-1F8E2D0A
run: scripts/report.py:main
in:
  source: /Data/samples.idts
params: {}
out:
  report: /Results/report.html
```

この source から読み取りたいのは次だけである。

- どの script / function を呼ぶか
- どの path を input に使うか
- どの param を渡すか
- どの path へ output を出すか

## 4. Input / Output 指定の UX

- 各 slot には file path を割り当てる
- slot 定義には `slot 名 / extension / format` がある
- UI では slot に合う extension の file を選びやすくする
- input 割り当ての基本導線は keyboard-first の popup / filter に寄せてよい
- `.idts` を要求する slot では bundle manifest を選ぶ
- output では slot の extension を見ながら保存先 path を指定する
- output slot には `作業 note へ自動挿入するか` と `どの input の data-note へ投影するか` が定義されていてよい

## 5. Block を実行する

1. app は input path を検証する
2. `.idts` input の場合は hidden bundle directory へ resolve する
3. app は `analysis-args.json` を作る
4. Python callable を実行する
5. 非 `.idts` output なら、指定 path に file が保存される
6. `.idts` output なら、visible manifest file はユーザー指定 path に保存され、中身は hidden bundle directory に配置される
7. 実行成功後、output file に対応する managed file metadata が作成または更新される
8. output slot が `auto_insert_to_work_note = true` を持つ場合、app は block 直下へ `![]()` を追記する
9. output slot が `project_to_inputs` を持つ場合、app は対応する data-note の末尾に provenance link と `![]()` を追記する
10. これらの自動追記は append-only とし、不要になった古い embed はユーザーが手で消せる

## 6. 結果を見る

- 通常 file は extension に応じて viewer で開く
- `.idts` file は bundle manifest viewer または bundle preview で開く
- renderable file は app 内で表示する
- unsupported file は外部アプリで開く

## 7. Data Note を見る

1. ユーザーは data-note 対象の managed data を選ぶ
2. app はその ID に紐づく system-managed data-note を開く
3. ユーザーは本文だけを編集する
4. 解析結果が投影される場合、app は provenance link と `![]()` を末尾へ追記してよい
5. target file の path が変わっても、同じ ID の note を開ける

## 8. Link / Embed の追従

1. ユーザーは note 本文で通常の Markdown 記法 `[label](path)` や `![](path)` を使う
2. provenance link では `[Experiment A](/Notes/ExperimentA.md#BLK-1F8E2D0A)` のような block deep link を使ってよい
3. file path が IntegralNotes 内で変わった場合、app は参照元 source を更新する
4. 外部ツールで path が変わった場合も、path / hash tracking で同一 file を追跡できたら、app は参照元 text file を scan して path を更新してよい
5. source 記法自体は path のままとし、ID 記法に変えない

## 9. process chain viewer

- process chain viewer は optional な workspace-tool plugin として実装し、Activity Bar item から main workspace の tab として開ける
- process chain viewer は workspace 全体ではなく、今開いている note 内の全 block、または今見ている file を起点にする
- note を起点にした場合、描画対象は各 block の直系の祖先 / 直系の子孫だけに絞る
- 共有 input を使うだけの sibling / cousin block は描画しない
- node click では元の file / block を開き、block link なら該当位置へ scroll する
- edge は常に `out = right`, `in = left` の port へ接続する
- hover preview は file node に対して追加してよいが、MVP では未実装でもよい

## 10. LLM 利用時の体験

LLM は block source を直接書ける。  
そのため block schema は人間と LLM の両方にとって単純である必要がある。

MVP の方針:

- LLM に覚えてもらうのは `run/use`, `in`, `params`, `out` のルールだけ
- slot 詳細は decorator から得た `displayName` / `description` / `extension` / `format` を見せればよい
- source 記法は path ベースのままにする
