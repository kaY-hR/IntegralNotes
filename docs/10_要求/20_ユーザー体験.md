# ユーザー体験

## 1. Workspace を開く

1. ユーザーは `cwd` を開く
2. app は `.store/` などの system-managed path を除いた workspace file を scan する
3. 未管理 file には ID を割り当てて managed file として取り込む
4. `.md` も画像も CSV も HTML も同じ managed file として追跡する
5. file の中身が変われば hash を更新し、path が変われば tracking によって追跡する

## 2. Python 処理 Block を置く

1. ユーザーは editor 上で `>` を入力する
2. app は `cwd` 配下の `.py` を走査し、decorator 付き関数一覧を出す
3. 候補一覧では `PCA` のような displayName を主表示し、`scripts/pca.py:main` を補足表示する
4. ユーザーが候補を選ぶ
5. app は `run: scripts/pca.py:main` を持つ YAML `itg-notes` block をノートへ挿入する

## 3. Block Source の見え方

Python block の note source は、人が読める最小形にする。

authoring 時は path をそのまま書ける。

```itg-notes
id: BLK-1F8E2D0A
run: scripts/pca.py:main
in:
  samples: /Data/samples.csv
  labels: /Data/labels.csv
params:
  n_components: 2
out:
  score: /Results/score.csv
  report: /Results/report.html
```

保存または実行前に、`in:` は managed data ID へ正規化される。

```itg-notes
id: BLK-1F8E2D0A
run: scripts/pca.py:main
in:
  samples: FL-7K2M9Q4D
  labels: FL-1A2B3C4D
params:
  n_components: 2
out:
  score: /Results/score.csv
  report: /Results/report.html
```

実行成功後は、`out:` も生成された managed data ID へ書き換わる。

```itg-notes
id: BLK-1F8E2D0A
run: scripts/pca.py:main
in:
  samples: FL-7K2M9Q4D
  labels: FL-1A2B3C4D
params:
  n_components: 2
out:
  score: FL-9X4Q2M1A
  report: FL-3N8K7Q2P
```

bundle を使う場合:

```itg-notes
id: BLK-1F8E2D0A
run: scripts/report.py:main
in:
  source: DTS-7K2M9Q4D
params: {}
out:
  report: /Results/report.html
```

この source から読み取りたいのは次だけである。

- どの script / function を呼ぶか
- どの managed data ID を input に使うか
- どの param を渡すか
- 実行前はどの path へ output を出すか
- 実行後はどの output ID が生成されたか

## 4. Input / Output 指定の UX

- 各 slot には file path を割り当てる
- slot 定義には `slot 名 / extension / format` がある
- UI では slot に合う extension の file を選びやすくする
- input 割り当ての基本導線は keyboard-first の popup / filter に寄せてよい
- `.idts` を要求する slot では bundle manifest を選ぶ
- output では slot の extension を見ながら保存先 path を指定する
- output slot には `作業 note へ自動挿入するか` と `どの input の data-note へ投影するか` が定義されていてよい

## 5. Block を実行する

1. app は `in:` の ID を検証し、authoring path が残っていれば managed data ID へ解決する
2. app は `in:` の ID を current path へ解決する
3. `.idts` input の場合は hidden bundle directory へ resolve する
4. app は `analysis-args.json` を作る
5. Python callable を実行する
6. 非 `.idts` output なら、指定 path に file が保存される
7. `.idts` output なら、visible manifest file はユーザー指定 path に保存され、中身は hidden bundle directory に配置される
8. 実行成功後、output file に対応する managed file / dataset metadata が作成または更新される
9. app は `out:` を生成された managed data ID へ書き換える
10. 実行済み block は provenance として read-only 表示になり、削除だけできる
11. output slot が `auto_insert_to_work_note = true` を持つ場合、app は block 直下へ `![]()` を追記する
12. output slot が `share_note_with_input` を持つ場合、app は output の data-note target を指定 input の note target として解決する
13. output slot が `embed_to_shared_note = true` を持つ場合、app は共有先 data-note の末尾に provenance link と `![]()` を追記する
14. これらの自動追記は append-only とし、不要になった古い embed はユーザーが手で消せる

## 6. 結果を見る

- 通常 file は extension に応じて viewer で開く
- `.idts` file は bundle manifest viewer または bundle preview で開く
- renderable file は app 内で表示する
- unsupported file は外部アプリで開く

## 7. Data Note を見る

1. ユーザーは data-note 対象の managed data を選ぶ
2. app はその data の `noteTargetId` に紐づく system-managed data-note を開く
3. ユーザーは本文だけを編集する
4. 解析結果が投影される場合、app は provenance link と `![]()` を末尾へ追記してよい
5. target file の path が変わっても、同じ ID の note を開ける

## 8. Link / Embed の追従

1. ユーザーは note 本文で通常の Markdown 記法 `[label](path)` や `![](path)` を使う
2. provenance link では `[Experiment A](/Notes/ExperimentA.md#BLK-1F8E2D0A)` のような block deep link を使ってよい
3. file path が IntegralNotes 内で変わった場合、app は参照元 source を更新する
4. 外部ツールで path が変わった場合も、path / hash tracking で同一 file を追跡できたら、app は参照元 text file を scan して path を更新してよい
5. source 記法自体は path のままとし、ID 記法に変えない

## 9. process chain viewer

- process chain viewer は optional な workspace-tool plugin として実装し、Activity Bar item から main workspace の tab として開ける
- process chain viewer は workspace 全体ではなく、今開いている note 内の全 block、または今見ている file を起点にする
- note を起点にした場合、描画対象は各 block の直系の祖先 / 直系の子孫だけに絞る
- 共有 input を使うだけの sibling / cousin block は描画しない
- node click では元の file / block を開き、block link なら該当位置へ scroll する
- process chain viewer は node click で target tab を開いても、自分自身の focus は維持してよい
- edge は常に `out = right`, `in = left` の port へ接続する
- hover preview は file node に対して追加してよいが、MVP では未実装でもよい

## 10. LLM 利用時の体験

LLM は block source を直接書ける。  
そのため block schema は人間と LLM の両方にとって単純である必要がある。

MVP の方針:

- LLM に覚えてもらうのは `run/use`, `in`, `params`, `out` のルールだけ
- slot 詳細は decorator から得た `displayName` / `description` / `extension` / `format` を見せればよい
- authoring 時の `in/out` では path を書ける
- 保存後の `in:` と実行後の `out:` は ID になる
- LLM 用 tool として、path から ID、ID から current path を解決できる API を用意する
- 通常 Markdown link / image の source 記法は path ベースのままにする

## 11. AI Chat から CLI を実行する

1. LLM は workspace の調査や処理に CLI 実行が必要な場合、`command` と `purpose` を持つ tool_call を出す
2. app は command をすぐ実行せず、共通 dialog に command / purpose / workingDirectory / timeout / warning を表示する
3. user は command を確認し、必要なら編集する
4. user が許可した場合、app は workspace folder を起点に PowerShell command を実行する
5. user が reject した場合、reject 理由を LLM へ返し、実行しない
6. 実行中は可能なら stdout / stderr を dialog に表示し、user は Cancel できる
7. timeout または Cancel 時はプロセス停止を試み、partial stdout / stderr を LLM へ返す
8. 実行後は workspace sync 相当を走らせ、生成・更新・削除された file を workspace view と managed metadata に反映する
9. AI Chat 履歴には tool trace として目的、承認結果、編集有無、exit code、出力要約を残す

MVP の制約:

- default timeout は 60 秒、上限は 300 秒
- stdout / stderr の LLM 返却はそれぞれ最大 20,000 文字
- `stdin` は未対応
- shell executable path は AI Chat settings で指定できる
- 毎回承認を必須にし、remember allow / deny は将来対応とする
- この機能は確認付き実行であり、OS レベルの sandbox ではない
