# ユーザー体験

## 1. 元データを登録する

1. ユーザーはデータ登録画面から file または directory を選ぶ
2. app は `ORD-...` を発行し、`path` と `hash` を含む metadata を `.store/.integral/` に記録する
3. 対象が `cwd` 外なら、app は workspace 内の default visible path へ copy してよい
4. hidden 管理したい場合は、app は `.store/` 配下 path を canonical path にしてよい
5. app は対応する data-note を `.store/.integral/data-notes/` 配下に作る
6. ユーザーは original data からその data-note を開き、「これは何のデータか」を記述できる

## 2. Python 処理 block を新規に置く

1. ユーザーは editor 上で `>` を入力し、Python callable 候補 popup を開く
2. app は `cwd` 配下の `.py` を走査し、decorator 付き関数一覧を出す
3. 候補一覧では `PCA` のような displayName を主表示し、`scripts/pca.py:main` を補足表示する
4. ユーザーが候補を選ぶ
5. app は `run: scripts/pca.py:main` を持つ YAML `itg-notes` block をノートへ挿入する
6. input / output slot は decorator 定義に従って初期化される

## 3. block source の見え方

Python block の note source は、人が読める最小形にする。

例:

```itg-notes
id: BLK-1F8E2D0A
run: scripts/pca.py:main
in:
  samples: /Data/samples.idts
  labels: /Data/labels.idts
params:
  n_components: 2
out:
  score: auto
  loading: auto
```

この source から読み取りたいのは次だけである。

- どの script / function を呼ぶか
- どの `.idts` を input に使うか
- どの param を渡すか
- どの output slot を持つか

## 4. input 指定の UX

各 input slot には次の 2 通りの指定方法がある。

- 既存 dataset を 1 つ選ぶ
- original data を複数選ぶ

original data を複数選んだ場合:

- ユーザーは source dataset 名を決める
- app は新しい source dataset を作る
- app は source dataset に対応する data-note も作る
- source dataset は `.idts` として作られる
- 実行時には app がその dataset を file / directory 群へ resolve する
- block source の `in:` には source dataset の `.idts` path を書く

## 5. block を実行する

1. app は output slot ごとに新しい output dataset を作る
2. app は output dataset に対応する data-note も作る
3. system 既定名の output dataset では、dataset 名に ID を使ってよい
4. app は `analysis-args.json` を作る
5. plugin が実行される
6. 成功したら hidden metadata 上の最新 output dataset 参照が更新される

再実行時:

- output dataset は新しい ID になる
- output dataset の `.idts` manifest は hidden `.store/.integral/datasets/` 配下に置いてよい
- 古い dataset はそのまま残る
- note source 自体は原則書き換えず、最新結果は hidden metadata から block 表示へ反映する
- Python には app が解決した dataset path の絶対パスが渡る

## 6. 結果を見る

結果表示は block を分ける。

- 処理 block
  - `dataset -> dataset`
- 表示 block
  - 入力 dataset を読み、renderable ファイルを表示する

標準表示 block では、dataset 内の

- html
- image
- text

を自動判定して表示する。

source dataset でも derived dataset でも同じで、app は datasetId から読める directory path を解決して扱う。

## 7. data-note を見る

1. ユーザーは original data または dataset を選ぶ
2. app はその ID に紐づく system-managed data-note を開く
3. ユーザーは本文だけを編集する
4. rename / move はできず、target data が変われば app が同じ ID の note を開く

## 8. 装置 plugin を使う

装置 plugin でも流れは同じ。

1. block を置く
2. input slot に dataset を割り当てる
3. 必要なら custom UI で設定する
4. 実行する
5. output dataset を得る
6. 別 block で結果を見る

## 9. LLM 利用時の体験

LLM は block source を直接書ける。  
そのため block schema は人間と LLM の両方にとって単純である必要がある。

MVP の方針:

- LLM に覚えてもらうのは `run/use`, `in`, `params`, `out` のルールだけ
- Python 固有の複雑な asset schema は持ち込まない
- 詳細説明は decorator から得た `displayName` / `description` を見せればよい

## 10. ノート本文から workspace file を開く

1. ユーザーは note 本文で `[` を入力する
2. app は `cwd` 配下の file 候補を一覧表示する
3. 候補一覧では file 名を主表示し、workspace root 相対 path を補足表示する
4. ユーザーが候補を選ぶと、app は `[label](/path/from/cwd)` を挿入する
5. `.md` では label は拡張子なし、非 `.md` では拡張子ありにする
6. ユーザーが link を click すると、app は target file を既存 open 経路で開く

open 挙動:

- `.md` は app 内 tab で開く
- renderable / text は app 内 viewer で開く
- unsupported は外部アプリで開く

rename / move:

- ユーザーが IntegralNotes 内で target file を rename / move した場合、app は `cwd` 配下の `.md` を走査して link target を自動更新する
- 外部ツールによる rename / move は MVP では追従しない

詳細は `docs/30_設計/60_ノートリンク記法.md` を参照

## 11. ノート本文へ画像を添付する

1. ユーザーは note editor 上で画像を貼り付ける、drop する、または image insert UI から file を選ぶ
2. app は画像を `Data/yyyyMMdd-HHmm-RRR.ext` に保存する
3. app は note 本文へ `![](/Data/...)` を挿入する
4. editor 表示時は、その workspace path を DOM 用 URL へ変換して画像を描画する
5. ユーザーが note を保存して app を再起動しても、同じ source から再表示できる

補足:

- 自動保存先は `Data/` だが、手書きで `![](/images/foo.png)` のような workspace 内画像を参照してもよい
- ユーザーが IntegralNotes 内で target image file を rename / move した場合、app は `.md` 内の image target を自動更新する
- explorer 上の clipboard image 貼り付けは別機能として扱う

詳細は `docs/30_設計/70_ノート画像添付.md` を参照
