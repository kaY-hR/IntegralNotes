# ユーザー体験

## 1. 元データを登録する

1. ユーザーはデータ登録画面から file または directory を選ぶ
2. app は `ORD-...` を発行し、`path` と `hash` を含む metadata を `.store/.integral/` に記録する
3. 対象が `cwd` 外なら、app は workspace 内の default visible path へ copy してよい
4. `visibility` はその path を hidden 扱いするかどうかだけを表し、別 canonical copy は持たない
5. path または hash の片方だけが後で変わった場合、app は `id / path / hash` tracking によって rename / move または内容更新として追随できる
6. app は対応する data-note を `.store/.integral/data-notes/` 配下に作る
7. ユーザーは original data からその data-note を開き、「これは何のデータか」を記述できる
8. 起動時または Sync 時に app は `cwd` を scan し、未管理の非 Markdown file を見つけたら自動で original data 登録する
9. 追跡中の managed data に対して path / hash の再解決候補が一意に見つかれば relink confirm を出し、候補が無ければ「管理対象から外す」confirm を出す
10. 管理対象から外した場合、対応する metadata / data-note が削除され、source dataset が参照していた `.idts` の `memberIds` も必要に応じて更新される

## 2. Python 処理 block を新規に置く

1. ユーザーは editor 上で `>` を入力し、Python callable 候補 popup を開く
2. app は `cwd` 配下の `.py` を走査し、decorator 付き関数一覧を出す
3. 候補一覧では `PCA` のような displayName を主表示し、`scripts/pca.py:main` を補足表示する
4. ユーザーが候補を選ぶ
5. app は `run: scripts/pca.py:main` を持つ YAML `itg-notes` block をノートへ挿入する
6. input / output slot は decorator 定義に従って初期化される
7. output slot は初期状態では `auto` で入る

## 3. block source の見え方

Python block の note source は、人が読める最小形にする。

例:

```itg-notes
id: BLK-1F8E2D0A
run: scripts/pca.py:main
in:
  samples: /Data/samples.idts
  labels: /Data/labels.idts
params:
  n_components: 2
out:
  score: auto
  loading: auto
```

この source から読み取りたいのは次だけである。

- どの script / function を呼ぶか
- どの `.idts` を input に使うか
- どの param を渡すか
- どの output slot を持つか

初回は `out:` が `auto` だが、実行後は生成された visible `.idts` path が書き戻されてよい。

## 4. input 指定の UX

各 input slot には次の 2 通りの指定方法がある。

- 既存 dataset を 1 つ選ぶ
- original data を複数選ぶ

既存 dataset を選ぶ UI では、dataset 名を主表示し、`DTS-...` のような ID は通常見せない。

original data を複数選んだ場合:

- ユーザーは source dataset 名を決める
- app は新しい source dataset を作る
- app は source dataset に対応する data-note も作る
- source dataset は `.idts` として作られる
- 実行時には app がその dataset を file / directory 群へ resolve する
- block source の `in:` には source dataset の `.idts` path を書く

## 5. block を実行する

1. app は output slot ごとに新しい output dataset を作る
2. app は output dataset に対応する data-note も作る
3. system 既定名の output dataset では、dataset 名を `{解析名}_{slot名}_yyyyMMddHHmm` とする
4. app は `analysis-args.json` を作る
5. plugin が実行される
6. 成功したら block source の `out:` に生成された visible `.idts` path が書き戻される

再実行時:

- output dataset は新しい ID になる
- output dataset の `.idts` manifest は `Data/` 配下に置く
- 古い dataset はそのまま残る
- note source の `out:` は最新実行結果の `.idts` path に更新されてよい
- Python には app が解決した dataset path の絶対パスが渡る

## 6. 結果を見る

結果表示は block を分ける。

- 処理 block
  - `dataset -> dataset`
- 表示 block
  - 入力 dataset を読み、renderable ファイルを表示する

標準表示 block では、dataset 内の

- html
- image
- text

を自動判定して表示する。

source dataset でも derived dataset でも同じで、app は datasetId から読める directory path を解決して扱う。

`.idts` file を file tab や picker で見るときも、基本は dataset 名を主表示する。

## 7. data-note を見る

1. ユーザーは original data または dataset を選ぶ
2. app はその ID に紐づく system-managed data-note を開く
3. ユーザーは本文だけを編集する
4. rename / move はできず、target data が変われば app が同じ ID の note を開く

## 8. 装置 plugin を使う

装置 plugin でも流れは同じ。

1. block を置く
2. input slot に dataset を割り当てる
3. 必要なら custom UI で設定する
4. 実行する
5. output dataset を得る
6. 別 block で結果を見る

## 9. LLM 利用時の体験

LLM は block source を直接書ける。  
そのため block schema は人間と LLM の両方にとって単純である必要がある。

MVP の方針:

- LLM に覚えてもらうのは `run/use`, `in`, `params`, `out` のルールだけ
- Python 固有の複雑な asset schema は持ち込まない
- 詳細説明は decorator から得た `displayName` / `description` を見せればよい

## 10. ノート本文から workspace file を開く

1. ユーザーは note 本文で `[` を入力する
2. app は `cwd` 配下の file 候補を一覧表示する
3. 候補一覧では file 名を主表示し、workspace root 相対 path を補足表示する
4. ユーザーが候補を選ぶと、app は `[label](/path/from/cwd)` を挿入する
5. `.md` では label は拡張子なし、非 `.md` では拡張子ありにする
6. ユーザーが link を click すると、app は target file を既存 open 経路で開く

open 挙動:

- `.md` は app 内 tab で開く
- renderable / text は app 内 viewer で開く
- unsupported は外部アプリで開く

rename / move:

- ユーザーが IntegralNotes 内で target file を rename / move した場合、app は `cwd` 配下の `.md` を走査して link target を自動更新する
- 外部ツールによる rename / move は MVP では追従しない

詳細は `docs/30_設計/60_ノートリンク記法.md` を参照

## 11. ノート本文へ画像を添付する

1. ユーザーは note editor 上で画像を貼り付ける、drop する、または image insert UI から file を選ぶ
2. app は画像を `Data/yyyyMMdd-HHmm-RRR.ext` に保存する
3. app は note 本文へ `![](/Data/...)` を挿入する
4. editor 表示時は、その workspace path を DOM 用 URL へ変換して画像を描画する
5. ユーザーが note を保存して app を再起動しても、同じ source から再表示できる

補足:

- 自動保存先は `Data/` だが、手書きで `![](/images/foo.png)` のような workspace 内画像を参照してもよい
- ユーザーが IntegralNotes 内で target image file を rename / move した場合、app は `.md` 内の image target を自動更新する
- explorer 上の clipboard image 貼り付けは、note attach ではなく tree 操作として別導線で扱う

詳細は `docs/30_設計/70_ノート画像添付.md` を参照

## 12. ノート本文に workspace file を埋め込む

1. ユーザーは note 本文で `![` または `!` を入力し、workspace file 候補から target を選ぶ
2. app は note 本文へ `![](/path/from/cwd)` を挿入する
3. editor は file kind に応じて、画像、iframe 系 preview、markdown preview、dataset preview、plugin viewer preview を inline で描画する
4. `html / svg / text / markdown / plugin viewer` などの embed preview は、見た目だけでなく中身も通常どおり scroll できる
5. embed が workspace target を持つ場合、右上に flat な `[別タブで開く]` button が出る
6. ユーザーがその button を押すと、対応する workspace file tab または managed data note tab が開く

補足:

- open 導線は embed surface 全体 click ではなく button に分離する
- `.idts` embed は dataset preview を優先し、通常の file tab とは別に既存 dataset 表示系で見せてよい
- external URL 画像のように workspace target でないものには `[別タブで開く]` を出さなくてよい

詳細は `docs/30_設計/40_標準描画と結果閲覧.md` を参照

## 13. 左サイドを切り替える

1. ユーザーは Explorer のさらに左にある Activity Bar から icon を選ぶ
2. app は対応する sidebar view を右隣の pane に表示する
3. built-in では少なくとも `Explorer`, `Search`, `Plugins` を持つ
4. plugin が `sidebarViews` contribution を持っていれば、その icon も Activity Bar に追加される

補足:

- Explorer 専用 shortcut は Explorer view が active なときだけ効く
- Plugin 管理は modal dialog ではなく `Plugins` sidebar view から行う
- Activity Bar は shell であり、view 本体は各 sidebar view が描画する

## 14. Explorer で tree 操作をする

1. ユーザーは Explorer で項目を click して単一選択する
2. `Ctrl/Cmd + click` で個別に複数選択できる
3. `Shift + click` で可視 tree 順の範囲選択ができる
4. `Ctrl/Cmd + Shift + click` で既存選択へ範囲追加できる
5. ユーザーが選択項目を drag すると、Explorer 内では通常 drag-drop で move、`Ctrl/Cmd + drag` で copy できる
6. 外部 OS から file / directory を Explorer へ drop すると、app は copy として workspace に取り込む
7. `Ctrl/Cmd + C` または右クリック `コピー` で、選択項目を explorer clipboard として保持できる
8. `Ctrl/Cmd + V` または右クリック `貼り付け` で、explorer clipboard の項目を現在位置へ貼り付けられる
9. 外部アプリが clipboard に置いた file / directory も、同じ `貼り付け` 導線で workspace に取り込める
10. clipboard image がある場合は、Explorer 上の `貼り付け` で `image.png` を既定名として保存できる

補足:

- 貼り付け先は選択中 directory、file 選択時はその親 directory、未選択時は workspace root とする
- `image.png` が既に存在する場合は衝突しない file 名へ調整してよい
- hidden フォルダ表示切り替えの対象は `.` または `_` で始まる directory であり、`_` で始まる file は hidden 扱いしない
- 右クリックメニューから `コピー / 貼り付け / 削除 / 名前を変更 / パスのコピー / 相対パスのコピー / DataSetに追加` を実行できる
- managed data を含む削除では、対応する metadata / data-note も同時に削除し、必要なら source dataset の `.idts` も更新する

## 15. workspace 全文検索をする

1. ユーザーは Activity Bar から `Search` icon を選ぶ
2. app は search pane を表示し、`query` 入力ごとに workspace text file を検索する
3. 結果は file ごとに group 表示され、match を click すると対応 file を開く
4. `replace` section を開くと、一括置換文字列を指定できる
5. `include files` / `exclude files` section を開くと、glob で対象 file を絞り込める
6. `case sensitive / whole word / regex` option は query に対してそのまま適用される
7. `Replace All` を実行すると、current query 条件に一致した text file へ一括置換が適用される

補足:

- replace 対象に未保存の markdown tab が含まれる場合、app は先に保存するよう促して置換を止める
- binary file や画像は検索対象に含めない
