# ユーザー体験

## 1. 元データを登録する

1. ユーザーはデータ登録画面から file または directory を選ぶ
2. 対象が `cwd` 外なら、app は canonical 実体を `.store/` へコピーし、`Data/` に alias を置く
3. 対象がすでに `cwd` 内なら、app は canonical 実体を `.store/` へ移し、元の path を alias として戻す
4. app は metadata を `.store/.integral/` に記録する
5. app は対応する data note を作る
6. ユーザーはノートからその data note を参照できる

## 2. Python 処理 block を新規に置く

1. ユーザーは「Python 処理 block を挿入」を選ぶ
2. entry の `.py` を選ぶ
3. 同梱ファイルを選ぶ
4. input slot 名を決める
5. output slot 名を決める
6. displayName と description を必要なら入れる
7. app は `PYS-...` を作る
8. app は block をノートへ挿入する

## 3. 既存 Python script から block を置く

1. ユーザーは `.py-scripts/` 登録済み一覧を開く
2. `displayName` と `description` を見て script を選ぶ
3. 各 input slot に対して dataset または original data 群を指定する
4. original data 群から source dataset を作る場合は dataset 名も決める
5. app は必要なら source dataset を作る
6. block をノートへ挿入する

## 4. input 指定の UX

各 input slot には次の 2 通りの指定方法がある。

- 既存 dataset を 1 つ選ぶ
- original data を複数選ぶ

original data を複数選んだ場合:

- ユーザーは source dataset 名を決める
- app は新しい source dataset を作る
- app は source dataset に対応する data note も作る
- data note file 名には dataset 名を使う
- source dataset は普通の file / directory 群として materialize される
- block には source dataset の `DTS-...` を書く

## 5. block を実行する

1. app は output slot ごとに新しい output dataset を作る
2. app は output dataset に対応する data note も作る
3. system 既定名の output dataset では、dataset 名に ID を使ってよい
4. app は `analysis-args.json` を作る
5. plugin が実行される
6. 成功したら block の `outputs` が `DTS-...` に更新される

再実行時:

- output dataset は新しい ID になる
- 古い dataset はそのまま残る
- block の `outputs` は最新結果へ置き換わる
- Python には `.store/{datasetId}/` の絶対パスが渡る

## 6. 結果を見る

結果表示は block を分ける。

- 処理 block
  - `dataset -> dataset`
- 表示 block
  - 入力 dataset を読み、renderable ファイルを表示する

標準表示 block では、dataset 内の

- html
- image
- text

を自動判定して表示する。

source dataset でも同じで、`links.json` のような専用 manifest を経由せず、dataset フォルダ自体をそのまま扱う。

## 7. 装置 plugin を使う

装置 plugin でも流れは同じ。

1. block を置く
2. input slot に dataset を割り当てる
3. 必要なら custom UI で設定する
4. 実行する
5. output dataset を得る
6. 別 block で結果を見る

## 8. LLM 利用時の体験

LLM は block JSON を直接書ける。  
そのため block schema は人間と LLM の両方にとって単純である必要がある。

MVP の方針:

- LLM に覚えてもらうのは `plugin`, `block-type`, `inputs`, `outputs` のルールだけ
- Python 固有の複雑な schema は持ち込まない
- 詳細説明は `script.json.description` を見せればよい
