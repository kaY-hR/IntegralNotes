# 分析データ添付と解析実行

## この文書の目的

- ELN 上で任意の分析データファイルをアップロードし、ノートに埋め込めるようにする。
- binary の読み出し、ベンダ SDK 利用、可視化、解析実行を plugin ベースで拡張できるようにする。
- とくに `Python実行Plugin` を主軸に、解析 step の chain 実行を成立させる。
- まだ仕様が大きく曖昧なため、まずは実装に進めるための共通語彙と責務分担を決める。

## 背景

IntegralNotes の大きな狙いは、研究者がノートを中心にして

- 分析データを添付する
- データを閲覧する
- 解析を実行する
- 結果を再びノートへ埋め込む

を一気通貫で行えることにある。

現在の plugin システムは、`itg-notes` block を plugin renderer で描画し、必要に応じて host action を呼ぶところまでを対象にしている。  
これは「ノート内の独自 UI block」を扱うには十分だが、「任意ファイルの添付」「binary の読み出し」「解析結果の chain 受け渡し」まではまだ表現できない。

## まず決める前提

### 添付 UI は Milkdown 画像アップロードの拡張ではなく専用化する

- Milkdown の画像挿入 UI は、画像を Markdown に埋め込む用途に寄っている。
- 今回必要なのは画像貼り付けではなく、任意 binary をノート資産として管理し、後段 plugin が参照できること。
- 必要なのは単なる upload ではなく、添付 ID、保存先、メタデータ、参照 block、viewer 選択、解析入力への受け渡しである。
- そのため、画像 UI の横展開ではなく、IntegralNotes 独自の `Attachment` モデルとして扱う。

### 解析の意味論は plugin 主導にする

- どの binary をどう読むか
- ベンダ DLL / exe をどう呼ぶか
- どの形式の中間データを作るか
- どう描画するか

これらは装置や解析手法ごとの差が大きいため、本体へ寄せすぎない。

一方で本体は、次のような最小の共通基盤は持つ。

- workspace 配下への保存
- ノートからの参照解決
- plugin 実行の仲介
- ジョブ履歴と成果物の永続化

### 解析チェーンはファイル受け渡しを正とする

- Python のメモリセッション共有や notebook 的状態保持は、初期段階では複雑すぎる。
- chain の再実行性、途中失敗からの再開、監査性を考えると、step 間はファイルで受け渡す方が自然である。
- したがって中間情報は `.integral/` 配下に永続化する。

## ドメインモデル

### Attachment

ユーザーがノートに関連づけてアップロードした元ファイル。

- 例: `sample01.lcd`, `raw.dat`, `analysis.py`
- 保存先は workspace 配下の `Attachments/` を基本とする
- ノート内では直接 path を持たず、attachment ID で参照する

持つべき情報:

- attachment ID
- 元ファイル名
- 保存先相対 path
- MIME / 拡張子
- 作成日時
- 作成元ノートや参照元ノート

### Artifact

plugin や解析 step が生成した成果物。

- 例: LC binary から抽出したクロマト JSON
- 例: Python script が生成した csv / json / png
- 例: viewer が描画対象にする正規化済みデータ

保存先は workspace 配下の `.integral/artifacts/` を基本とする。

持つべき情報:

- artifact ID
- kind
- schema version
- producing plugin / step
- 入力 attachment / artifact の参照
- 保存先相対 path

### Job / Step

解析の実行単位。

- 1 step = 1 plugin-defined operation
- `Python実行Plugin` では 1 step = 1 script 実行
- chain は step 群と依存関係の列

持つべき情報:

- job ID
- step ID
- runner plugin ID
- 入力 attachment / artifact
- 出力 artifact
- status
- startedAt / finishedAt
- log

## ノート表現

ノートに埋め込むのは raw path ではなく参照 block とする。

### 想定 block

#### `Core.AttachmentRef`

添付ファイルそのものを表す block。

- attachment ID を保持する
- ファイル名、種別、サイズなどを表示する
- 対応 plugin があれば「開く」「解析へ渡す」「viewer で表示」を出せる

#### `Core.ArtifactViewer`

成果物または添付参照を viewer plugin で描画する block。

- target ID を保持する
- target が attachment か artifact かを持つ
- viewer plugin が描画を担当する

#### `Core.AnalysisStep` / `Core.AnalysisChain`

解析実行 block。

- 使用する runner plugin
- 入力参照
- script path やパラメータ
- 実行結果の job / artifact 参照

最初は `AnalysisStep` 単体から始め、chain は複数 step の接続モデルとして後から広げてもよい。

## workspace レイアウト案

```text
Workspace/
  Notes/
  Attachments/
    sample01.lcd
    scripts/
      peak_pick.py
  .integral/
    jobs/
      <jobId>/
        request.json
        result.json
        stdout.log
        stderr.log
    artifacts/
      <artifactId>/
        manifest.json
        payload.json
```

意図:

- ユーザーが持ち込んだ元ファイルは `Attachments/`
- 実行履歴や中間成果物は `.integral/`
- ノートは ID 参照で両者をつなぐ

## 現在の plugin 契約

詳細は `docs/20_アーキテクチャ/pluginシステム.md` を参照。

現状の契約は概ね次の通り。

- plugin は `blocks[*].type` を manifest で宣言する
- renderer は `iframe` で block を描画する
- host は `runIntegralPluginAction` を export する
- renderer は app から block 情報を受ける
- renderer は app に `params` 更新か action 実行要求を返す
- host action の返り値は `summary` と `logLines` が中心である

つまり、現在の契約は「block を編集して、単発 action を叩く」ことに最適化されている。

## なぜ plugin 契約を拡張する必要があるか

### 1. いまの契約には「ファイル」の概念がない

現在の block は JSON `params` をやり取りするだけで、

- attachment とは何か
- どこに保存されているか
- どの plugin が読めるか

を表せない。

しかし今回やりたいことは「binary ファイルを起点にする」ことであり、ここを契約に載せないと app と plugin が同じ対象物を見られない。

### 2. いまの host action は「解析 step」の返り値として弱い

現在の host action は、ざっくり言うと

- actionId
- blockType
- params

を受けて、

- 成功したか
- サマリ
- ログ

を返すだけである。

これは「ボタンを押して何かする」には十分だが、解析 chain には足りない。

解析 step には最低でも

- どの入力を読んだか
- 何を出力したか
- 出力 artifact の ID は何か
- 再実行可能か
- 次の step がどれを入力にすべきか

が必要である。

つまり、単発 action をそのまま chain 実行へ流用すると、「成功したらしい」ことしか残らず、後続 step につなげられない。

### 3. いまの renderer 契約では viewer の入力が足りない

現在 renderer が受け取る中心情報は「この block の JSON」である。

だが viewer plugin は今後、

- attachment descriptor
- artifact descriptor
- 読み込み状態
- 生成元 step 情報

のような情報を受け取る必要がある。

たとえば LC viewer は「この attachment ID の `.lcd` を読め」だけではなく、

- どの plugin がその file type に対応するか
- 既に正規化済み artifact があるか
- viewer が raw binary を読むべきか、artifact を描くべきか

を知る必要がある。

### 4. `Python実行Plugin` を主軸にするなら、「plugin 本体」と「ユーザー script」を分けて表現しないといけない

plugin 自体は `%APPDATA%/IntegralNotes/plugins/*` に install される。  
一方で実行対象の `.py` は workspace 配下に置かれる可能性が高い。

つまり app は、

- install 済み plugin
- workspace 上の user-owned script
- script の入力ファイル
- script が出力した artifact

を区別して扱う必要がある。

現状契約は「install された plugin block を描く」ことには強いが、「plugin が workspace 上の任意 script を runner として実行する」ことは表現していない。

### 5. plugin 間連携をさせるには、共通の成果物契約が必要

理想は

- vendor plugin が binary から構造化データを作る
- Python plugin がその構造化データを加工する
- graph plugin が結果を描画する

という流れである。

これを成立させるには、plugin ごとの private データの受け渡しだけでは足りない。

最低でも

- artifact に ID がある
- kind / schema version がある
- 入出力関係が追える

という共通契約が必要になる。

## 何を足すべきか

最初の拡張は、次の 3 つに分けて考える。

### 1. Manifest の拡張

現在の `blocks` に加えて、少なくとも次の役割を宣言できるようにする。

- viewer contribution
  - どの attachment / artifact kind を描画できるか
- runner contribution
  - どの step type を実行できるか

これにより app は

- この添付はどの plugin が表示できるか
- この step はどの plugin が実行できるか

を判断できる。

### 2. Host 実行結果の拡張

現在の action result に加えて、step 実行用の返り値が必要になる。

最低限ほしいもの:

- status
- log
- output artifacts
- output manifest
- 実行ディレクトリ

ここは「UI action」と「解析 step 実行」を分けて契約化した方がよい。  
単なるボタン action と再現可能な解析 step は、意味が違うためである。

### 3. Renderer 入力モデルの拡張

renderer が block JSON だけでなく、参照対象の解決済み descriptor を受け取れるようにする。

最低限ほしいもの:

- attachment descriptor
- artifact descriptor
- viewer 用 meta
- 実行状態 / job 状態

これにより renderer は

- raw file viewer
- 成果物 viewer
- 実行中 step の状態表示

を同じ系で扱える。

## 本体と plugin の責務分担

### 本体が持つもの

- workspace と保存先管理
- attachment / artifact / job の ID 管理
- ノート上の参照 block 管理
- plugin 実行の仲介
- 実行履歴と最低限の監査

### plugin が持つもの

- binary 読み出しロジック
- vendor SDK / DLL / exe 呼び出し
- Python 実行ロジック
- 構造化データの定義
- 専用 viewer / 専用編集 UI

この分担にする理由は、本体を装置非依存のまま保ちつつ、解析の種類だけを plugin で増やせるようにするためである。

## `Python実行Plugin` の位置づけ

`Python実行Plugin` は「任意の py ファイルを実行できる plugin」であり、最も汎用的な runner になる想定。

最初の前提:

- 1 step = 1 script
- script は workspace 上の path を参照する
- 入力は attachment / artifact の ID から解決する
- 出力は artifact として `.integral/artifacts/` に保存する

この形なら、

- session 制御が不要
- 失敗時の切り戻しが簡単
- chain の入出力が明示的になる

ため、初期実装として扱いやすい。

## フェーズ案

### Phase 1

- Attachment モデル導入
- 任意ファイル upload
- `Core.AttachmentRef` 表示
- plugin なしでもファイル参照は残る

### Phase 2

- viewer contribution 導入
- attachment / artifact viewer block 導入
- LC などの binary viewer plugin を接続

### Phase 3

- runner contribution 導入
- `Python実行Plugin` による単発 step 実行
- artifact 永続化

### Phase 4

- 複数 step の chain
- artifact 依存関係の追跡
- 実行履歴 UI

## いったんの結論

今回必要なのは「plugin をもっと何でもできるようにする」ことではなく、  
**ファイル・成果物・解析 step を app と plugin が共通に理解できる最小契約を増やすこと**である。

現状契約は `itg-notes` block 描画には十分だが、分析データ ELN の中核である

- 任意 binary 添付
- binary の読解
- 解析結果の受け渡し
- chain 実行

を支えるには情報量が足りない。

したがって次の設計方針を採る。

- 添付は `Attachment`
- 解析結果は `Artifact`
- 実行単位は `Job / Step`
- ノートは ID 参照 block でこれらをつなぐ
- plugin 契約は viewer / runner / artifact を扱えるように広げる

この上で、まずは `Python実行Plugin` を主軸に MVP を組み、その後に装置専用 plugin を載せていく。
