# 全体アーキテクチャ

## 目的

IntegralNotes の全体構成を、今回確定した前提だけで整理する。

- block schema は共通
- `plugin` と `block-type` を分ける
- Python は `general-analysis` plugin で扱う
- `managed file` を基本にし、必要なときだけ `.idts` dataset を使う
- 標準表示は app 内蔵機能とする

## レイヤ

### 1. ノートレイヤ

- Markdown note
- 見たまま編集 / body-only raw text editor
- workspace file link
- workspace image attachment
- `itg-notes` code block

ここには block の宣言だけを保存する。

補足:

- note 本文の内部 link 記法は標準 Markdown link を使う
- note 本文の画像埋め込みは標準 Markdown image 記法を使う
- Python block は `run: relative/path.py:function` を canonical form にする
- 汎用 plugin block は `use: plugin-id/block-type` を canonical form にする
- editor は tab 内右上の mode toggle で `見たまま編集` と `body-only raw text` を切り替える

### 2. データレイヤ

- managed file
- dataset
- managed data registry
- data-note store
- hidden storage (`.store/`)
- metadata storage (`.store/.integral/`)
- runtime metadata / log
- dataset resolver

### 3. 実行レイヤ

- plugin 実行
- Python callable 実行
- 装置 plugin 実行
- output file / dataset 登録

### 4. 表示レイヤ

- block card
- 汎用 settings UI
- plugin 独自 settings UI
- 標準 viewer
- plugin viewer

## block の二層構造

IntegralNotes の block は、ユーザー向け source と app 内部の正規化表現を分ける。

### user-facing source

Python block 例:

authoring 時は input に path を書ける。

```itg-notes
id: BLK-1F8E2D0A
run: scripts/pca.py:main
in:
  samples: /Data/samples.csv
  labels: /Data/labels.csv
params:
  n_components: 2
out:
  score: /Results/score.csv
  report: /Results/report.html
```

保存または実行前には、input は ID へ正規化される。

```itg-notes
id: BLK-1F8E2D0A
run: scripts/pca.py:main
in:
  samples: FL-7K2M9Q4D
  labels: FL-1A2B3C4D
params:
  n_components: 2
out:
  score: /Results/score.csv
  report: /Results/report.html
```

実行成功後は output も ID へ書き換わり、block は provenance として read-only 表示になる。実行済み block には再実行 button を出さず、Delete と Undo を出してよい。Undo は生成済み output と参照を整理し、同じ block 定義から新しい draft を作り直す操作である。

```itg-notes
id: BLK-1F8E2D0A
run: scripts/pca.py:main
in:
  samples: FL-7K2M9Q4D
  labels: FL-1A2B3C4D
params:
  n_components: 2
out:
  score: FL-9X4Q2M1A
  report: FL-3N8K7Q2P
```

generic plugin block 例:

```itg-notes
id: BLK-6D3C2B1A
use: shimadzu-lc/run-sequence
in:
  method: FL-7K2M9Q4D
params:
  methodName: Gradient 01
out:
  raw-result: /Results/raw-result.lcd
```

### internal normalized form

app 内部では次の共通 schema に正規化する。

```json
{
  "id": "BLK-1F8E2D0A",
  "plugin": "general-analysis",
  "block-type": "scripts/pca.py:main",
  "params": {
    "n_components": 2
  },
  "inputs": {
    "samples": "FL-7K2M9Q4D",
    "labels": "FL-1A2B3C4D"
  },
  "outputs": {
    "score": "/Results/score.csv",
    "report": "/Results/report.html"
  }
}
```

## plugin の大分類

### core-display

app 内蔵の表示 plugin。  
renderable な file を読む。

`.idts` file を workspace file として開いた場合は、manifest viewer ではなく対応する dataset の data-note を開く。dataset の中身を表示したい場合は、dataset ID から readable directory path を resolve し、標準表示 block や viewer 側がその中の file を読む。

### general-analysis

workspace の `.py` を走査する汎用 Python 解析 plugin。  
decorator 付き関数を block 候補として公開し、`block-type` は `relative/path.py:function` になる。

### device / domain plugins

装置制御や専用 UI を持つ plugin。  
block-type は plugin 側が静的または独自動的に定義する。

## 標準表示と可視化 plugin の関係

MVP では次のように分ける。

- 標準表示
  - app が内蔵する
  - `html / image / text` を表示する
- 可視化 plugin
  - 非閲覧データを renderable な file 群を含む dataset に変換する処理 plugin

これにより、

- 表示ロジックを共通化できる
- 可視化 plugin を「特殊な解析 plugin」として扱える

## Python を専用資産化しない理由

MVP では Python file 自体を source of truth にする。  
理由は次のとおり。

- ユーザーは explorer を見ただけで script の所在を理解できる
- `.py-scripts/` への copy や独自資産 ID を減らせる
- `run: scripts/pca.py:main` だけで block source を読める

このため、Python 側は

- workspace file
- decorator metadata
- `analysis-args.json`

までに留める。

## 実行の考え方

block 実行時に app が行うのは「コンパイル」ではなく、実行準備である。

実行準備の役割:

- `BLK-...` の補完
- input ID の存在確認
- input に path が書かれている場合は managed data ID へ正規化
- `.idts` input の場合は dataset ID から `.idts` manifest file path へ resolve
- 非 `.idts` output の場合は target file path を確定
- `.idts` output の場合は visible output folder path を確定し、その folder 内に `{folder名}.idts` manifest を作る準備をする
- output metadata の作成または更新
- `.store/.integral/runtime/` への `analysis-args.json` と log 配置
- plugin 実行
- block source の `out:` への生成 managed data ID 反映

このとき app は slot ごとの ID 契約を維持しつつ、実行時だけ ID を current filesystem path に解決する。
Python 側の契約は `inputs / outputs` の絶対 path で揃える。

## ノートリンクと workspace file open

通常 note 本文から workspace file を開くときは、新しい file viewer 系を別に持たず、既存の workspace file open 経路を再利用する。

- `.md` は note tab として開く
- renderable / text は標準 viewer で開く
- unsupported は外部アプリ導線へ流す

また、IntegralNotes 内の explorer rename / move は、`cwd` 配下 `.md` の link target 更新 hook を持つ。  
詳細は `docs/30_設計/60_ノートリンク記法.md` を参照。

## ノート画像添付

note editor 内の画像添付もノートレイヤの責務として扱う。

- source 上は `![](/Data/...)` のような workspace root 相対 path を保持する
- insert / paste / drop の入口差は UI だけで、永続化 policy は共通 upload hook に寄せる
- editor 描画時だけ、workspace path を DOM 用の loadable URL へ変換する
- `Data/` に保存した画像は ordinary workspace file であり、他の workspace file と同様に managed file tracking 対象になりうる

## data-note

data-note は通常の workspace note とは別レイヤで扱う。

- `.md` managed file を除く managed data に data-note を持たせる
- data-note は `noteTargetId` に紐づく system-managed Markdown
- 保存場所は `.store/.integral/data-notes/{noteTargetId}.md`
- ユーザーは本文だけを編集し、rename / move は行わない
- app は path ではなく `noteTargetId` から data-note を開く

## 互換性方針

未リリースのため、旧 schema との互換は持たない。  
内部正規形としての

- `plugin`
- `block-type`
- `params`
- `inputs`
- `outputs`

を唯一の正としつつ、note source は人間が読める shorthand を採用する。
