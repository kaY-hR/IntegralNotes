# 全体アーキテクチャ

## 目的

IntegralNotes の全体構成を、今回確定した前提だけで整理する。

- block schema は共通
- `plugin` と `block-type` を分ける
- Python は `general-analysis` plugin で扱う
- `original data` と `dataset` を分ける
- 標準表示は app 内蔵機能とする

## レイヤ

### 1. ノートレイヤ

- Markdown note
- WYSIWYG editor
- `itg-notes` code block

ここには block の宣言だけを保存する。

### 2. データレイヤ

- original data
- dataset
- data note
- script asset (`PYS-...`)

### 3. 実行レイヤ

- plugin 実行
- Python 実行
- 装置 plugin 実行
- output dataset 作成

### 4. 表示レイヤ

- 汎用 settings UI
- plugin 独自 settings UI
- 標準表示 block
- plugin 独自 UI

## 共通 block schema

すべての block は同じ外形を持つ。

```json
{
  "id": "BLK-XXXX",
  "plugin": "plugin-id",
  "block-type": "type-id",
  "params": {},
  "inputs": {},
  "outputs": {}
}
```

意味:

- `plugin`
  - どの plugin が担当するか
- `block-type`
  - その plugin の中でどの処理/画面を表すか

## plugin の大分類

### core-display

app 内蔵の表示 plugin。  
renderable な dataset を読む。

### general-analysis

workspace の `.py-scripts/` を走査する汎用 Python 解析 plugin。  
`block-type` は `PYS-...` になる。

### device / domain plugins

装置制御や専用 UI を持つ plugin。  
block-type は plugin 側が静的または独自動的に定義する。

## 標準表示と可視化 plugin の関係

MVP では次のように分ける。

- 標準表示
  - app が内蔵する
  - `html / image / text` を表示する
- 可視化 plugin
  - 非閲覧データを renderable な dataset に変換する処理 plugin

これにより、

- 表示ロジックを共通化できる
- 可視化 plugin を「特殊な解析 plugin」として扱える

## Python を共通基盤へ入れすぎない理由

MVP では Python 実行を一般化しすぎない。  
理由は次のとおり。

- 解析スクリプトは使い捨てに近いことが多い
- 厳密な schema や env 管理を最初から強制すると重い
- まず欲しいのは「任意の `.py` を登録して実行できること」

したがって、Python 側は

- `script.json`
- `.py-scripts/PYS-.../`
- `analysis-args.json`

までに留める。

## 実行の考え方

block 実行時に app が行うのは「コンパイル」ではなく、実行準備である。

実行準備の役割:

- `BLK-...` の補完
- input dataset の存在確認
- output dataset の新規作成
- `analysis-args.json` の生成
- plugin 実行
- block `outputs` の書き戻し

## 互換性方針

未リリースのため、旧 schema との互換は持たない。  
今回決める

- `plugin`
- `block-type`
- `params`
- `inputs`
- `outputs`

を唯一の正とする。
