# データモデル

## ID 体系

- original data: `BLB-...`
- dataset: `CNK-...`
- block: `BLK-...`
- Python script asset: `PYS-...`

ID は意味を持たない opaque ID とする。  
表示名や検索性は別フィールドに持たせる。

## Original data

### 定義

original data は、ユーザーが持ち込む元データの永続単位である。

### 性質

- 原本
- ユーザーが認識する基本単位
- `1 original data = 1 data note`

### 例

- 生の測定ファイル
- 生の測定フォルダ
- 手で追加した CSV やテキスト

## Data note

data note は、ノート上でデータを参照するための Markdown facade である。

### 方針

- original data には必ず data note を作る
- dataset は必要なものだけ data note を作る

これにより、

- 元データは常にノートから参照しやすい
- 中間成果物はノイズを増やしすぎない

## Dataset

### 定義

dataset は、すべての処理 block が input / output として扱う単位である。  
実体はフォルダとする。

### 性質

- immutable
- `1 input slot = 1 dataset`
- `1 output slot = 1 dataset`
- block には `datasetId` だけを書く

## source dataset

original data 選択の結果を処理 block に渡すための dataset。

### 性質

- `kind` は固定で `source-bundle`
- `createdByBlockId` は `null`
- 1 slot に original data を複数選んだ場合でも、内部では 1 source dataset に束ねる
- 同じ original data 組み合わせでも毎回新しく作る

### 構成

- `dataset.json`
- `links.json`

`links.json` は、元の original data を参照するための軽い manifest である。

## derived dataset

処理 block の output として生成される dataset。

### 性質

- `createdByBlockId` を持つ
- `kind` は script/plugin が明示しなければ `block-type.slotName`
- 実データファイル群を中に持つ

## dataset.json

全 dataset に `dataset.json` を持たせる。  
source / derived で内容差はあってよいが、形式は揃える。

最小例:

```json
{
  "datasetId": "CNK-7K2M9Q4D",
  "kind": "source-bundle",
  "createdByBlockId": null,
  "createdAt": "2026-04-13T10:00:00+09:00"
}
```

derived 例:

```json
{
  "datasetId": "CNK-9X4Q2M1A",
  "kind": "PYS-7K2M9Q4D.result",
  "createdByBlockId": "BLK-1F8E2D0A",
  "createdAt": "2026-04-13T10:05:00+09:00"
}
```

## Block

block はノート上の処理宣言である。

### 必須キー

- `id`
- `plugin`
- `block-type`
- `params`
- `inputs`
- `outputs`

### block の責務

- どの plugin が担当するか
- どの block-type か
- どの dataset を input に使うか
- 現在どの dataset を output に持っているか

### block に持たせないもの

- 実行コマンド
- Python の絶対パス
- 一時ログ
- 元の選択ファイルの絶対パス

## Python script asset

`general-analysis` plugin が扱う script 資産。

### 性質

- `PYS-...` ID を持つ
- `.py-scripts/PYS-.../` に保存される
- block-type はそのまま `PYS-...`

### 最小ファイル

- `script.json`
- `entry` で指定された `.py`
- 自動/手動で同梱された補助ファイル

## Provenance

MVP では provenance は次で追える。

- block の `inputs`
- block の `outputs`
- dataset の `createdByBlockId`
- source dataset の `links.json`

これで、後から block と dataset の関係を scan で復元できる。

## GC

dataset は immutable だが、永遠に保持するわけではない。

GC 候補:

- どの block の `inputs` にもない
- どの block の `outputs` にもない
- data note として公開されてもいない

これにより、source dataset や旧 output dataset が増えすぎるのを防ぐ。
