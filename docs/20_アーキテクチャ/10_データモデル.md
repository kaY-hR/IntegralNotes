# データモデル

## ID 体系

- original data: `ORD-...`
- dataset: `DTS-...`
- block: `BLK-...`

ID は意味を持たない opaque ID とする。  
表示名や検索性は別フィールドに持たせる。

## managed data の共通モデル

original data / dataset は、どちらも共通の managed data として扱う。

最小フィールド:

- `id`
- `entityType`
- `representation`
- `path`
- `hash`
- `visibility`
- `provenance`
- `displayName`
- `createdAt`

意味:

- `id`
  - stable identity
- `entityType`
  - `original-data` または `dataset`
- `representation`
  - `file`, `directory`, `dataset-json`
- `path`
  - current canonical workspace relative path
- `hash`
  - current content identity
- `visibility`
  - `visible` または `hidden`
- `provenance`
  - `source` または `derived`

metadata は `.store/.integral/{id}.json` に置く。  
`path` は current workspace relative path を表し、original data では visible path を基本とする。  
このモデルでは、`Data/` は default location に過ぎず、特別な canonical 意味は持たない。

## path / hash tracking

app は managed data を `id / path / hash` で追跡する。

基本規則:

- `path same + hash same`
  - 変更なし
- `path changed + hash same`
  - rename / move
- `path same + hash changed`
  - 内容更新
- `path changed + hash changed`
  - 候補が一意なら追跡、曖昧なら confirm

hash 方針:

- `file`
  - content hash
- `directory`
  - relative path と child hash 一覧から作る tree hash
- `dataset-json`
  - manifest content hash

## Original data

### 定義

original data は、ユーザーが持ち込む元データの永続単位である。

### 性質

- 原本
- ユーザーが認識する基本単位
- `1 original data = 1 data-note`
- current path は visible workspace path を基本とする
- tracking と provenance の仕組みは dataset と共通にする

### representation

- `file`
- `directory`

### 補足

- original data の canonical 実体を `.store/objects` に隠し、visible path を alias とする構成は採らない
- `cwd` 外から登録した original data は、workspace 内の default visible path へ copy して管理する
- source dataset の materialize や data-note metadata は、original data の current visible path を正として扱う

## Note attachment image

note attachment image は、note editor から保存される workspace 内画像 file である。

### 性質

- note 本文の `![alt](target)` から参照される
- 既定保存先は `Data/yyyyMMdd-HHmm-RRR.ext`
- ordinary workspace file であり、original data ではない
- `ORD-...` のような opaque ID を持たない
- `.store/.integral/*.json` metadata も自動生成しない

## Data note

data-note は、managed data にぶら下がる system-managed Markdown である。

### 方針

- original data には必ず data-note を作る
- dataset にも必ず data-note を作る
- data-note は `.store/.integral/data-notes/{id}.md` に置く
- data-note は user-managed workspace file としては扱わない
- data-note の target は file 名や path ではなく `id` で解決する
- ユーザーは本文だけを編集できる
- rename / move は system 管理とし、ユーザーにはさせない
- data-note 本文は「この data は何か」の説明を主目的とする

## Markdown note

通常の Markdown note でも frontmatter を許容する。

### 方針

- `cwd` 配下の `.md` は read / save 時に frontmatter と本文を標準的に分離する
- app の editor / viewer には本文だけを渡し、frontmatter は hidden metadata として保持する
- rename / move に伴う link / image target の rewrite は本文だけに適用し、frontmatter は user 管理の metadata としてそのまま保持する
- data-note は通常 Markdown note とは別の system-managed note として扱う

## Dataset

### 定義

dataset は、すべての処理 block が input / output として扱う単位である。

### 性質

- `1 input slot = 1 dataset`
- `1 output slot = 1 dataset`
- `1 dataset = 1 data-note`
- dataset は `datasetId` とは別に `name` を持つ
- metadata は `.store/.integral/{datasetId}.json`
- path / hash tracking は original data と共通にする

### representation

- `dataset-json`

dataset の canonical entrypoint は source / derived を問わず `.idts` manifest とする。  
実行や表示で file system access が必要な場合、app は datasetId から manifest を読み、read-only な directory path へ resolve する。

## source dataset

original data 選択の結果を処理 block に渡すための dataset。

### 性質

- `kind` は固定で `source-bundle`
- `createdByBlockId` は `null`
- ユーザーが明示作成する場合は `name` を入力する
- 1 slot に original data を複数選んだ場合でも、内部では 1 source dataset に束ねる
- current path は visible `.idts` を default にしてよい

## derived dataset

処理 block の output として生成される dataset。

### 性質

- `createdByBlockId` を持つ
- `kind` は script/plugin が明示しなければ `block-type.slotName`
- system 既定の `name` は `datasetId` でよい
- 実データファイル群は `.store/objects/{datasetId}/` 配下に持てる
- current path は hidden `.store/.integral/datasets/{datasetId}.idts` を default にしてよい
- block 再実行時は新しい derived dataset を作る

## dataset metadata

全 dataset に `.store/.integral/{datasetId}.json` を持たせる。  
source / derived で内容差はあってよいが、形式は揃える。

最小例:

```json
{
  "id": "DTS-7K2M9Q4D",
  "entityType": "dataset",
  "representation": "dataset-json",
  "displayName": "samples",
  "path": "datasets/samples.idts",
  "hash": "sha256:6F4E...",
  "visibility": "visible",
  "provenance": "source",
  "kind": "source-bundle",
  "createdByBlockId": null,
  "createdAt": "2026-04-13T10:00:00+09:00",
  "memberIds": ["ORD-1A2B3C4D"]
}
```

derived 例:

```json
{
  "id": "DTS-9X4Q2M1A",
  "entityType": "dataset",
  "representation": "dataset-json",
  "displayName": "DTS-9X4Q2M1A",
  "path": ".store/.integral/datasets/DTS-9X4Q2M1A.idts",
  "hash": "sha256:91A2...",
  "visibility": "hidden",
  "provenance": "derived",
  "kind": "scripts/pca.py:main.score",
  "createdByBlockId": "BLK-1F8E2D0A",
  "createdAt": "2026-04-13T10:05:00+09:00",
  "dataPath": ".store/objects/DTS-9X4Q2M1A"
}
```

## Block

block はノート上の処理宣言である。

### internal normalized key

- `id`
- `plugin`
- `block-type`
- `params`
- `inputs`
- `outputs`

### block の責務

- どの plugin が担当するか
- どの block-type か
- どの dataset を input に使うか
- どの output slot を持つか

### block に持たせないもの

- 実行コマンド
- Python の絶対パス
- 一時ログ
- 元の選択ファイルの絶対パス

### 補足

- note source 上では `run:` / `use:` shorthand を許容する
- `.idts` path は app が `datasetId` に解決する
- 最新 output dataset の `.idts` path は block source の `out:` に保持してよい

## Python callable definition

`general-analysis` plugin が扱う Python callable 定義は、managed data ではない。

### 性質

- ID ではなく `relative/path.py:function` で識別する
- workspace file 自体が source of truth
- decorator から `displayName`, `description`, `inputs`, `outputs` を読む
- `.py-scripts/` のような専用コピー領域は持たない

## Provenance

MVP では provenance は次で追える。

- block の `inputs`
- block source の `out:` に書かれた latest output `.idts` path
- dataset の `createdByBlockId`
- source / derived dataset manifest の `memberIds` / `dataPath`

これで、後から block と dataset の関係を scan で復元できる。

## GC

block 実行で生成される output dataset は immutable だが、永遠に保持するわけではない。

GC 候補:

- どの block の `inputs` にもない
- どの block の `out:` にも出てこない
- data-note として pin されてもいない

これにより、source dataset や旧 output dataset が増えすぎるのを防ぐ。
