# データモデル

## ID 体系

- managed file: opaque ID
- block: `BLK-...`
- datatype: 任意の string

managed file の prefix 自体は implementation detail とする。  
この文書では例として `MF-...` を使う。

## Managed File

### 定義

managed file は、workspace 内で ID 追跡される file である。  
中心の追跡単位は managed file とし、複数 file を束ねたい場合だけ `.idts` dataset を使う。

### スコープ

- `cwd` 内の通常 file は、`.store/` などの system-managed path を除き managed file にできる
- `.md` も managed file に含める
- `.store/.integral/data-notes/*.md` のような system-managed note は別扱いにしてよい

### 最小フィールド

- `datasetId`
- `path`
- `hash`
- `datatype`
- `createdByBlockId`

必要なら後から次を足せる。

- `createdAt`
- `displayName`
- `bundleRootPath`
- `bundleMemberIds`

### 意味

- `id`
  - stable identity
- `path`
  - current canonical workspace relative path
- `hash`
  - current content identity
- `datatype`
  - 解析間の I/O 互換性を表す意味ラベル
- `createdByBlockId`
  - `null` なら user/source 起点
  - `BLK-...` なら block 生成物

metadata は `.store/.integral/` 配下に持つ。  
物理実装は per-file JSON でも SQLite でもよいが、logical model はこの形に揃える。

## Path / Hash Tracking

app は managed file を `id / path / hash` で追跡する。

基本規則:

- `path same + hash same`
  - 変更なし
- `path changed + hash same`
  - rename / move
- `path same + hash changed`
  - 内容更新
- `path changed + hash changed`
  - 候補が一意なら追跡、曖昧なら confirm

hash 方針:

- 通常 file
  - content hash
- `.idts`
  - manifest content hash

`path -> id` は current lookup index として持てるが、identity の source of truth ではない。

## Datatype

datatype は managed file / dataset / slot が共有する意味ラベルである。
物理的な suffix や `.idts` 表現は `extension(s)` と file representation で扱い、datatype では扱わない。

MVP では datatype 専用 registry や ID を持たず、任意の string とする。
output slot が生成する datatype を宣言し、生成物 metadata に同じ datatype を保存する。
input slot は要求 datatype を宣言し、picker / validation は完全一致を強い候補として扱う。

運用上、共有される skill や user 定義の datatype は `{userId}/xxxx` のように namespace を付けることを推奨する。
ただし namespace は enforcement せず、衝突回避のための慣習に留める。

例:

- `shimadzu-lc/chromatogram-json`
- `shimadzu-lc/pca-result-bundle`
- `demo/table-csv`
- `{userId}/html-report`

datatype は slot 制約、検索、将来の graph 構築に使える。viewer 選択はまず拡張子や representation を見る。

## Bundle (`.idts`)

### 定義

`.idts` は optional な bundle manifest file である。  
1 slot で複数 file を扱いたいときに使う。

### 性質

- visible workspace file である
- それ自体も managed file として追跡する
- 1 file が複数 bundle に所属できる
- runtime では hidden bundle directory に resolve できる

### manifest が持てる情報

- `id`
- `name`
- `memberIds`

必要に応じて次も持てる。

- `dataPath`
- `datatype`
- `noteTargetId`

### 例

```json
{
  "datasetId": "DTS-1A2B3C4D",
  "name": "samples",
  "datatype": "demo/source-bundle",
  "dataPath": ".store/objects/DTS-1A2B3C4D",
  "memberIds": ["MF-11111111", "MF-22222222"]
}
```

## Data Note

data-note は、`.md` managed file を除く managed data にぶら下がる system-managed Markdown である。

方針:

- `.md` を除く managed file と dataset に data-note を作る
- data-note は `.store/.integral/data-notes/{id}.md` に置く
- data-note の target は file path ではなく `id` で解決する
- ユーザーは本文だけを編集できる

## Markdown Note

通常の Markdown note も managed file である。

方針:

- `cwd` 配下の `.md` は read / save 時に frontmatter と本文を標準的に分離する
- `.md` managed file には別の data-note をぶら下げない
- app の editor / viewer には本文だけを渡し、frontmatter は hidden metadata として保持する
- rename / move に伴う link / image target の rewrite は本文だけに適用する

## Block

block はノート上の処理宣言である。

### internal normalized key

- `id`
- `plugin`
- `block-type`
- `params`
- `inputs`
- `outputs`

### block の責務

- どの plugin が担当するか
- どの block-type か
- どの managed data ID を input に使うか
- 実行前にどの path へ output を書くか
- 実行後にどの managed data ID が output として生成されたか

### 補足

- note source 上では `run:` / `use:` shorthand を許容する
- `inputs` は保存上 managed data ID を正とする
- `outputs` は実行前 path、実行後 managed data ID を正とする
- authoring 時の input path は、保存または実行前に current index で ID へ解決する
- block 自体は managed file ではなく、note source の一部である
- 実行済み block は provenance として read-only 表示にする

## Slot

slot は Python callable や plugin block の I/O 契約を表す。

### 最小フィールド

- `name`
- `extension` または `extensions`
- `datatype`

例:

```json
{
  "name": "samples",
  "extensions": [".csv"],
  "datatype": "demo/table-csv"
}
```

```json
{
  "name": "report",
  "extension": ".html",
  "datatype": "demo/html-report"
}
```

`.idts` は拡張子の 1 つであり、slot が bundle を要求することを表す。

## Provenance

専用の `provenance` enum は必須にしない。  
生成物かどうかは `createdByBlockId` の有無で表す。

- `createdByBlockId = null`
  - source / user 起点
- `createdByBlockId = BLK-*`
  - block 生成物

## 関係の復元

MVP では次から relation を復元できる。

- managed file metadata の `createdByBlockId`
- block の `inputs`
- block の `outputs`
- bundle manifest の `memberIds`

これで、後から block と file の derive 関係を scan で再構築できる。
