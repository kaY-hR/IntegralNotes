# データモデル

## ID 体系

- blob: `BLB-...`
- chunk: `CNK-...`
- block: `BLK-...`
- Python script asset: `PYS-...`

ID は意味を持たない opaque ID とする。  
表示名や検索性は別フィールドに持たせる。

## Blob

### 定義

blob は、ユーザーが持ち込む元データの永続単位である。

### 性質

- 原本
- ユーザーが認識する基本単位
- `1 blob = 1 artifact note`

### 例

- 生の測定ファイル
- 生の測定フォルダ
- 手で追加した CSV やテキスト

## Artifact note

artifact note は、ノート上でデータを参照するための Markdown facade である。

### 方針

- blob には必ず artifact note を作る
- chunk は必要なものだけ artifact note を作る

これにより、

- 元データは常にノートから参照しやすい
- 中間成果物はノイズを増やしすぎない

## Chunk

### 定義

chunk は、すべての処理 block が input / output として扱う単位である。  
実体はフォルダとする。

### 性質

- immutable
- `1 input slot = 1 chunk`
- `1 output slot = 1 chunk`
- block には `chunkId` だけを書く

## source chunk

blob 選択の結果を処理 block に渡すための chunk。

### 性質

- `kind` は固定で `source-bundle`
- `createdByBlockId` は `null`
- 1 slot に blob を複数選んだ場合でも、内部では 1 source chunk に束ねる
- 同じ blob 組み合わせでも毎回新しく作る

### 構成

- `chunk.json`
- `links.json`

`links.json` は、元の blob を参照するための軽い manifest である。

## derived chunk

処理 block の output として生成される chunk。

### 性質

- `createdByBlockId` を持つ
- `kind` は script/plugin が明示しなければ `block-type.slotName`
- 実データファイル群を中に持つ

## chunk.json

全 chunk に `chunk.json` を持たせる。  
source / derived で内容差はあってよいが、形式は揃える。

最小例:

```json
{
  "chunkId": "CNK-7K2M9Q4D",
  "kind": "source-bundle",
  "createdByBlockId": null,
  "createdAt": "2026-04-13T10:00:00+09:00"
}
```

derived 例:

```json
{
  "chunkId": "CNK-9X4Q2M1A",
  "kind": "PYS-7K2M9Q4D.result",
  "createdByBlockId": "BLK-1F8E2D0A",
  "createdAt": "2026-04-13T10:05:00+09:00"
}
```

## Block

block はノート上の処理宣言である。

### 必須キー

- `id`
- `plugin`
- `block-type`
- `params`
- `inputs`
- `outputs`

### block の責務

- どの plugin が担当するか
- どの block-type か
- どの chunk を input に使うか
- 現在どの chunk を output に持っているか

### block に持たせないもの

- 実行コマンド
- Python の絶対パス
- 一時ログ
- 元の選択ファイルの絶対パス

## Python script asset

`general-analysis` plugin が扱う script 資産。

### 性質

- `PYS-...` ID を持つ
- `.py-scripts/PYS-.../` に保存される
- block-type はそのまま `PYS-...`

### 最小ファイル

- `script.json`
- `entry` で指定された `.py`
- 自動/手動で同梱された補助ファイル

## Provenance

MVP では provenance は次で追える。

- block の `inputs`
- block の `outputs`
- chunk の `createdByBlockId`
- source chunk の `links.json`

これで、後から block と chunk の関係を scan で復元できる。

## GC

chunk は immutable だが、永遠に保持するわけではない。

GC 候補:

- どの block の `inputs` にもない
- どの block の `outputs` にもない
- artifact note として公開されてもいない

これにより、source chunk や旧 output chunk が増えすぎるのを防ぐ。
