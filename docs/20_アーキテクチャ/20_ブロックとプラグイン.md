# ブロックとプラグイン

## 目的

共通 block schema と plugin 責務を定義する。  
ここでの主題は「どう実装するか」ではなく、「何を誰が持つか」である。

## 共通 block schema

すべての block は次の schema を使う。

```json
{
  "id": "BLK-1F8E2D0A",
  "plugin": "general-analysis",
  "block-type": "PYS-7K2M9Q4D",
  "params": {},
  "inputs": {
    "source": "CNK-7K2M9Q4D"
  },
  "outputs": {
    "result": null
  }
}
```

### ルール

- `id` は `BLK-...`
- `plugin` は固定 plugin ID
- `block-type` は plugin 内で stable な type ID
- `params` は JSON object
- `inputs` は `Record<string, string | null>`
- `outputs` は `Record<string, string | null>`

### 補足

- `inputs` の `null` は未設定を表す
- `outputs` の `null` は未実行を表す
- output slot 名は先に書いておく

## なぜ `plugin` と `block-type` を分けるか

旧設計では `type` が両方の役割を兼ねていた。  
しかし、今回の整理では次を分けたい。

- `plugin`
  - どの plugin が担当するか
- `block-type`
  - その plugin の中のどの処理/画面か

これにより、

- `general-analysis` の中で多数の `PYS-...` を扱える
- 装置 plugin でも Python plugin でも schema を共通にできる

## plugin の役割

plugin は次の情報または機能を持つ。

- plugin ID
- displayName
- description
- block-type 一覧の提供
- block 実行
- 必要なら custom settings UI

MVP では「custom result UI」は必須契約にしない。  
結果表示は原則として標準表示 block に寄せる。

## block-type 一覧の提供方法

### 静的 block-type

装置 plugin などでは、plugin 側が block-type を固定で持つ。

例:

- `shimadzu-lc` plugin
  - `run-sequence`
  - `export-chromatogram`

### 動的 block-type

`general-analysis` plugin は `cwd/.py-scripts/` を走査し、`script.json` を読んで block-type 一覧を動的生成する。

例:

- `general-analysis` plugin
  - `PYS-7K2M9Q4D`
  - `PYS-9X4Q2M1A`

## settings UI

### 汎用 settings UI

MVP では block schema から最低限の編集 UI を生成する。

- input slot への chunk 割当
- output slot の確認
- Python plugin では `params` は `{}` 固定

### custom settings UI

装置 plugin など、特殊な設定が必要な plugin は custom settings UI を持てる。  
ただし block に保存される内容は共通 schema へ還元する。

## 結果表示

結果表示は plugin 契約の中心にしない。  
MVP では次で整理する。

- block は output chunk を生成する
- renderable なファイルの表示は `core-display` が担当する
- 非閲覧データを renderable な結果へ変換したい場合は、別の処理 block を使う

## `general-analysis` plugin の特例

`general-analysis` は plugin システムの中では特例的だが、block schema 自体は特例にしない。

特例なのは次だけである。

- block-type が `PYS-...` の動的 ID
- `.py-scripts/` を workspace 資産として持つ
- Python 実行を行う
- `params` を MVP では使わない

## app 本体の責務

- block schema の validation
- `BLK-...` の自動補完
- chunk 作成
- plugin の発見
- block-type 一覧の取得
- block 実行の仲介
- `outputs` の書き戻し

## plugin 側の責務

- block-type の意味付け
- 実行内容の実装
- 必要なら custom settings UI
- output chunk への結果書き出し

## 非目標

MVP では次を plugin 契約へ入れない。

- Jupyter セッション管理
- JSON Schema ベースの params フォーム生成
- result UI override の一般化
- kind 制約の厳密 enforcement
