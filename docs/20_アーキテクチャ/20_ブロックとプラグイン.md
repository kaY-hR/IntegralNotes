# ブロックとプラグイン

## 目的

共通 block schema と plugin 責務を定義する。  
ここでの主題は「どう実装するか」ではなく、「何を誰が持つか」である。

## block の二層構造

### note source

ユーザーが note 上で読む block は、人が理解しやすい source とする。

Python 例:

```itg-notes
id: BLK-1F8E2D0A
run: scripts/pca.py:main
in:
  samples: /Data/samples.idts
params:
  n_components: 2
out:
  result: auto
```

generic plugin 例:

```itg-notes
id: BLK-6D3C2B1A
use: shimadzu-lc/run-sequence
in:
  method: /Data/method.idts
params:
  methodName: Gradient 01
out:
  raw-result: auto
```

### internal normalized schema

app 内部では、すべての block を次の schema に正規化する。

```json
{
  "id": "BLK-1F8E2D0A",
  "plugin": "general-analysis",
  "block-type": "scripts/pca.py:main",
  "params": {
    "n_components": 2
  },
  "inputs": {
    "samples": "DTS-7K2M9Q4D"
  },
  "outputs": {
    "result": null
  },
  "outputConfigs": {
    "result": {
      "directory": "/Data",
      "name": "result"
    }
  }
}
```

### ルール

- `id` は `BLK-...`
- `plugin` は固定 plugin ID
- `block-type` は plugin 内で stable な type ID
- `params` は JSON object
- `inputs` は `Record<string, string | null>`
- `outputs` は `Record<string, string | null>`
- `outputConfigs` は `Record<string, { directory: string; name: string } | null>`

### 補足

- user-facing source の `.idts` path は app が `datasetId` に解決する
- `outputs` の `null` は未実行の `auto` を表す
- `outputConfigs` は「次回実行時にどこへ何名で出すか」の user-facing 設定を表す
- 実行後は output slot ごとの visible `.idts` path を note source の `out:` に書き戻してよい
- user-facing な dataset 表示は dataset 名を主とし、ID は内部参照に限る

## なぜ `plugin` と `block-type` を分けるか

今回の整理では次を分けたい。

- `plugin`
  - どの plugin が担当するか
- `block-type`
  - その plugin の中のどの処理/画面か

これにより、

- `general-analysis` の中で多数の Python callable を扱える
- 装置 plugin でも Python plugin でも schema を共通にできる

## plugin の役割

plugin は次の情報または機能を持つ。

- plugin ID
- displayName
- description
- block-type 一覧の提供
- file viewer の提供
- sidebar view の提供
- block 実行
- 必要なら custom settings UI

MVP では「custom result UI」は plugin 契約の中心にしない。  
結果表示は原則として標準 viewer / viewer plugin に寄せる。

## block-type 一覧の提供方法

### 静的 block-type

装置 plugin などでは、plugin 側が block-type を固定で持つ。

例:

- `shimadzu-lc` plugin
  - `run-sequence`
  - `export-chromatogram`

### 動的 block-type

`general-analysis` plugin は workspace の `.py` を走査し、decorator 付き関数を block-type 一覧として動的生成する。

例:

- `general-analysis` plugin
  - `scripts/pca.py:main`
  - `scripts/normalize.py:run`

## settings UI

### 汎用 settings UI

MVP では block schema から最低限の編集 UI を生成する。

- input slot への dataset 割当
- param object の編集
- output slot の確認

### custom settings UI

装置 plugin など、特殊な設定が必要な plugin は custom settings UI を持てる。  
ただし block に保存される内容は共通 schema へ還元する。

## 結果表示

結果表示は plugin 契約の中心にしない。  
MVP では次で整理する。

- block は output dataset を生成する
- renderable なファイルの表示は `core-display` または viewer plugin が担当する
- 非閲覧データを renderable な結果へ変換したい場合は、別の処理 block を使う

## 左サイド拡張

左サイドの pane は `sidebar view` contribution として扱う。

- Activity Bar shell は app core
- Explorer / Plugins は built-in sidebar view
- external plugin も `sidebarViews` で同じ host に pane を追加できる

これは viewer plugin とは別の責務である。

## `general-analysis` plugin の特例

`general-analysis` は plugin システムの中では特例的だが、block schema 自体は特例にしない。

特例なのは次だけである。

- block-type が `relative/path.py:function` の動的 ID
- workspace file 自体が source of truth
- Python 実行を行う
- slot 定義や display 情報を decorator から読む

## app 本体の責務

- block source の parse / normalize
- `BLK-...` の自動補完
- dataset 作成
- plugin の発見
- block-type 一覧の取得
- block 実行の仲介
- output dataset の生成と `.idts` path の block source 反映

## plugin 側の責務

- block-type の意味付け
- 実行内容の実装
- 必要なら custom settings UI
- output dataset への結果書き出し

## 非目標

MVP では次を plugin 契約へ入れない。

- Jupyter セッション管理
- JSON Schema ベースの params フォーム生成
- result UI override の一般化
- kind 制約の厳密 enforcement
