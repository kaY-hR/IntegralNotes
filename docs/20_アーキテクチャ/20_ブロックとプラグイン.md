# ブロックとプラグイン

## 目的

共通 block schema と plugin 責務を定義する。  
ここでの主題は「どう実装するか」ではなく、「何を誰が持つか」である。

## Block の二層構造

### note source

ユーザーが note 上で読む block は、人が理解しやすい source とする。

Python 例:

```itg-notes
id: BLK-1F8E2D0A
run: scripts/pca.py:main
in:
  samples: /Data/samples.csv
params:
  n_components: 2
out:
  score: /Results/score.csv
```

bundle を使う例:

```itg-notes
id: BLK-6D3C2B1A
run: scripts/report.py:main
in:
  source: /Data/samples.idts
params: {}
out:
  report: /Results/report.html
```

### internal normalized schema

app 内部では、すべての block を次の schema に正規化する。

```json
{
  "id": "BLK-1F8E2D0A",
  "plugin": "general-analysis",
  "block-type": "scripts/pca.py:main",
  "params": {
    "n_components": 2
  },
  "inputs": {
    "samples": "/Data/samples.csv"
  },
  "outputs": {
    "score": "/Results/score.csv"
  }
}
```

### ルール

- `id` は `BLK-...`
- `plugin` は固定 plugin ID
- `block-type` は plugin 内で stable な type ID
- `params` は JSON object
- `inputs` は `Record<string, string | null>`
- `outputs` は `Record<string, string | null>`

### 補足

- user-facing source の path は app が runtime path に解決する
- `.idts` path は bundle manifest として特別扱いされる
- UI 上の output path editor は内部で `directory + stem` に分けて保持してもよいが、logical contract は path とする

## なぜ `plugin` と `block-type` を分けるか

- `plugin`
  - どの plugin が担当するか
- `block-type`
  - その plugin の中のどの処理か

これにより、`general-analysis` の中で多数の Python callable を扱える。

## Plugin の役割

plugin は次の情報または機能を持つ。

- plugin ID
- displayName
- description
- block-type 一覧の提供
- file viewer の提供
- sidebar view の提供
- block 実行
- 必要なら custom settings UI

## Slot の意味

block-type は input slot / output slot を定義する。  
slot は少なくとも次を持てる。

- `name`
- `extension` または `extensions`
- `format`

MVP では `1 slot = 1 path` を基本とする。  
複数 file を扱いたい場合は `.idts` bundle を 1 path として渡す。

## `general-analysis` plugin の特例

`general-analysis` は plugin システムの中では特例的だが、block schema 自体は特例にしない。

特例なのは次だけである。

- block-type が `relative/path.py:function` の動的 ID
- workspace file 自体が source of truth
- Python 実行を行う
- slot 定義や display 情報を decorator から読む

## app 本体の責務

- block source の parse / normalize
- `BLK-...` の自動補完
- managed file metadata の追跡
- plugin の発見
- block-type 一覧の取得
- block 実行の仲介
- path 参照の検証と rewrite

## plugin 側の責務

- block-type の意味付け
- 実行内容の実装
- 必要なら custom settings UI
- output path への結果書き出し

## 非目標

MVP では次を plugin 契約へ入れない。

- Jupyter セッション管理
- JSON Schema ベースの params フォーム生成
- result UI override の一般化
- format 制約の厳密 enforcement
