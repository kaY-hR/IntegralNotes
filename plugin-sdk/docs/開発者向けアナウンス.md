# Plugin 開発者向けアナウンス

## 前提

IntegralNotes の plugin は `external plugin` のみを前提にしています。  
app 本体が読むのは `userData/plugins/*` に install された artifact だけです。

repo 内の `plugins/*` は plugin source の置き場であり、runtime の install 先ではありません。

## 最小構成

```text
my-plugin/
  integral-plugin.json
  renderer/
    index.html
  host/
    index.cjs
```

## 現在の contract

### manifest

- file name
  - `integral-plugin.json`
- schema
  - `plugin-sdk/schemas/integral-plugin.schema.json`
- 必須
  - `apiVersion`
  - `id`
  - `namespace`
  - `displayName`
  - `description`
  - `version`
- contribution
  - `blocks`
  - `viewers`
  - `sidebarViews`
  - この 3 つのうち少なくとも 1 つは必須

### renderer

- `iframe` mode のみ
- app から `postMessage` で block 情報が渡る
- message type
  - app -> plugin
    - `integral:set-block`
    - `integral:set-viewer`
    - `integral:set-sidebar-view`
    - `integral:action-state`
    - `integral:viewer-resolve-workspace-file-url-result`
  - plugin -> app
    - `integral:update-params`
    - `integral:request-action`
    - `integral:viewer-resolve-workspace-file-url`
- plugin renderer は block JSON 全体の保存責務は持たないが、block UI 全体の見た目と操作導線は持ってよい
- app が `renderer/index.html` を `srcDoc` に流し込む際、相対 asset path は `renderer/index.html` 基準で解決される
  - 例: `renderer/index.html` から `../gradient.png` を参照可能

renderer 実装の最小例:

```js
window.addEventListener("message", (event) => {
  if (!event.data || event.data.type !== "integral:set-block") {
    return;
  }

  render(event.data.payload);
});

function updateParams(nextParams) {
  window.parent.postMessage(
    {
      type: "integral:update-params",
      payload: {
        params: nextParams
      }
    },
    "*"
  );
}

function requestAction(actionId) {
  window.parent.postMessage(
    {
      type: "integral:request-action",
      payload: {
        actionId
      }
    },
    "*"
  );
}
```

### viewer renderer

viewer contribution では app から `integral:set-viewer` が渡る。

```js
window.addEventListener("message", (event) => {
  if (!event.data || event.data.type !== "integral:set-viewer") {
    return;
  }

  const model = event.data.payload;
  renderViewer(model.file.data, model.presentation);
});
```

`model.file.data` は text file なら UTF-8 text、binary file なら data URL で渡る。  
`model.presentation` は `"full"` または `"embed"`。

viewer が表示対象 file から参照される別 workspace file を読む必要がある場合は、parent window へ request を送る。

```js
window.parent.postMessage(
  {
    type: "integral:viewer-resolve-workspace-file-url",
    payload: {
      requestId: "request-1",
      path: "relative-image.png"
    }
  },
  "*"
);
```

app は viewer 対象 file の位置を基準に `path` を解決し、結果を返す。

```js
{
  type: "integral:viewer-resolve-workspace-file-url-result",
  payload: {
    requestId: "request-1",
    path: "relative-image.png",
    relativePath: "Data/relative-image.png",
    url: "data:image/png;base64,..."
  }
}
```

失敗時は `payload.error` に message が入る。

sidebar view renderer の最小例:

```js
window.addEventListener("message", (event) => {
  if (!event.data || event.data.type !== "integral:set-sidebar-view") {
    return;
  }

  renderSidebar(event.data.payload);
});
```

### host

- `module` runtime のみ
- `host/index.cjs` が CommonJS module として読まれる
- export 名は `runIntegralPluginAction`

## 開発フロー

1. `plugins/<plugin-name>/` などで source を実装する
2. `integral-plugin.json`, `renderer/index.html`, `host/index.cjs` を揃える
3. zip 配布物を作る
4. GUI install か `install.bat` で `userData/plugins` へ入れる
5. IntegralNotes で確認する

## 配布方法

### 現在

- GUI
  - IntegralNotes の `Plugins` 画面から `Install ZIP`
- zip + bat
  - 開発中の配布向け
  - ユーザーに Node.js は不要

### 将来

- interactive renderer
  - plugin 専用 GUI で値を変更し、その結果を Markdown の `itg-notes` JSON に反映する
- plugin store
  - GUI から一覧取得、download、install

## この repo で使えるコマンド

- 配布物生成
  - `npm run plugins:package:all`
- plugin 単位
  - `npm --prefix plugins/shimadzu-lc run package:release`
- 開発用 local install
  - `npm run plugins:install:all`

## 生成される配布物

```text
plugins/dist/<pluginId>/
  <pluginId>-<version>.zip
  install-<pluginId>.bat
  uninstall-<pluginId>.bat
```

## いまの制約

- plugin renderer が返せる更新は現在 `params` のみ
- app 本体の raw JSON textarea は通常 UI から外しており、編集体験は plugin renderer 側が中心
- plugin renderer は action button を自前で描画できるが、実行権限そのものは app / host 側
- action 実行結果は `integral:action-state` で app -> plugin に返る
- validation / serialize は app 本体側
- `type` や top-level field の更新は app 側で保持・再 serialize する
- host は `stdio executable` ではなく module load の MVP
- Windows 以外の zip packaging / zip import は未対応
