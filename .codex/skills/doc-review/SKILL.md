`doc-review` は、この repo の仕様文書群と実装の整合性を確認するためのスキルです。

## 対象範囲

- 仕様文書の既定集合は `docs/**/*.md`
- ただし `docs/00_履歴/**` は除外する
- 実装比較の対象は `src/**`
- `integral-analysis` MCP は無関係なので使わない

明示指定がなければ、`docs/10_要求` を起点に `docs/20_アーキテクチャ` と `docs/30_設計` を照合し、必要な論点だけ `src` 実装まで追う。

## 進め方

1. まずレビューの軸を決める。
   - 文書同士の矛盾探索
   - 文書と実装の乖離探索
   - 両方

2. `docs/10_要求/*.md` を先に読み、この repo の canonical な要求を拾う。
   特に次の論点で分解すると追いやすい。
   - managed data / dataset / data-note
   - block schema / plugin / Python 実行
   - note link / image attach / embed preview
   - Explorer / Search / Sidebar などの UI 挙動

3. `docs/20_アーキテクチャ` と `docs/30_設計` を同じ論点で照合する。
   次の種類の不整合を優先して探す。
   - schema key 名の不一致
   - ID 命名規則や path 規約の不一致
   - source of truth の不一致
   - lifecycle や作成物の保存場所の不一致
   - UX 手順や操作結果の不一致

4. 実装比較が必要な場合だけ `src` を読む。
   - 実装の所在を `rg` で絞ってから読む
   - 仕様に対して opposite behavior があるかを見る
   - 単に未着手なのか、別仕様で実装されているのかを分ける

5. 指摘は次の 3 区分で整理する。
   - `矛盾`: 文書同士、または文書と実装で両立しない主張がある
   - `未実装/追随不足`: 仕様は具体的だが、`src` に対応実装が見当たらない、または部分的
   - `要確認`: 証拠が弱い、または文面が曖昧で断定できない

## 出力方針

- findings-first で返す
- 重要度が高いものから並べる
- 各指摘に根拠ファイルへの参照を付ける
- 実装レビューでは `docs` と `src` の両方を示す
- 長い要約より、具体的な不整合の列挙を優先する

指摘フォーマットの目安:

- `矛盾 | 高`: 何が食い違っているかを一文で書く。根拠: `[要求]`, `[設計]`, `[実装]`
- `未実装/追随不足 | 中`: 仕様上必要だが、現状の `src` では確認できない。根拠: `[要求]`, `[実装]`
- `要確認 | 低`: 曖昧さや追加確認が必要な点。根拠: `[要求]`, `[設計]`

## この repo での注意

- `docs/Issues.md` も既定では仕様文書集合に含めてよい。ただし issue の性質上、正式仕様より将来案や作業メモに近い記述が混ざる可能性があるので、`docs/10_要求` と矛盾した場合は要求側を優先する
- 仕様が複数段階で書かれているため、まず `docs/10_要求` を基準にし、その後にアーキテクチャと設計が追随できているかを見る
- `src` と比較するときは、機能未完了の可能性を前提に、単なる未実装と明確な逆挙動を分ける
